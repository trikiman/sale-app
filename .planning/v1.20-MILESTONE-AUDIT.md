## Late Insert — Phase 66.1 Stale-Color Phantom Strip

**Discovered:** 2026-05-12 during user review of milestone audit. User screenshot showed the MiniApp rendering 2 green product cards while VkusVill's own `/promo/` page stated "Зелёных ценников сейчас нет". Banner correctly warned "Данные устарели: зелёные" but products remained tappable.

**Root cause:** `backend/main.py::get_products` flagged `sourceFreshness.green.isStale=true` via `_build_source_freshness(stale_minutes=10)` but never filtered products of the stale type from the response. Users could tap phantom cards → `/api/cart/add` → VkusVill returned `product_gone` 410 (if the item was actually removed).

**Fix scope (commits `f821d45` + `c00d644`):**
- `backend/main.py::get_products` (~8 LOC): when any color's `isStale=true`, filter products of that type out of the response. Banner fields (`dataStale`, `staleInfo`, `sourceFreshness`, `greenLiveCount`) preserved untouched.
- `backend/test_products_stale_filter.py` (NEW, 3 tests): pins the stale-green drop case, the all-stale empty-list case, and the none-stale no-regression case.
- `scripts/verify_v1.20.sh` Phase 66.1 block (smoke gate 66.1-A).
- Full set of planning docs: 66.1-CONTEXT + 66.1-01-PLAN + 66.1-02-PLAN + 66.1-VERIFICATION.

**Live MCP evidence (captured 2026-05-12):**

| | Pre-fix (port 18067, HEAD~1 = `f74b008`) | Post-fix (port 18066, HEAD = `f821d45`) |
|---|---|---|
| Fixture | 2 green + 1 red + 1 yellow, green file aged 30 min | identical |
| `products` length | 4 | 2 |
| green products returned | 2 (phantom) | 0 |
| red/yellow returned | 1/1 | 1/1 |
| `dataStale` | `true` | `true` |
| `staleInfo` | `["green (30m)"]` | `["green (30m)"]` |
| `sourceFreshness.green.isStale` | `true` | `true` |
| `greenLiveCount` | 2 | 2 |

Same fixture, same banner, only `products[]` differs. Screenshots in `.planning/phases/66.1-stale-color-phantom-strip/mcp-{prefix,postfix}-*.png`. Delta: 2 phantom greens dropped, banner preserved, red+yellow untouched.

**Why it wasn't caught earlier:** the bug requires *simultaneous* scraper stall AND VkusVill actually having zero green items. Most of the time these don't coincide — our stale list looks close enough to VkusVill's live list that nobody noticed. The audit-period screenshot caught the exact overlap.

**This phase adds NO new requirement.** It closes a pre-existing UX regression that predates v1.20 (likely present since v1.9 when `sourceFreshness` landed without a filter). Treated as late insert, not new requirement line.

### Risks for production

- **First EC2 deploy of the milestone is a 16-commit jump.** Five phases land at once from the operator's perspective. Mitigation: the smoke script supports per-phase runs (`./scripts/verify_v1.20.sh 62` etc.), so the operator can deploy and gate each phase individually if desired; all five rollback procedures are commit-scoped.
- **Warmup daemon starts firing on deploy.** PERF-03 authenticated `GET /personal/` per linked user per 20 min is benign but novel traffic to VkusVill. Anti-spam floor (≤ 1/user/15 min) + pool-unhealthy gate + breaker-open gate all tested; worst case is the 3-state breaker flips to `open` and warmup is quiesced until half_open probes succeed.
- **Phase 64 swap is latent.** `FAST_CART_ADD_URL = None` and `USE_FAST_CART_ADD_ENDPOINT = False` ship as dead code paths. Smoke 64-B explicitly asserts URL is None. If an operator later populates `FAST_CART_ADD_URL` without also updating smoke 64-B expectations, the smoke gate will flip to a green "swap ready" state — by design — but the flip needs operator awareness.
- **Phase 65 frontend 8 s → 5 s is user-visible.** If PERF-03/04/06/07 don't actually land live p95 < 5 s on EC2, users see AbortController timeouts more often than before, with polling as the safety net. The polling path is covered by backend unit tests but not by an end-to-end Playwright run in CI — NEEDS_OPERATOR-1 is the gate.
- **Phase 66 `/api/health/deep` now has a mutable `cart_add` block that can flip the overall `status` to `degraded`/`unhealthy`.** If live cart-add p95 exceeds 6 s on any given hour after deploy, external uptime monitors will flag the endpoint as degraded. This is the correct signal; operators should calibrate downstream alert thresholds before enabling.
