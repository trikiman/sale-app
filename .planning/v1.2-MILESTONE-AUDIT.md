## Critical Gaps (3 issues)

### Gap 1: MiniTimeline bars always empty 🔴

**Affected:** HIST-09 (partial)

The `HistoryCard` component renders a `MiniTimeline` using `product.day_pattern`, but the `/api/history/products` API endpoint **does not return this field**. The `product_catalog` table also has no `day_pattern` column — this data only exists via `prediction.py` which is called only for the detail endpoint.

**Frontend expects:**
```js
product.day_pattern   // → undefined (not in API response)
product.last_old_price // → undefined (not in API response)
product.confidence    // → undefined (not in API response)
```

**API returns:**
```json
{
  "id": "...", "name": "...", "last_known_price": 171,
  "total_sale_count": 1, "avg_discount_pct": 10,
  "usual_time": "04:55", "avg_window_min": 929
  // ← NO day_pattern, NO last_old_price, NO confidence
}
```

**Fix needed:** Either add batch predictions to `/api/history/products`, or compute and cache `day_pattern` in `product_catalog`.

### Gap 2: No old price displayed on cards 🟡

**Affected:** HIST-09 (visual completeness)

The `HistoryCard` shows `product.last_old_price` with strikethrough styling, but the API only returns `last_known_price` (sale price). The `product_catalog` table doesn't store `old_price`. The old price IS available in `sale_sessions.old_price` but isn't joined.

### Gap 3: Ghost cards never trigger 🟡

**Affected:** HIST-11 (unsatisfied)

Ghost card logic: `const isGhost = !hasSales && !product.is_currently_on_sale`

The issue: `hasSales = product.total_sale_count > 0` — for the 16K+ seeded products that have never been on sale, `total_sale_count` is indeed 0, BUT these products are sorted to the very end (`ORDER BY last_sale_at DESC NULLS LAST`). With infinite scroll showing 50 per page, users would need to scroll through 300+ pages of products-with-sales before seeing any ghost cards. In practice, ghost cards are invisible to users.
