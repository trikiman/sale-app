## Hotfix — Phase 67 Reprobe Bridge-Route

**Discovered:** 2026-05-12 within 2 minutes of Phase 67 deploying to EC2. Scheduler logs showed `{admitted_count:14, probed:14, passed:0, failed_hosts:[all 14]}` on the very first reprobe cycle. Two minutes in, all 14 admitted nodes were in 4h cooldown.

**Root cause:** `_probe_vkusvill(proxy=None)` performs a DIRECT HTTP request to `vkusvill.ru` without going through the xray bridge. From EC2, the direct egress IP is non-RU-residential AWS, which VkusVill's anti-bot immediately flags. Every probe returned `/vpn-detected/` or 403, never the real catalog.

**Fix (commit `a234f6d`):**
- Introduced `REPROBE_BRIDGE_ADDR = "127.0.0.1:10808"` in `keepalive/reprobe.py`.
- `_run_cycle` passes `proxy=REPROBE_BRIDGE_ADDR` to `_probe_vkusvill` so the probe egresses through the RU outbound same as production cart-add traffic.
- Admission probe path (`_probe_candidates_in_parallel`) intentionally kept `proxy=None` — that probe runs through a per-candidate xray instance on a unique port and is a different signal.

**Live post-hotfix evidence:**
- Manually cleared 10 false-positive `reprobe_fail` cooldowns from `data/vkusvill_cooldowns.json` via `python3 -c` edit.
- Restarted `saleapp-scheduler`.
- Next reprobe cycle: `{admitted_count:10, probed:10, passed:10, failed_hosts:[], duration_ms:16466}` — 10/10 pass.
- Pool snapshot 3 cycles later: `{size:8, active_outbounds:8, quarantined_count:0, dead_by_success_rate_count:0}`.
- 6/6 tests in `tests/test_reprobe_loop.py` green before and after hotfix (tests already mocked `_probe_vkusvill` correctly).

**Why it wasn't caught earlier:** unit tests mocked `_probe_vkusvill` to return True/False directly — they tested the orchestration logic, not the probe itself. The direct-probe signature was inherited from Phase 67's admission path where direct=through-one-candidate-xray, and we missed that reprobe calls needed the DIFFERENT proxy value. Live deploy was the only place that would have surfaced it.
