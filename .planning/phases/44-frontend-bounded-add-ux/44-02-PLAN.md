# Plan 02: Quantity Support Contract

## Objective

Provide the smallest safe contract needed for the new quantity control: cart items must preserve non-integer quantities, and the app needs a real set/decrement path instead of the current remove-whole-line behavior.

## must_haves

- [ ] Cart item payload keeps decimal quantities instead of coercing `Q` to `int`
- [ ] There is a backend route that can set or adjust quantity without always deleting the whole cart line
- [ ] Quantity `0` removes the line item
- [ ] CartPanel uses the same quantity semantics that the new card/detail control will rely on

## Tasks

<task id="01" title="Add decimal-safe cart quantity serialization and a set-quantity backend route">
<read_first>
- backend/main.py
- cart/vkusvill_api.py
- .planning/phases/44-frontend-bounded-add-ux/44-CONTEXT.md
- .planning/phases/43-backend-cart-response-contract/43-CONTEXT.md
</read_first>

<action>
Extend the cart contract so Phase 44 can represent and update real quantities:

1. Stop coercing cart item `Q` and `MAX_Q` to `int` in `backend/main.py:/api/cart/items`.
2. Serialize `quantity` and `max_q` as number values that preserve decimals when VkusVill returns non-integer quantities.
3. Add a concrete backend route for quantity changes, for example `POST /api/cart/set-quantity`, with:
   - `user_id`
   - `product_id`
   - `quantity`
4. In `cart/vkusvill_api.py`, add a helper that uses `basket_update.php` to set the target quantity for an existing cart line instead of only deleting it.
5. If `quantity == 0`, route the action through the existing remove behavior so zero still means remove.
6. Keep the existing add/remove routes working for compatibility.
</action>

<acceptance_criteria>
- `/api/cart/items` no longer forces `quantity` and `max_q` through `int(...)`
- backend/main.py exposes a set-quantity cart route
- cart/vkusvill_api.py contains a helper that can set a target cart quantity through `basket_update.php`
- setting `quantity = 0` removes the cart line
- legacy `/api/cart/add` and `/api/cart/remove` routes remain present
</acceptance_criteria>
</task>

<task id="02" title="Move CartPanel quantity changes onto the new quantity contract">
<read_first>
- miniapp/src/CartPanel.jsx
- backend/main.py
- cart/vkusvill_api.py
- .planning/phases/44-frontend-bounded-add-ux/44-CONTEXT.md
</read_first>

<action>
Update `miniapp/src/CartPanel.jsx` so it uses the new quantity semantics instead of pretending remove is a decrement:

1. Replace the current `delta > 0 ? /api/cart/add : /api/cart/remove` quantity path with calls to the new set-quantity route.
2. Keep plus/minus behavior, but derive the next quantity from the current item quantity instead of assuming remove decrements.
3. Preserve the existing busy-state UX and zero-removal behavior.
4. Ensure the cart panel can display non-integer weighted quantities without truncating them to an integer string.
</action>

<acceptance_criteria>
- CartPanel.jsx no longer decrements quantity by calling `/api/cart/remove` as if it were a one-step decrement
- CartPanel.jsx uses the new quantity route for plus/minus changes
- CartPanel.jsx can render decimal weighted quantities without truncation
- quantity `0` still removes the item from the cart panel
</acceptance_criteria>
</task>
