# Plan 19-02: CSS Performance — Backdrop-filter & Reduced Motion

<objective>
Remove GPU-heavy `backdrop-filter: blur()` on tablets/mobile devices and add `prefers-reduced-motion` support. Desktop keeps all existing effects unchanged.
</objective>

## Tasks

<task id="1">
<title>Disable backdrop-filter on tablets and mobile</title>
<read_first>
- miniapp/src/index.css (search for all `backdrop-filter` occurrences — there are 9)
</read_first>
<action>
Add a media query at the END of `miniapp/src/index.css` that disables backdrop-filter on touch devices / small screens. This OVERRIDES the existing styles without modifying them — desktop keeps everything.

Add this block at the end of `index.css`:

```css
/* === PERF-12: Disable backdrop-filter on tablets/mobile (GPU bottleneck) === */
@media (max-width: 1024px), (pointer: coarse) {
  .card-vertical,
  .product-detail-overlay,
  .cart-panel-overlay,
  .header-bar,
  .category-bar,
  .login-overlay,
  .toast-message,
  .status-bar,
  .search-bar {
    backdrop-filter: none !important;
    -webkit-backdrop-filter: none !important;
  }
}
```

The selectors should match ALL elements that currently have `backdrop-filter` in the CSS. Read the CSS file to identify the exact class names. The `(pointer: coarse)` media query catches iPads in landscape mode (wider than 1024px but still a touch device).

IMPORTANT: Do NOT modify any existing CSS rules. Only ADD new rules in a media query at the end. This ensures desktop (mouse + wide screen) is completely unchanged.
</action>
<acceptance_criteria>
- index.css contains `@media (max-width: 1024px), (pointer: coarse)` block
- The media query block contains `backdrop-filter: none !important` for at least 5 selectors
- No existing `backdrop-filter` rules in the file are modified or removed
- Desktop browser (mouse, > 1024px) still sees all backdrop-filter effects
</acceptance_criteria>
</task>

<task id="2">
<title>Add prefers-reduced-motion support</title>
<read_first>
- miniapp/src/index.css (search for `animation` and `transition` — there are 60+ instances)
</read_first>
<action>
Add a `prefers-reduced-motion` media query at the END of `miniapp/src/index.css` that disables/simplifies animations for users who request reduced motion in their OS settings.

Add this block:

```css
/* === PERF-05: Respect prefers-reduced-motion === */
@media (prefers-reduced-motion: reduce) {
  *,
  *::before,
  *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
}
```

This is the standard accessibility pattern — it reduces all animations to near-instant while still triggering `animationend`/`transitionend` events (so JS handlers don't break). The `!important` ensures it overrides all existing animation declarations.

IMPORTANT: Do NOT modify any existing CSS rules. Only ADD new rules in the media query.
</action>
<acceptance_criteria>
- index.css contains `@media (prefers-reduced-motion: reduce)` block
- The block contains `animation-duration: 0.01ms !important`
- The block contains `transition-duration: 0.01ms !important`
- No existing animation/transition rules in the file are modified
</acceptance_criteria>
</task>
