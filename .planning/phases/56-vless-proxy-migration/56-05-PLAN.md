# Plan 56-05: Production Deployment + Rollback Rehearsal

## Goal

Deploy the VLESS+xray stack to EC2 production, prove it works against real VkusVill traffic (cart-add + scraper), and rehearse the one-operation rollback on a throwaway branch so we know the recovery path works BEFORE we actually need it.

This is the phase's verification plan. Everything prior (56-01 through 56-04) is code work; this plan is operational work plus the end-to-end live proof that the milestone's success criteria are met.

## Scope

- Live verification on production against real VkusVill endpoints (cart add, remove, scraper cycle)
- Rollback rehearsal: on a sacrificial branch, `git revert` the 56-04 commit, verify pytest still passes and the old SOCKS5 path comes back cleanly (if that path still worked — note: it may not, because SOCKS5 pool is dead; goal is to prove the code path is restored, not that SOCKS5 works)

**Out of scope:**

- Revisiting cart-add UX (v1.13/v1.14 territory)

## EC2 systemd Layout

```

  saleapp-xray.service          # xray-core process
  saleapp-scheduler.service     # existing scheduler, now depends on saleapp-xray
  saleapp-backend.service       # existing FastAPI backend (if run under systemd — otherwise via nginx/gunicorn)
```

### `saleapp-xray.service`

```ini
[Unit]
Description=Xray-core SOCKS5 bridge for saleapp (VLESS+Reality)
Documentation=file:///home/ubuntu/saleapp/docs/PROXY_MIGRATION.md
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=ubuntu
Group=ubuntu
WorkingDirectory=/home/ubuntu/saleapp
ExecStart=/home/ubuntu/saleapp/bin/xray/current/xray -config /home/ubuntu/saleapp/bin/xray/configs/active.json
Restart=always
RestartSec=5s
KillMode=process
StandardOutput=append:/home/ubuntu/saleapp/bin/xray/logs/xray.stdout.log
StandardError=append:/home/ubuntu/saleapp/bin/xray/logs/xray.stderr.log
# Security hardening
NoNewPrivileges=true
ProtectSystem=strict
ProtectHome=read-only
ReadWritePaths=/home/ubuntu/saleapp/bin/xray /home/ubuntu/saleapp/data /home/ubuntu/saleapp/.cache
PrivateTmp=true
# Resource limits
LimitNOFILE=65536

[Install]
WantedBy=multi-user.target
```

**Design notes:**
- `ProtectSystem=strict + ReadWritePaths` keeps xray sandboxed; it cannot write anywhere except its own paths
- `Restart=always` + 5s delay absorbs transient failures; prolonged failure is caught by the health-check cron (see below)
- xray writes logs directly to disk via stdout/stderr — Python wrapper log rotation is bypassed in prod (systemd journal + append files are enough)

### `saleapp-scheduler.service` (update existing)

Add to `[Unit]`:
```ini
After=network-online.target saleapp-xray.service
Requires=saleapp-xray.service
```

This causes systemd to start xray BEFORE the scheduler, and to restart the scheduler if xray is restarted (matches the assumption that the scheduler has an open xray connection).

### Health-check cron (simple operational safeguard)

```cron
*/5 * * * * ubuntu /home/ubuntu/saleapp/scripts/xray_healthcheck.sh >> /home/ubuntu/saleapp/logs/xray_healthcheck.log 2>&1
```

`xray_healthcheck.sh`:

```bash
#!/bin/bash
set -eo pipefail

# If xray systemd unit is active but SOCKS5 port is not accepting, restart it
if systemctl is-active --quiet saleapp-xray; then
  if ! timeout 2 bash -c '</dev/tcp/127.0.0.1/10808' 2>/dev/null; then
    echo "$(date -Iseconds) xray active but port 10808 unresponsive — restarting" >&2
    systemctl restart saleapp-xray
  fi
fi
```

## Deployment Script (`scripts/deploy_v1_15.sh`)

Invocation (local → EC2 via SSH):

```bash
./scripts/deploy_v1_15.sh
```

Behavior:

```bash
#!/bin/bash
set -euo pipefail

HOST="${HOST:-ubuntu@13.60.174.46}"
SSH_KEY="${SSH_KEY:-./scraper-ec2-new}"
SSH="ssh -i $SSH_KEY $HOST"

echo ">>> [1/8] Pulling latest code on EC2"
$SSH "cd /home/ubuntu/saleapp && git fetch origin && git checkout main && git pull origin main"

echo ">>> [2/8] Ensuring Python deps"
$SSH "cd /home/ubuntu/saleapp && python3 -m pip install -r requirements.txt --upgrade"

echo ">>> [3/8] Installing xray-core v24.11.30"
$SSH "cd /home/ubuntu/saleapp && python3 scripts/bootstrap_xray.py --version 24.11.30"

echo ">>> [4/8] Running initial VLESS pool refresh"
$SSH "cd /home/ubuntu/saleapp && python3 -c 'from vless.manager import VlessProxyManager; pm = VlessProxyManager(); n = pm.refresh_proxy_list(); print(f\"admitted {n} nodes\"); assert n >= 5, \"pool below minimum\"'"

echo ">>> [5/8] Installing systemd units"
scp -i "$SSH_KEY" systemd/saleapp-xray.service "$HOST:/tmp/"
scp -i "$SSH_KEY" systemd/saleapp-scheduler.service "$HOST:/tmp/"
$SSH "sudo mv /tmp/saleapp-xray.service /etc/systemd/system/ && sudo mv /tmp/saleapp-scheduler.service /etc/systemd/system/"
$SSH "sudo systemctl daemon-reload"

echo ">>> [6/8] Installing health-check cron"
scp -i "$SSH_KEY" scripts/xray_healthcheck.sh "$HOST:/home/ubuntu/saleapp/scripts/"
$SSH "chmod +x /home/ubuntu/saleapp/scripts/xray_healthcheck.sh"
$SSH "(crontab -l 2>/dev/null | grep -v xray_healthcheck; echo '*/5 * * * * /home/ubuntu/saleapp/scripts/xray_healthcheck.sh >> /home/ubuntu/saleapp/logs/xray_healthcheck.log 2>&1') | crontab -"

echo ">>> [7/8] Enabling and starting xray service"
$SSH "sudo systemctl enable saleapp-xray && sudo systemctl restart saleapp-xray"
sleep 3
$SSH "sudo systemctl status saleapp-xray --no-pager -l | head -n 20"

echo ">>> [8/8] Restarting scheduler to pick up xray dependency"
$SSH "sudo systemctl restart saleapp-scheduler"
sleep 3
$SSH "sudo systemctl status saleapp-scheduler --no-pager -l | head -n 10"

echo ">>> Deploy complete. Run live verification:"
echo "   ./scripts/verify_v1_15.sh"
```

**Real-world rollback (only if v1.15 causes a production incident):**

```bash
ssh -i ./scraper-ec2-new ubuntu@13.60.174.46 <<'EOF'
cd /home/ubuntu/saleapp
git log --oneline -n 20
# Identify the 56-04 commit hash from the log
git revert --no-edit <hash>
git push origin main
sudo systemctl stop saleapp-xray
sudo systemctl disable saleapp-xray
sudo systemctl restart saleapp-scheduler
EOF
```

After a real rollback, the SOCKS5 pool is back in code but is still dead in practice (0% alive). Real rollback is an emergency measure to unblock scheduler startup, not a fix — the path forward after rollback is to re-land v1.15 with whatever fix was needed, not to stay on legacy.

## Operator's Guide (`docs/PROXY_MIGRATION.md`)

Key sections:

1. **Architecture overview** — one diagram showing Python clients → `socks5://127.0.0.1:10808` → xray-core → VLESS nodes → internet → vkusvill.ru. ASCII-art is fine.

2. **Daily operations**
   - "How do I see if xray is healthy?" → `systemctl status saleapp-xray`; `tail -f bin/xray/logs/xray.stderr.log`
   - "How do I see the current pool size?" → `python3 -c 'from vless.manager import VlessProxyManager; pm = VlessProxyManager(); print(pm.xray_status())'`
   - "How do I force a pool refresh?" → `python3 -c 'from vless.manager import VlessProxyManager; pm = VlessProxyManager(); print(pm.refresh_proxy_list())'`
   - "Where are the logs?" → `bin/xray/logs/`, `logs/scheduler.log`, `data/proxy_events.jsonl`

3. **Troubleshooting**
   - Scheduler won't start → check `saleapp-xray` status first (Requires= dependency)
   - Pool empty → run refresh manually; if still empty, check VLESS source URL reachability
   - Pool keeps emptying → VLESS source may have gone stale; check `.planning/phases/56-vless-proxy-migration/56-CONTEXT.md` for the canonical URL and alternatives
   - Cart-add fails with timeout → likely current node is VkusVill-blocked; check `data/proxy_events.jsonl` for recent `vkusvill_cooldown` entries; `pm.next_proxy()` rotates

4. **Rollback** — step-by-step as above

5. **Upgrading xray-core** — bump `XRAY_VERSION` in `vless/installer.py`, update `XRAY_SHA256`, run `scripts/bootstrap_xray.py --force`, restart `saleapp-xray`

## Risks & Mitigations

- **EC2 doesn't allow outbound to VLESS ports (some providers restrict).** → Verify during deploy: if xray can't connect to any node, fail fast. VLESS typically uses 443 which is almost never blocked outbound.
- **First refresh on EC2 rate-limits the geo providers.** → `scripts/geo_providers.py` has 10 providers with cache. First run is slow; subsequent runs are fast. Cache is warmed by the deploy step.
- **systemd unit has an error and masks the real failure.** → Deploy script `systemctl status` output is captured; if the unit fails to start, the script exits non-zero with the status output visible.
- **Live verification step 5 requires a valid VkusVill cookie.** → Document how to obtain one (manual browser → dev tools → copy PHPSESSID) in `PROXY_MIGRATION.md`. Verification is expected to be run manually after deploy, not from CI.
- **Rehearsal reveals pytest failures after revert.** → Fix in code before considering 56-04 complete; the rehearsal is the final acceptance gate for 56-04's rollback contract.
