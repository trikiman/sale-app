## API Contract (Exact Compatibility with Existing `ProxyManager`)

Every public method below exists on the current `proxy_manager.ProxyManager`. Signatures and return types MUST match. Read `proxy_manager.py` lines 66-413 for the source of truth on behavior. The table below lists each method and how its SOCKS5 semantics translate to VLESS semantics.

| Method | SOCKS5 Behavior | VLESS Behavior (Required) |
|--------|-----------------|---------------------------|
| `__init__(log_func=None)` | Loads pool, prunes cooldowns | Same — plus ensures xray is installed, creates (doesn't start) `XrayProcess` |
| `check_direct() -> bool` | Probes vkusvill.ru direct | Same — direct probe, no proxy, identical logic |
| `check_direct_cached(ttl=60) -> bool` | TTL cache over `check_direct` | Same — identical |
| `note_direct_result(ok: bool)` | Updates cache | Same — identical |
| `get_working_proxy(allow_refresh=True) -> str \| None` | Returns `"ip:port"` of first pool entry or None | Returns `"127.0.0.1:<xray-port>"` if pool has ≥1 healthy VLESS node AND xray is running; else None. Starts xray lazily if not running. |
| `get_proxy_for_chrome() -> str \| None` | Returns `f"socks5://{get_working_proxy()}"` | Same — returns `"socks5://127.0.0.1:10808"` |
| `pool_count() -> int` | Number of proxies in pool | Number of **VLESS nodes** in pool (NOT 1 — this reflects underlying diversity, important for `pool_healthy()` logic) |
| `pool_healthy() -> bool` | `pool_count() >= MIN_HEALTHY` | Same constant `MIN_HEALTHY = 7`. Healthy if ≥7 VLESS nodes are admitted to the active pool AND xray is running. |
| `remove_proxy(addr: str) -> None` | Removes `addr` from pool | If `addr == "127.0.0.1:10808"` (the local xray endpoint), treat as ambiguous and log a warning, don't remove anything (caller should use `remove_vless_node(host)` for node-level removal). If `addr` is a VLESS host:port, remove from pool and rebuild+restart xray. Maintains backward compatibility for callers that still pass the legacy "proxy address" format. |
| `next_proxy() -> str \| None` | Remove current, return next | Since xray is the single address, this method rotates the **underlying VLESS node** inside xray: triggers node-level rotation and returns `"127.0.0.1:10808"` again if the pool is still non-empty. |
| `mark_vkusvill_blocked(addr, reason="timeout")` | Adds to 4h cooldown | Same — but `addr` is expected to be the VLESS host IP, not `127.0.0.1:10808`. When caller only knows the local endpoint, use `mark_current_node_blocked(reason)` helper instead. |
| `is_in_vkusvill_cooldown(addr) -> bool` | Cooldown lookup | Same |
| `cooldown_addrs() -> set[str]` | Set of cooldown IPs | Same |
| `ensure_pool() -> int` | Refreshes if low/stale | Same semantics, but uses VLESS fetch pipeline |
| `refresh_proxy_list(exclude=None) -> int` | SOCKS5 fetch+test | VLESS fetch+parse+geo-filter+test+rebuild xray config+restart xray |
| `is_cache_stale() -> bool` | TTL check | Same |

### New Methods (Additions, Not Replacements)

```python
class VlessProxyManager:
    def remove_vless_node(self, host: str) -> None:
        """Remove a specific VLESS node by its host IP. Rebuilds xray config
        and restarts xray. Use when a specific node is proven dead (e.g., TLS
        handshake fail through that node). Does NOT put the node in cooldown.
        """

    def mark_current_node_blocked(self, reason: str = "timeout") -> None:
        """Mark whichever VLESS node xray is currently using as VkusVill-blocked.
        xray's `status` API identifies the active node; the identified host enters
        the 4h cooldown via the existing mark_vkusvill_blocked path.
        """

    def current_node(self) -> dict | None:
        """Return metadata for the currently-active VLESS outbound (best effort).
        Returns None if not introspectable. Used for diagnostics / admin UI.
        """

    def xray_status(self) -> dict:
        """Return {running: bool, pid: int|None, inbound_port: int, pool_size: int,
        last_config_reload: iso8601, last_crash_at: iso8601|None}.
        """
```

## Refresh Pipeline (`refresh_proxy_list`)

1. Prune expired VkusVill cooldowns (`_prune_expired_cooldowns`) — reuse existing helper, identical behavior
2. `vless.sources.fetch_igareck_list()` → raw text
3. `vless.parser.parse_vless_list(text)` → list of `VlessNode` + parse errors
4. `vless.sources.filter_ru_nodes(nodes)` → RU-only nodes
5. Drop any RU node whose `host` IP is in `cooldown_addrs()` — these are in a 4h block
6. Optionally drop nodes matching `exclude`
7. **Test nodes in parallel** (reuse the pattern from `scripts/test_ru_proxy_pipeline.py`):
   - For each RU node, build a single-node xray config, start a temporary xray on a unique port, probe vkusvill.ru via that port, stop xray
   - OR (simpler) build one combined xray config with ALL RU nodes as outbounds + a balancer, but temporarily override routing to pin each outbound one-at-a-time via separate probe runs. Pick whichever is simpler for the executing AI; both are correct.
   - Parallelism: ~8 concurrent node tests, each with a ~10s hard timeout
   - A node is "admitted" if the probe returns 200 with "vkusvill" in the body (same logic as `_probe_vkusvill`)
8. Write the admitted nodes to `data/vless_pool.json`
9. Build final xray config with all admitted nodes + balancer: `vless.config_gen.build_xray_config(admitted)`
10. `self._xray.restart(new_config=config)` — xray comes up with the fresh pool
11. Emit `vless_refresh_start` event and `vless_node_admitted` per node
12. Return count of new admissions

**Subnet diversity rule** — preserve from SOCKS5 implementation: max 3 nodes per /24 subnet in the admitted pool. Sort admitted by measured speed, keep diverse first 30. Implementation is copy-paste from `proxy_manager.py:370-383`.

**Time budget:** entire refresh should complete in under 180 seconds for a ~55-candidate list. If it exceeds this budget, log a warning but continue (consistent with existing outer-timeout pattern).

## Lazy xray Lifecycle

- `VlessProxyManager.__init__` does NOT start xray (keeps constructor cheap and mockable)
- xray starts lazily on first call to `get_working_proxy()` or `refresh_proxy_list()`
- xray auto-installs if not present (calls `vless.installer.install()`) — but this is a slow path; most production runs will have xray pre-installed via `scripts/bootstrap_xray.py` during deploy
- xray stops on process exit via `atexit` handler registered in `__init__`
- xray also stops on `sys.exit` / SIGTERM via `signal.signal(SIGTERM, ...)`, but ONLY if `VlessProxyManager` is the module-level singleton (which is how current production uses it)

## Risks & Mitigations

- **Parallel node-testing with multiple xray processes may exhaust ports or hit OS fd limits.** → Hard cap concurrency at 8. Each test xray uses a distinct high port (random 20000-30000 range). Processes are ref-counted and reaped via `XrayProcess.stop()` in a `finally`.
- **xray restart during cart-add in flight causes in-flight request to fail.** → Acceptable v1 behavior; restart frequency is low (only on refresh or on node removal). Caller retries are the existing pattern. 56-03 doesn't need a zero-downtime reload. If needed later, xray's "hot reload" via API can be added as an enhancement.
- **Pool empties between `pool_healthy()` check and `get_working_proxy()` call.** → `get_working_proxy` always re-checks pool state under lock; returns None if emptied.
- **Node metadata grows unboundedly over weeks.** → `data/vless_pool.json` is overwritten on each refresh with only the admitted set, so it stays bounded at ~30 entries.
