## Implementation Steps

### Step 1 — Create `scripts/verify_v1_17.sh`

Copy `scripts/verify_v1_15.sh` to `scripts/verify_v1_17.sh`. The file already has 5 checks (service status, egress country, scraper cycle, cart-add via scheduler, miniapp cart-add via Vercel).

Changes to make:

**Step 2 (egress check):** The v1.15 version treated non-RU egress as a "caveat." v1.17 must FAIL on any non-RU response because 57-03 restored geo verification during admission. Replace:

```bash
# v1.15 (lenient)
if [ "$country" = "RU" ]; then
    echo "  ✓ RU egress: $ip"
    ru_count=$((ru_count + 1))
else
    echo "  ⚠ Non-RU egress: $ip (country: $country) — caveat, not blocker"
fi

# v1.17 (strict)
if [ "$country" = "RU" ]; then
    echo "  ✓ RU egress: $ip"
    ru_count=$((ru_count + 1))
else
    echo "  ✗ Non-RU egress: $ip (country: $country) — FAIL"
    exit 1
fi
```

Require `ru_count -eq 15` (not `-ge 10` like v1.15). This is the whole point of 57-03.

**Step 5 (Vercel miniapp cart-add):** The v1.15 script had this step marked as "skipped — API contract changed." Un-skip it. The current `/api/cart/add` endpoint should be tested against a Vercel deployment of the miniapp. Read the current `miniapp/src/services/api.ts` or equivalent to learn the exact endpoint shape, then hit it via `curl`:

```bash
# Example — adapt to actual endpoint
curl -sS -X POST "https://vkusvillsale.vercel.app/api/cart/add" \
    -H "Content-Type: application/json" \
    -d '{"product_id": 731, "price_type": 1}'
```

Expected: HTTP 200 with `{"success": true}` or a reasonable error (never a 504 Gateway Timeout, which would indicate the VLESS bridge hung).

If the exact endpoint shape has changed since phase 56 was planned, update the curl command and document the new shape. Do NOT skip this step — the whole reason v1.17 exists is because this path was never tested.

### Step 2 — Create `scripts/deploy_v1_17.sh`

Copy `scripts/deploy_v1_15.sh` to `scripts/deploy_v1_17.sh`. The file already has 9 steps. Add a new step 10 (after all systemd restarts) to force a pool refresh so observatory is populated:

```bash
# Step [10/10]: Force refresh so new xray policy + observatory take effect.
echo "[10/10] Triggering pool refresh to apply v1.17 xray config..."
sudo -u ubuntu bash -c 'cd /home/ubuntu/saleapp && python -c "
from vless.manager import VlessProxyManager
pm = VlessProxyManager(auto_install_xray=False)
# Force full refresh — v1.17 changed config_gen; without this the existing
# active.json is stale (pre-observatory). leastPing with uninitialized
# observatory data falls back to random, so this is a soft requirement
# but strongly recommended.
pm.refresh_proxy_list()
print(f'Pool size after refresh: {pm.pool_count()}')
"'

# Restart xray one more time to pick up the fresh active.json.
sudo systemctl restart saleapp-xray.service

sleep 3
if ! sudo systemctl is-active --quiet saleapp-xray.service; then
    echo "  ✗ saleapp-xray failed to restart after refresh — aborting"
    sudo journalctl -u saleapp-xray -n 50 --no-pager
    exit 1
fi

echo "  ✓ xray restarted with fresh config"
```

Update step count in the file header from 9 to 10.

### Step 7 — Update `.planning/ROADMAP.md`

Below the v1.15 section, add:

```markdown
## v1.17 — VLESS Timeout Hardening (SHIPPED 2026-04-23)

Follow-up to v1.15/v1.16. Fixed three root causes for the "middle of
connection timeout" bug reported by the user after the v1.15 rollout:

1. xray config was missing the `policy` block → default `connIdle=300s`
   kept dead connections alive for 5 minutes. Now set to 30s.
2. xray had no `observatory` → dead outbounds stayed in the random
   balancer forever. Now probed every 5 minutes, `leastPing` strategy.
3. `remove_proxy("127.0.0.1:10808")` was a silent no-op. Now rotates
   via `mark_current_node_blocked`, breaking the hang-retry loop.

Plus timeout alignment in cart + backend for VLESS handshake cost
(3-5s observed) and restored egress geo-verification in admission
probes (v1.16 PR #7 had removed it, violating plan D-05).

Phase: `.planning/phases/57-vless-timeout-hardening/`
Inspection: `.planning/phases/56-vless-proxy-migration/INSPECTION-2026-04-23.md`
```
