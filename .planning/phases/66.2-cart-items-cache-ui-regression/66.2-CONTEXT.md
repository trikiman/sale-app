# Phase 66.2 — Cart-Items Cache UI Regression Fix

**Milestone:** v1.20 Cart-Add Latency & User-Facing Responsiveness (second late insert)
**Phase number:** 66.2 (decimal insert after 66.1, before milestone close)
**Phase slug:** cart-items-cache-ui-regression
**Date captured:** 2026-05-12
**Requirements covered:** bugfix on PERF-06 integration boundary — no new requirement

---

## Domain

User observation after v1.20 + 66.1 deploy: "tap add — ends in success but no ± stepper shown, just a clock icon. Tap again — stepper appears normally."

Root cause chain:
1. `POST /api/cart/add` succeeds. VkusVill returns `Q_ITEMS=3` (int, the total cart size, not a per-item list).
2. `backend/main.py::cart_add_endpoint` stamps `_cart_add_attempts[attempt_id].cart_items = 3` and `last_known_cart_at_monotonic = now()` (line ~3875).
3. Frontend `handleAddToCart` seeds optimistic state: `cartItemsById[pid] = {quantity: 1, unit, ...}` (line ~976).
4. Frontend calls `refreshCartState()` immediately after success (line ~987) to sync the full cart.
5. `refreshCartState` fetches `/api/cart/items/{user_id}`.
6. Phase 63 PERF-06 cache-hit path fires: attempt is <12s old and has `last_known_cart_at_monotonic` stamped, so backend returns `{items_count: 3, total_price: 0, items: [], from_cache: true}` (line ~4212 — `_items_out = []` when `cart_items` is not a list).
7. Frontend `refreshCartState` did not know about `from_cache`. It called `buildCartItemMap(cart.items || [])` which returned empty `itemIds` Set + empty `itemsById` dict.
8. Frontend set `cartItemsById = {}` — wiping the optimistic seed from step 3.
9. UI stepper component reads `cartItemsById[pid]` → null → falls back to the "status unknown" ⟲ clock icon.
10. ~12 seconds later the cache TTL (`CART_ITEMS_CACHE_TTL_S = 12.0`) expires. Next `refreshCartState` hits the real `basket_recalc.php` path, gets the full items list, rebuilds `cartItemsById` correctly, stepper renders.

Net effect: stepper blinks to ⟲ for 12 seconds after every add. User saw this live on the deployed build.

Phase 63's CONTEXT documented this as a trade-off: "We therefore cannot reconstruct the full items list from the cache — return an empty items list + the count. This is a deliberate tradeoff." But the UI contract treated an empty items list as authoritative, not advisory. The cache-hit path was effectively breaking UX to save bridge bandwidth.

## SPEC Lock

LOCKED — planner must NOT re-litigate:

- **Fix surface:** frontend-only (`miniapp/src/App.jsx::refreshCartState`). Backend Phase 63 behavior stays unchanged — the cache hit is still real, scrapers still pause, `basket_recalc.php` is still skipped. Only the UI stops trusting the empty `items` array when `from_cache: true`.
- **Cache-hit detection:** JSON response fields `from_cache === true` AND `Array.isArray(items) && items.length === 0` AND `items_count > 0`. Triple-check guards against false positives (e.g. genuinely empty cart that isn't from cache — which would have `items_count === 0` and not match).
- **On cache hit:** update `cartCount` from `items_count` (keeps header badge fresh) but leave `cartItemIds` and `cartItemsById` untouched (preserves the optimistic state set by `handleAddToCart`).
- **Other paths untouched:** cache-miss, source-unavailable, empty-cart-for-real, error — all keep their existing behavior.
- **No backend change:** keeps 66.2 scoped; Phase 63 tests continue to pass untouched.
- **OPS-09 smoke:** no new EC2 smoke check needed (frontend-only change). Existing 65-C (AbortController verification via grep) is the closest pattern; 66.2 is just "the build succeeds and the check compiles".
- **OPS-11 rollback rehearsal:** single-commit revert, no data-structure changes.

## Decisions

### D1. Why not patch the backend?

Two reasons:

1. The backend legitimately doesn't have the items list to return. `basket_add.php` response contains `Q_ITEMS` (total count) but not the per-item array. To populate a real items list we'd need to call `basket_recalc.php` once after every add to stamp the snapshot — which is exactly what Phase 63 PERF-06 was designed to avoid. Defeats the optimization.
2. Frontend already has optimistic state seeded by `handleAddToCart`. That optimistic state is authoritative for the short window where we're also cache-serving; the UI should trust its own seed, not an empty cache-hit list.

### D2. Why preserve `cartCount` but not `cartItemIds` / `cartItemsById`?

`cartCount` is the header badge (e.g. "🛒 3"). It's safe to update from the authoritative server count. `cartItemIds` / `cartItemsById` drive the per-product UI (stepper); the cache can't contribute to those, so we leave them alone. The optimistic state in `cartItemsById` will be reconciled on the next non-cache fetch (12s later).

### D3. Why not just skip `refreshCartState` after cart-add?

That was Option A in the bug analysis. Rejected because:
- We'd lose server-side verification of the add.
- Network-level retries and reconnection paths rely on `refreshCartState`.
- Skipping it would make the stepper always use optimistic data until the next user-triggered cart open, losing the 12s convergence window that normally catches add-but-server-rejected cases.

### D4. Why not gate the cache on the caller (send a `no_cache=1` query param from `handleAddToCart`)?

Over-engineered. The problem is the UI discarded its own optimistic state; simplest fix is to stop doing that. Adding a query param requires backend and frontend both to know about the opt-out contract — two moving pieces for a one-liner fix.

## Locked Defaults

- Detection guard: `from_cache === true && Array.isArray(items) && items.length === 0 && items_count > 0`. All three must hold; any single failure falls through to normal handling.
- `cartCount` still gets updated on cache hit (header badge truth).
- `cartItemIds` / `cartItemsById` preserved on cache hit (stepper truth).
- Return tuple gains `fromCache: true` on cache-hit short-circuit so callers that care (polling paths, debug logs) can tell.
