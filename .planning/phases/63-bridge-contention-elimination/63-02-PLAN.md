# 63-02: Cart-Items 12-s Cache + Scraper Slots + Smoke Checks

**Delivers:** PERF-06 (cart-items short-circuits basket_recalc during in-flight cart-add), the scraper-side of PERF-07 (3 callsites), and OPS-09 smoke (63-A..D).

**Estimated effort:** ~60 min. ~80-LOC edit in `backend/main.py`, ~5 LOC each in 3 scrapers, ~40 LOC in `scripts/verify_v1.20.sh`.

**Risk:** Medium on `backend/main.py::cart_items_endpoint` (hot path); low on scrapers (wrapped loop only affects latency, not correctness); negligible on the smoke script.

---

## Goal

Close the loop on Phase 63. Cart-add already holds the bridge semaphore (63-01). This plan (a) lets `/api/cart/items` skip `basket_recalc.php` on the contended bridge during the 12 s window after a cart-add success, and (b) wraps the three scraper detail-fetch batches so they yield to cart-add. Ships the Phase-63 smoke checks operators run from their laptop.

## What Was Built

### `backend/main.py::cart_items_endpoint` cache path

Immediately after `_validate_user_header` + the existing Phase 62 nudge, wrapped in an outer `try/except` so a failure in the cache-check path never breaks the endpoint:

1. Lazy-import `is_pending_cache_fresh`, `_emit_event`, `_hash_user_id` from `cart.bridge_semaphore`.
2. Under `_cart_add_attempts_lock`: prune, find all attempts matching `str(a.get("user_id")) == str(user_id)`, take the max by `last_known_cart_at_monotonic` (falling back to `created_at`).
3. Call `is_pending_cache_fresh(latest, now_monotonic)`.
4. On hit: emit `cart_items_cache_hit {user_id_hash, age_ms}`, return:
   ```python
   {
       "items_count": items_count_out,
       "total_price": latest.get("cart_total") or 0,
       "items": items_out,
       "from_cache": True,
   }
   ```
   Note: `cart_items` stored in the attempt record is VkusVill's `totals.Q_ITEMS` integer, not a list, so the cached response returns `items: []` + the authoritative count. The frontend already has the post-cart-add snapshot from the `/api/cart/add` 200 body; the 12 s cache window exists to suppress basket_recalc traffic, not to reconstruct the full cart.
5. On miss: classify reason (`no_record` / `no_snapshot` / `stale`), emit `cart_items_cache_miss`, fall through to the unmodified basket_recalc path below.

### `backend/main.py` two cart-add success sites now stamp `last_known_cart_at_monotonic`

- Site 1 (`cart_add_endpoint`, after upstream `cart.add()` returned `success=True`):
  ```python
  _update_cart_add_attempt(
      pending_attempt["attempt_id"],
      status="success",
      source="cart_add_success",
      last_error=None,
      cart_items=result.get("cart_items"),
      cart_total=result.get("cart_total"),
      last_known_cart_at_monotonic=_monotime_success_fast.monotonic(),
  )
  ```
- Site 2 (`cart_add_status_endpoint`, after polling `basket_recalc` and finding the product):
  ```python
  _update_cart_add_attempt(
      attempt_id,
      status="success",
      ...,
      last_known_cart_at_monotonic=_monotime_success_status.monotonic(),
  )
  ```

Existing `_update_cart_add_attempt` forwards `**fields` directly into the dict (`attempt.update(fields)`), so no helper changes needed.

### Scrapers

**`scrape_green.py::_fetch_stock_per_product`** - wraps the `for p in products:` detail-visit loop with `async with scraper_slot("scrape_green"):` (single acquire per batch, not per product). The return-to-cart-page cleanup stays outside the slot.

**`scrape_red.py::scrape_red_prices_async`** and **`scrape_yellow.py::scrape_yellow_prices_async`** - wrap the 20-iteration pagination loop. The initial single page load and final product extraction happen outside the slot (they're one-shot, low-frequency, and wrapping them would unnecessarily extend bridge-held time).

All three files add `from cart.bridge_semaphore import scraper_slot` at the top of the scrape function (not module-level) so broken imports never block scraper startup.
