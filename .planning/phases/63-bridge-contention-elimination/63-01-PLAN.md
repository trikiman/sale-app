## What Was Built

### `cart/bridge_semaphore.py`

Exports:

- `CART_ADD_IN_FLIGHT : threading.BoundedSemaphore(1)` - the single global semaphore for the shared VLESS SOCKS5 bridge. BoundedSemaphore so a double-release is caught.
- `CART_ITEMS_CACHE_TTL_S = 12.0` and `SCRAPER_BRIDGE_TIMEOUT_S = 10.0` - locked constants from `63-CONTEXT.md`.
- `cart_add_slot()` - sync `contextmanager` that acquires blocking and releases on exit. Guards `ValueError` on release for defensive resilience.
- `scraper_slot(scraper_name, timeout=SCRAPER_BRIDGE_TIMEOUT_S)` - async `asynccontextmanager`. Uses `await asyncio.to_thread(sem.acquire, True, timeout)` so the event loop is not blocked while waiting. Emits `scraper_paused_for_cart_add {scraper, waited_ms, timed_out}` to `data/proxy_events.jsonl` regardless of acquire outcome. On timeout, yields anyway (graceful degradation - scrapers must never starve completely).
- `is_pending_cache_fresh(attempt, now_monotonic=None)` - decision helper for the cart-items cache hit path. Returns True iff attempt has `last_known_cart_at_monotonic` within 12 s AND `cart_items`/`cart_total` are not None.
- Internal helpers `_hash_user_id(uid) -> str` (sha256[:12]) and `_emit_event(event, payload)` (never-raises append to `data/proxy_events.jsonl`).

### `cart/vkusvill_api.py::VkusVillCart.add()`

Added right after the existing Phase 62 `CART_ADD_ACTIVE` flag setter block:

```python
_bridge_sem = None
_bridge_acquired = False
try:
    from cart.bridge_semaphore import CART_ADD_IN_FLIGHT as _bridge_sem
    _bridge_sem.acquire(blocking=True)
    _bridge_acquired = True
except Exception:
    _bridge_sem = None
    _bridge_acquired = False
```

and in the existing `finally:` block (which already clears `CART_ADD_ACTIVE`):

```python
if _bridge_acquired and _bridge_sem is not None:
    try:
        _bridge_sem.release()
    except ValueError:
        pass
```

Phase 62's CART_ADD_ACTIVE flag remains unchanged. Both flag and semaphore live in the same `try/finally`; release order doesn't matter because they protect different consumers (keepalive.warmup watches the flag; scrapers wait on the semaphore).

**Why lazy import inside try:** `cart.bridge_semaphore` is a new module; making its availability a hard dependency of cart-add would risk a boot-time ImportError on a partial deploy. The try/except degrades gracefully - cart-add proceeds without the slot held, which is exactly the pre-Phase-63 behavior.

### `tests/test_bridge_semaphore.py`

7 tests, all passing locally (Python 3.12, pytest 9.0.2, ~0.65 s):

1. `test_cart_add_slot_acquires_and_releases` - non-blocking acquire asserts slot held inside `with`, released after.
2. `test_scraper_slot_waits_then_acquires` - holder thread releases after 0.3 s; scraper_slot waits and reports 0.15-1.5 s elapsed.
3. `test_scraper_slot_timeout_then_proceeds` - holder never releases; scraper_slot times out after 0.3 s and yields anyway, asserting `proceeded is True` and elapsed 0.2-2.0 s.
4. `test_is_pending_cache_fresh_within_12s` - record with timestamp 10 s ago + snapshot -> True.
5. `test_is_pending_cache_fresh_stale_at_12_1s` - record with timestamp 12.1 s ago -> False.
6. `test_is_pending_cache_fresh_missing_snapshot` - timestamp fresh but cart_items/cart_total None -> False.
7. `test_is_pending_cache_fresh_no_record` - None, {}, and record with snapshot but no timestamp -> False.

Autouse fixture `_reset_semaphore` swaps the module-level `CART_ADD_IN_FLIGHT` attribute with a fresh `BoundedSemaphore(1)` between tests so acquire/release state can't leak, and points `PROXY_EVENTS_PATH` at `tmp_path` so the tests don't pollute real `data/`.
