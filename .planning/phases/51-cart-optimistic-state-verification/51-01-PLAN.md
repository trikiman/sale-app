# Plan 51-01: Cart Optimistic State Verification

## Goal

Verify cart-add → quantity stepper flow works end-to-end on production and ensure optimistic cart state is preserved when backend returns source_unavailable.

## Context

This is a gap-closure phase for the v1.13 milestone. The optimistic cart state and quantity stepper were implemented in earlier phases but need verification that:
1. The quantity stepper appears correctly after cart-add success
2. Optimistic cart items aren't overwritten when backend returns source_unavailable fallback

## Tasks

### Task 1: Verify quantity stepper appears after cart-add

<read_first>
- miniapp/src/App.jsx (cart state handling)
- miniapp/src/ProductCard.jsx (stepper rendering condition)
- miniapp/src/CartQuantityControl.jsx (stepper component)
</read_first>

<action>
1. Review the current cart state flow in App.jsx — specifically how `cartStatus` transitions from 'pending' → 'success' and how this affects product card rendering

2. Verify ProductCard.jsx shows CartQuantityControl when:
   - `cartStatus[product.id] === 'success'` OR
   - `product.id` exists in `cartProducts` array

3. Check that the cart state object structure preserves optimistic adds correctly:
   - After add-to-cart click, cartStatus[productId] becomes 'pending'
   - On success response, cartStatus[productId] becomes 'success'
   - CartQuantityControl should render when status is 'success'

4. Document any gaps in the flow where stepper might not appear
</action>

### Task 2: Verify optimistic state preserved on source_unavailable

<read_first>
- miniapp/src/App.jsx (refreshCartState function)
- cart/vkusvill_api.py (backend cart API)
</read_first>

<action>
1. Locate `refreshCartState` or equivalent cart refresh function in App.jsx

2. Verify the logic that handles `source_unavailable` response:
   - When backend returns source_unavailable (VkusVill API temporarily down)
   - The frontend should NOT overwrite optimistic cart items
   - The cart should remain in its current state (showing optimistic adds)

3. Check if there's proper guards:
   ```javascript
   // Expected pattern — only update cart if backend returned actual data
   if (response.data && !response.source_unavailable) {
     setCartProducts(response.data);
   }
   ```

4. If the guard is missing or incomplete, add the protection
</action>

<acceptance_criteria>
- [ ] refreshCartState checks for source_unavailable before overwriting cart
- [ ] Optimistic cart items remain visible when backend is unavailable
- [ ] No "cart cleared" flash when refresh fails
</acceptance_criteria>
