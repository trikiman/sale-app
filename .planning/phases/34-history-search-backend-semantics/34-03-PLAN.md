# Plan 03: Search Completeness Regression Coverage

## Objective

Lock the Phase 34 search contract down with targeted regression coverage so live-sale, history-only, and catalog-only matches keep working as the History page evolves in Phase 35.

## must_haves

- [ ] Automated backend tests cover live-sale, history-only, and catalog-only search matches
- [ ] Tests prove default non-search mode still excludes zero-history catalog rows
- [ ] Tests prove fuzzy fallback respects explicit `group` / `subgroup` constraints
- [ ] Tests prove `/api/groups?scope=all` exposes broader catalog scope than `/api/groups?scope=history`

<action>
Create `backend/test_history_search.py` with concrete fixture coverage for the Phase 34 contract:

1. Build a temporary SQLite database for the test module and point the backend DB path at it for the duration of the tests.
2. Seed `product_catalog` with at least these rows:
   - `live-1` — name contains the search term, `total_sale_count > 0`, belongs to a test group/subgroup
   - `history-1` — name contains the search term, `total_sale_count > 0`, same group/subgroup family, no active sale session
   - `catalog-1` — name contains the search term, `total_sale_count = 0`, belongs to a catalog-only group/subgroup
   - `other-1` — control row that does not match the search term
3. Seed `sale_sessions` so `live-1` is currently active and `history-1` has historical but inactive data.
4. Add `TestClient` cases that assert:
   - `/api/history/products?search=<term>` returns `live-1`, `history-1`, and `catalog-1`
   - `/api/history/products?page=1&per_page=50` without search does NOT return `catalog-1`
   - fuzzy fallback still honors `group` and `subgroup` constraints when the exact term misses
   - `/api/groups?scope=history` excludes the catalog-only group while `/api/groups?scope=all` includes it
5. Keep assertions concrete on returned IDs/fields rather than only counting rows.
</action>
