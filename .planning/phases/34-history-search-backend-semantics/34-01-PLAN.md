# Plan 01: History Search API Contract

## Objective

Tighten the `/api/history/products` backend contract so search-mode queries the full local catalog, preserves the existing mixed-result response shape, and applies fuzzy fallback with the same filter semantics as the exact-match path.

## must_haves

- [ ] Search-mode requests do not apply the default `pc.total_sale_count > 0` dataset gate
- [ ] Exact search and fuzzy fallback use the same `group`, `subgroup`, and explicit `filter` semantics
- [ ] Mixed search results keep the existing History API shape, including live-sale enrichment
- [ ] No remote VkusVill search call is introduced in this phase

## Tasks

<task id="01" title="Make search-mode an explicit full-catalog query path in backend/main.py">
<read_first>
- backend/main.py (current `history_get_products` and `_fuzzy_search_fallback` implementations)
- .planning/phases/34-history-search-backend-semantics/34-CONTEXT.md
- .planning/REQUIREMENTS.md
</read_first>

<action>
Update `backend/main.py` so the History search contract is explicit and consistent:

1. In `history_get_products()`, keep two clearly separated base modes:
   - default history mode when `search` is empty: keep `pc.total_sale_count > 0`
   - search mode when `search` is non-empty after normalization: do NOT add `pc.total_sale_count > 0`
2. Keep search normalization exactly aligned with the existing copied-text handling for non-breaking spaces and curly quotes.
3. Apply optional `category`, `group`, `subgroup`, and explicit `filter` conditions on top of the active base mode rather than mixing hidden history restrictions back into search mode.
4. Extend `_fuzzy_search_fallback(...)` to accept and apply `group` and `subgroup` constraints in addition to `category` and `filter`, then call it with the full active search context from `history_get_products()`.
5. Keep `predicted_soon` as an explicit history-based filter even in search mode; if the user applies it, it is allowed to require historical rows because that restriction is intentional rather than hidden.
6. Preserve the existing response shape for all returned rows: `id`, `name`, `group`, `subgroup`, `total_sale_count`, `is_currently_on_sale`, `last_old_price`, and prediction fields when history exists.
</action>

<acceptance_criteria>
- backend/main.py contains a search-active branch in `history_get_products()` that does not append `pc.total_sale_count > 0`
- backend/main.py defines `_fuzzy_search_fallback(` with `group` and `subgroup` parameters
- `/api/history/products?search=<term>` can return rows with `total_sale_count = 0`
- `/api/history/products?search=<term>` still returns `is_currently_on_sale` for active sale rows
- `predicted_soon` remains an explicit history-based filter path instead of silently changing global search semantics
</acceptance_criteria>
</task>
