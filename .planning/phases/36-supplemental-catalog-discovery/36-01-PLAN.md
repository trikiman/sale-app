# Plan 01: Supplemental Discovery Snapshot

## Objective

Build the core supplemental discovery job: a repeatable seed-query contract plus a sidecar artifact that captures stable product identities and discovery metadata without touching `category_db.json`, `product_catalog`, or runtime search paths.

## must_haves

- [ ] Discovery inputs live in a repo-tracked seed query file
- [ ] Discovery accepts only rows with stable numeric VkusVill `product_id`
- [ ] Discovery deduplicates by `product_id` across all queries/pages
- [ ] Discovery writes a sidecar JSON artifact rather than mutating `category_db.json`
- [ ] Discovery output includes enough metadata for Phase 37 merge work

## Tasks

<task id="01" title="Create the supplemental discovery query contract and sidecar schema">
<read_first>
- .planning/phases/36-supplemental-catalog-discovery/36-CONTEXT.md
- .planning/phases/36-supplemental-catalog-discovery/36-RESEARCH.md
- scrape_categories.py
- scrape_merge.py
</read_first>

<action>
Create a repo-tracked discovery input file at `data/catalog_discovery_queries.json` and define the sidecar artifact contract that Phase 36 will produce.

1. Add `data/catalog_discovery_queries.json` with these top-level keys:
   - `known_gaps`: array of concrete parity-gap queries; include `"цезарь"` as the first known-gap seed
   - `broad_seeds`: array of broader exploratory queries used for additional discovery coverage
   - `disabled`: array for temporarily muted queries without deleting them
2. Keep the file UTF-8 JSON with stable ordering and indentation so diffs stay reviewable.
3. Define the target output schema for `data/catalog_discovery.json` in the implementation:
   - `updated_at`
   - `seed_count`
   - `query_count`
   - `discovered_count`
   - `net_new_count`
   - `products`
4. Each entry in `products` must be keyed by `product_id` and carry these fields:
   - `product_id`
   - `name`
   - `url`
   - `image_url`
   - `group_hint`
   - `subgroup_hints`
   - `source_queries`
   - `source_type`
   - `first_seen_at`
   - `last_seen_at`
5. Do not write any discovery result into `data/category_db.json` or SQLite in this task.
</action>

<acceptance_criteria>
- `data/catalog_discovery_queries.json` exists
- `data/catalog_discovery_queries.json` contains `known_gaps`, `broad_seeds`, and `disabled`
- `data/catalog_discovery_queries.json` contains the string `цезарь`
- `scrape_catalog_discovery.py` or its module-level constants define `data/catalog_discovery.json` as the output path
- the planned `products` record shape includes `product_id`, `source_queries`, and `first_seen_at`
- no code path in this task writes to `data/category_db.json` or `product_catalog`
</acceptance_criteria>
</task>

<task id="02" title="Implement scrape_catalog_discovery.py with stable-ID filtering and per-query dedupe">
<read_first>
- scrape_categories.py
- utils.py
- .planning/phases/36-supplemental-catalog-discovery/36-CONTEXT.md
- .planning/phases/36-supplemental-catalog-discovery/36-RESEARCH.md
</read_first>

<action>
Create `scrape_catalog_discovery.py` as a standalone offline discovery script that mirrors the current scraper conventions while staying isolated from runtime catalog consumers.

1. Reuse the current low-concurrency HTTP pattern:
   - define `MAX_CONCURRENT = 3`
   - reuse VkusVill-like headers
   - use proxy-compatible HTTP transport in the same style as `scrape_categories.py`
2. Load and normalize seed queries from `data/catalog_discovery_queries.json`, skipping anything listed in `disabled`.
3. For every result candidate returned by the discovery source:
   - extract a stable numeric `product_id`
   - discard rows without a stable numeric ID
   - keep `name`, canonical `url`, image hint, and any group/subgroup hint only when present
4. Deduplicate strictly by `product_id`:
   - merge `source_queries` as a unique sorted list
   - preserve the best available `name`, `url`, and `image_url` rather than overwriting non-empty values with blanks
   - update `last_seen_at` on repeat sightings
5. Emit summary stats at save time:
   - total queries processed
   - total products discovered
   - net-new products compared to any existing `data/catalog_discovery.json`
6. Save the result to `data/catalog_discovery.json` using the repo’s existing JSON artifact formatting pattern.
7. Keep the script standalone and executable from the repo root:
   - `python scrape_catalog_discovery.py`
8. Do not call `scrape_merge.py`, `seed_product_catalog()`, or any runtime API endpoint from this script.
</action>

<acceptance_criteria>
- `scrape_catalog_discovery.py` exists
- `scrape_catalog_discovery.py` contains `MAX_CONCURRENT = 3`
- `scrape_catalog_discovery.py` reads `data/catalog_discovery_queries.json`
- `scrape_catalog_discovery.py` writes `data/catalog_discovery.json`
- `scrape_catalog_discovery.py` discards candidates without a stable numeric `product_id`
- `scrape_catalog_discovery.py` merges duplicate discoveries by `product_id` and accumulates `source_queries`
- running `python scrape_catalog_discovery.py` can produce `data/catalog_discovery.json` without invoking `scrape_merge.py` or SQLite writes
</acceptance_criteria>
</task>
