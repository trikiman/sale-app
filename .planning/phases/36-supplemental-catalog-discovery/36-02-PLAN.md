# Plan 02: Discovery Orchestration & Regression Harness

## Objective

Add a thin orchestration path and regression harness so supplemental discovery can be rerun intentionally, observed via logs/stats, and verified against the stable-ID/dedupe contract before Phase 37 merges anything into the main local catalog.

## must_haves

- [ ] Discovery can be rerun intentionally without piggybacking on the main category scraper
- [ ] Orchestration does not auto-merge into `category_db.json` or `product_catalog`
- [ ] Automated tests protect seed loading, stable-ID filtering, dedupe, and artifact stats
- [ ] Discovery remains sidecar-only and local-search runtime behavior stays unchanged

## Tasks

<task id="01" title="Add repeatable orchestration for catalog discovery runs without auto-merge">
<read_first>
- backend/main.py
- scrape_catalog_discovery.py
- .planning/phases/36-supplemental-catalog-discovery/36-CONTEXT.md
- .planning/phases/36-supplemental-catalog-discovery/36-RESEARCH.md
</read_first>

<action>
Expose the new discovery job through the existing backend orchestration pattern without changing current category or runtime search flows.

1. In `backend/main.py`, add a dedicated background-run path for the discovery script, following the same threading/log-buffer style as `/api/admin/run/categories`.
2. Add a new endpoint:
   - `POST /api/admin/run/catalog-discovery`
3. The endpoint must:
   - refuse duplicate concurrent discovery runs using the existing `_run_locks` / `scraper_status` pattern
   - execute `scrape_catalog_discovery.py`
   - stream stdout into `log_buffer`
   - update `scraper_status["catalog_discovery"]`
4. The endpoint must NOT:
   - trigger `scrape_merge.py`
   - rebuild `category_db.json`
   - seed SQLite
   - change `/api/history/products`
5. Return a response shaped like the repo’s existing background-run endpoints:
   - `{"started": true, "message": "..."}`
</action>

<acceptance_criteria>
- `backend/main.py` contains `@app.post("/api/admin/run/catalog-discovery")`
- `backend/main.py` contains `scraper_status["catalog_discovery"]` or equivalent status bucket
- `POST /api/admin/run/catalog-discovery` starts `scrape_catalog_discovery.py` in a background thread
- the endpoint does not call `scrape_merge.py`
- the endpoint does not write to `category_db.json` or SQLite directly
- the endpoint returns JSON with a `started` field
</acceptance_criteria>
</task>

<task id="02" title="Add regression coverage for seed loading, stable IDs, dedupe, and sidecar-only behavior">
<read_first>
- backend/test_categories.py
- backend/test_history_search.py
- scrape_catalog_discovery.py
- backend/main.py
- .planning/phases/36-supplemental-catalog-discovery/36-VALIDATION.md
</read_first>

<action>
Create `backend/test_catalog_discovery.py` to protect the Phase 36 contract.

1. Add tests for seed file loading:
   - `known_gaps`, `broad_seeds`, and `disabled` are read correctly
   - disabled queries do not execute
2. Add tests for stable-ID filtering:
   - candidates with numeric IDs are accepted
   - candidates without stable numeric IDs are dropped
3. Add tests for dedupe by `product_id`:
   - the same product found by multiple queries yields one row
   - `source_queries` accumulates all contributing queries
   - blank metadata does not overwrite richer existing values
4. Add tests for artifact stats:
   - `seed_count`, `query_count`, `discovered_count`, and `net_new_count` are present
5. Add at least one orchestration test for the new admin route:
   - background endpoint starts successfully
   - no auto-merge call is triggered
6. Keep existing `backend/test_history_search.py` behavior unchanged; do not rewrite it for discovery in this phase.
</action>

<acceptance_criteria>
- `backend/test_catalog_discovery.py` exists
- `backend/test_catalog_discovery.py` contains a test covering disabled seed queries
- `backend/test_catalog_discovery.py` contains a test that drops candidates without numeric IDs
- `backend/test_catalog_discovery.py` contains a dedupe test that asserts merged `source_queries`
- `backend/test_catalog_discovery.py` contains an orchestration test for `/api/admin/run/catalog-discovery`
- `pytest backend/test_catalog_discovery.py backend/test_categories.py backend/test_history_search.py -q` exits 0
</acceptance_criteria>
</task>
