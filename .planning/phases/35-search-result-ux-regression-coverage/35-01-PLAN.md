# Plan 01: Mixed Search Result UI States

## Objective

Add clear, search-mode-aware state labels and support text so mixed History search results read as intentional live/history/catalog cards without changing the page layout.

## must_haves

- [ ] Search-active cards clearly distinguish live-sale, history-only, and catalog-only matches
- [ ] Catalog-only matches no longer rely on generic `Нет данных` wording alone
- [ ] The default non-search History page remains visually familiar and is not flooded with extra status noise
- [ ] Result-state logic is extracted into a small reusable helper instead of being buried in JSX branches

## Tasks

<task id="01" title="Extract search result state helper and use it inside HistoryCard">
<read_first>
- miniapp/src/HistoryPage.jsx
- .planning/phases/35-search-result-ux-regression-coverage/35-CONTEXT.md
- .planning/phases/34-history-search-backend-semantics/34-CONTEXT.md
</read_first>

<action>
Create a new helper module `miniapp/src/historySearchState.js` and integrate it into `HistoryPage.jsx`:

1. Export a helper that classifies a product into one of three states:
   - live-sale match
   - history-only match
   - catalog-only match
2. Export a second helper (or enrich the same object) that returns the user-facing label/support copy for the three states.
3. In `HistoryCard`, accept a `searchActive` prop from `HistoryPage` and only show the new state language when `searchActive` is true.
4. Keep the existing sale badge and live dot for live-sale matches, but add an explicit status row/chip so users do not need to infer "currently on sale" from color alone.
5. For history-only matches, add a neutral status label that makes the card read as "seen on sale before" rather than "live right now".
6. For catalog-only matches, replace the purely generic no-data feel with intentional support copy that tells the user the product exists in the catalog but has no recorded sale history yet.
7. Do not change card click behavior, search API usage, sorting controls, or favorites behavior in this task.
</action>

<acceptance_criteria>
- `miniapp/src/historySearchState.js` exists
- `HistoryPage.jsx` imports the new helper
- `HistoryCard` receives a `searchActive` flag and uses it to gate the new state cues
- active search results render distinct user-facing labels for live-sale, history-only, and catalog-only states
- catalog-only cards no longer show only a generic `Нет данных` message without contextual explanation
</acceptance_criteria>
</task>
