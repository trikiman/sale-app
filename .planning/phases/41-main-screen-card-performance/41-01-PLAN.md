# Plan 01: Faster First Useful Content

## Objective

Make the main sale screen feel fast on first open by reducing the fully blocking loading path and refreshing product data in the background.

## must_haves

- [ ] First useful content appears before the current full blocking spinner path
- [ ] Background refresh still reconciles with the latest `/api/products` payload
- [ ] Existing cart/favorite/detail behavior stays intact

## Tasks

<task id="01" title="Add a last-good payload hydration path before the blocking network fetch completes">
<read_first>
- miniapp/src/App.jsx
- .planning/phases/41-main-screen-card-performance/41-CONTEXT.md
</read_first>

<action>
Update `miniapp/src/App.jsx` so the main screen can hydrate from the last good product payload before the blocking `/api/products` fetch completes.

1. Persist the latest successful `/api/products` response (or the minimum safe subset of it) in localStorage.
2. On app start, if a cached payload exists, render it immediately as the first useful content instead of waiting on a fresh network response.
3. Keep the current network fetch, but treat it as background reconciliation when cached data exists.
4. Preserve stale-warning behavior so cached-but-old data is clearly marked rather than hidden.
</action>

<acceptance_criteria>
- App.jsx persists a last-good products payload
- App.jsx can render cached product data before a fresh network response arrives
- background refresh still updates the screen with the latest `/api/products` payload
- stale cached data is clearly marked rather than silently shown as fresh
</acceptance_criteria>
</task>
