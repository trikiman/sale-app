# Plan 02: Card Enrichment Responsiveness

## Objective

Keep product cards responsive while missing weight/detail data is fetched, instead of letting visible-card enrichment create lag spikes.

## must_haves

- [ ] Card rendering remains responsive while enrichment runs
- [ ] Extra detail/weight fetches are throttled or deferred
- [ ] Existing card metadata contract is preserved as much as possible

## Tasks

<task id="01" title="Throttle and defer visible-card enrichment work">
<read_first>
- miniapp/src/App.jsx
- miniapp/src/productMeta.js
- backend/main.py
- .planning/phases/41-main-screen-card-performance/41-CONTEXT.md
</read_first>

<action>
Reduce visible-card enrichment pressure in `miniapp/src/App.jsx`:

1. Replace the current eager visible-card `Promise.allSettled(...)` weight fetch burst with a lower-pressure queue or deferred strategy.
2. Limit concurrent detail/weight fetches so the main thread and network are not hit by multiple card follow-ups at once.
3. Reuse cached resolved weights so the same product does not repeatedly trigger enrichment work.
4. Keep the current card UI contract intact unless the only safe optimization is to omit non-critical enrichment from the main grid.
</action>

<acceptance_criteria>
- App.jsx no longer fires the current eager burst of visible-card weight requests without throttling/defer logic
- repeated renders do not repeatedly enqueue the same product for enrichment
- cards remain interactive while enrichment is pending
- `miniapp/src/productMeta.js` still governs whether enrichment is needed
</acceptance_criteria>
</task>
