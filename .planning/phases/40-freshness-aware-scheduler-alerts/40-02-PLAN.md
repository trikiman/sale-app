# Plan 02: Per-Source Freshness And User Warnings

## Objective

Make freshness explicit for green, red, and yellow, preserve the last valid snapshot per color, and reuse the existing MiniApp banner pattern to warn all users when any color is older than 10 minutes.

## must_haves

- [ ] All colors use a 10-minute stale threshold
- [ ] `/api/products` exposes per-source freshness
- [ ] Last valid snapshots remain visible when a color is stale
- [ ] The MiniApp reuses existing warning UI to warn all users

## Tasks

<task id="01" title="Add per-source freshness payloads to backend products and admin status">
<read_first>
- backend/main.py
- scheduler_service.py
- .planning/phases/40-freshness-aware-scheduler-alerts/40-CONTEXT.md
</read_first>

<action>
Extend `backend/main.py` freshness reporting:

1. Replace the current generic stale-only aggregation in `/api/products` with a per-source freshness object for `green`, `red`, and `yellow`.
2. For each color, expose:
   - `updatedAt`
   - `ageMinutes`
   - `isStale`
   - `status`
3. Keep using the last valid on-disk snapshot for each color; do not hide stale colors.
4. Extend `/admin/status` so operators see the same per-source freshness data and any degraded scheduler/cycle status.
</action>

<acceptance_criteria>
- `/api/products` returns per-source freshness for green, red, and yellow
- `/api/products` marks any color older than 10 minutes as stale
- `/admin/status` exposes the same per-source freshness contract
- stale colors remain present in the returned product data instead of being hidden
</acceptance_criteria>
</task>

<task id="02" title="Reuse the existing MiniApp warning surface for all-color staleness">
<read_first>
- miniapp/src/App.jsx
- backend/main.py
- .planning/phases/40-freshness-aware-scheduler-alerts/40-CONTEXT.md

<action>
Update `miniapp/src/App.jsx` so the existing warning/banner surface warns all users when any color is stale.

1. Consume the new per-source freshness payload from `/api/products`.
2. Reuse the current stale-warning area instead of adding a new alert family.
3. Make the warning name the stale colors explicitly (green/red/yellow) and tell users the data may be outdated.
4. Preserve the current last-valid product display while the warning is visible.
</action>

<acceptance_criteria>
- App.jsx reads per-source freshness from `/api/products`
- the existing warning surface is reused for multi-color staleness
- the warning text identifies which colors are stale
- stale products remain visible under the warning
</acceptance_criteria>
</task>
