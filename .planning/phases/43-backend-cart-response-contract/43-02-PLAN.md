# Plan 02: Pending Add Contract And Attempt Registry

## Objective

Add the backend contract that Phase 44 will consume: ambiguous adds return `pending` quickly, repeated identical adds reuse the same unresolved attempt, and later callers can ask the backend for the final outcome.

## must_haves

- [ ] `/api/cart/add` supports an explicit opt-in path for pending-aware behavior without breaking legacy callers
- [ ] Repeated unresolved adds for the same `user + product` reuse one pending attempt for a short dedupe window
- [ ] The backend exposes an authenticated status lookup route for cart add attempts
- [ ] Reconciliation happens outside the original add request path

## Tasks

<task id="01" title="Add an opt-in pending response mode to /api/cart/add without regressing legacy callers">
<read_first>
- backend/main.py
- cart/vkusvill_api.py
- miniapp/src/App.jsx
- miniapp/src/ProductDetail.jsx

</read_first>

<action>
Extend the backend add contract in `backend/main.py` so Phase 44 can opt in to pending-aware behavior without breaking the current frontend first:

1. Extend `CartAddRequest` with these concrete optional fields:
   - `allow_pending: bool = False`
   - `client_request_id: str | None = None`
2. Keep the current success response shape for successful adds:
   - `success`
   - `cart_items`
   - `cart_total`
3. When `allow_pending` is `False`, preserve the legacy timeout behavior for now so the current frontend still sees the same timeout/error semantics until Phase 44 opts in.
4. When `allow_pending` is `True` and the cart client returns the new pending result, respond with HTTP `202 Accepted` and a JSON body containing:
   - `success: false`
   - `pending: true`
   - `status: "pending"`
   - `attempt_id`
   - `product_id`
   - `user_id`
   - `expires_in_ms`
5. Keep clear hard failures mapped separately:
   - unauthenticated → `401`
   - sold-out / invalid product → `400`
   - upstream unavailable / malformed response → `502` or `500` as appropriate
</action>

<acceptance_criteria>
- backend/main.py defines `allow_pending` and `client_request_id` on `CartAddRequest`
- `/api/cart/add` preserves the current legacy timeout/error path when `allow_pending` is false
- `/api/cart/add` returns HTTP 202 with `pending = true` and `status = "pending"` when `allow_pending` is true and the add outcome is ambiguous
- the 202 response includes `attempt_id`, `product_id`, `user_id`, and `expires_in_ms`
- hard failures remain distinct from pending responses
</acceptance_criteria>
</task>

<task id="02" title="Track unresolved add attempts and expose a reconciliation status route">
<read_first>
- backend/main.py
- cart/vkusvill_api.py
- .planning/phases/43-backend-cart-response-contract/43-CONTEXT.md
- backend/test_cart_pending_contract.py
</read_first>

<action>
Add a short-lived pending-attempt registry in `backend/main.py` and a status lookup route that can reconcile ambiguous adds after the original response has returned:

1. Create an in-memory registry for pending cart attempts keyed by:
   - `attempt_id`
   - `(user_id, product_id)` for dedupe
2. Store these concrete fields per attempt:
   - `attempt_id`
   - `user_id`
   - `product_id`
   - `status`
   - `created_at`
   - `expires_at`
   - `last_error`
   - `client_request_id`
3. Use a dedupe window of roughly `5` seconds for the same `(user_id, product_id)` pair. If an unresolved attempt already exists inside that window, return the same `attempt_id` instead of issuing a second upstream add call.
4. Add an authenticated lookup route:
   - `GET /api/cart/add-status/{attempt_id}`
5. In that lookup route, if an attempt is still `pending`, run one bounded reconciliation check using `get_cart()` outside the original add request path, then update the attempt to one of:
   - `success`
   - `failed`
   - `expired`
6. Return a concrete status payload that includes:
   - `attempt_id`
   - `status`
   - `product_id`
   - `cart_items`
   - `cart_total`
   - `source`
   - `last_error`
</action>

<acceptance_criteria>
- backend/main.py defines a pending-attempt registry keyed by attempt ID and user/product pair
- repeated pending adds for the same user and product within 5 seconds return the same `attempt_id`
- backend/main.py exposes `GET /api/cart/add-status/{attempt_id}`
- the status route can move an attempt from `pending` to `success`, `failed`, or `expired`
- the status payload includes `attempt_id`, `status`, `product_id`, `cart_items`, `cart_total`, `source`, and `last_error`
</acceptance_criteria>
</task>
