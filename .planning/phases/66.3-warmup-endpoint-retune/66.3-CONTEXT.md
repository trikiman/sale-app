## Domain

Research on 2026-05-12 showed Phase 62 picked the wrong warmup endpoint. Live probes on EC2 (through our VLESS bridge, using a real user's stored cookies):

| URL | Authenticated | TTFB (4 samples) | Outcome |
|---|---|---|---|
| `https://vkusvill.ru/personal/` (Phase 62 choice) | yes | **5.3 / 5.4 / 6.3 / 7.4 s** | times out at 5 s in 4/5 scheduler cycles |
| `https://vkusvill.ru/ajax/delivery_order/basket_recalc.php` POST | yes (`user_id`+`sessid`) | **1.6 / 1.7 / 1.9 / 2.3 s** | well under the 3 s PERF-05 p95 budget |
| `https://vkusvill.ru/` anonymous | no | 1.0 s | does not touch session — no warmup value |
| `https://vkusvill.ru/favicon.ico` | no | 0.16 s | static — no session logic |

Live ledger state (`data/warmup_events.jsonl`, 5 most recent events):

```
11:00:23 /personal/ outcome=timeout   latency=5198ms
11:20:18 /personal/ outcome=timeout   latency=5181ms
11:20:23 /personal/ outcome=ok        latency=5002ms   sessid_changed=true
11:40:18 /personal/ outcome=timeout   latency=5288ms
11:40:23 /personal/ outcome=timeout   latency=5251ms
```

**Warmup has a 1-in-5 success rate in production.** Phase 62's entire purpose (keep the session warm to eliminate the ~1.5 s cold-sessid tax on cart-add) is almost never fulfilled because the warmup itself times out before VkusVill finishes rendering `/personal/`.

The v1.20 REQUIREMENTS.md claimed `/personal/` was "~400 ms TTFB" on 2026-05-05. That measurement was either wrong at the time or VkusVill has slowed `/personal/` since. It doesn't matter which — actual production data says use a different endpoint.

## SPEC Lock

LOCKED — planner must NOT re-litigate:

- **Warmup URL changes from** `https://vkusvill.ru/personal/` **to** `https://vkusvill.ru/ajax/delivery_order/basket_recalc.php` **(POST).**
- **Payload:** `user_id={user_id}&sessid={sessid}` — the 2 fields VkusVill needs to authenticate the session. `user_id` and `sessid` are both already present in our stored `cookies.json` (set by `cart/vkusvill_api.py::_persist_session_metadata`). If either is missing for a user, the warmup SKIPS that user (same-behavior-as-before fall-through — don't hammer VkusVill with an unauthenticated request).
- **Headers:** `Content-Type: application/x-www-form-urlencoded; charset=UTF-8`, `X-Requested-With: XMLHttpRequest`, plus the existing Cookie jar already used by Phase 62. Mirrors what `cart/vkusvill_api.py` does for the real cart calls.
- **Timeout stays at 5 s** (existing `WARMUP_TIMEOUT`). Live probes are 1.6–2.3 s so 5 s is generous. No need to widen — if basket_recalc ever DOES take 5+ s that's a real incident worth surfacing as an outage.
- **Success criterion** unchanged from Phase 62: HTTP 200 + non-empty body. basket_recalc returns `{"success":"Y","error":"","basket":{...}}` on success, which satisfies both.
- **Schema of `warmup_events.jsonl` unchanged** — still 8 keys (`timestamp_iso`, `user_id_hash`, `trigger`, `endpoint`, `success`, `outcome`, `latency_ms`, `sessid_changed`). Only the `endpoint` string changes from `"GET /personal/"` to `"POST basket_recalc"`.
- **PERF-03 anti-spam floor (≤1/user/15 min) unchanged.**
- **Fallback**: if `user_id` or `sessid` missing in cookies.json, log `outcome="skipped_missing_session"` and move on. Never 404 VkusVill or hammer with incomplete auth.
- **No test regression**: existing tests in `tests/test_keepalive_warmup.py` monkeypatch `httpx.get` — they'll need updating to monkeypatch `httpx.post` instead. Schema tests stay valid.

## Decisions

### D1. Why `basket_recalc.php` vs other ajax endpoints?

Probed alternatives (most returned HTTP 404 — they don't exist):

- `/ajax/check_auth.php`, `/ajax/basket_count.php`, `/ajax/basket_list.php`, `/ajax/get_user_info.php`, `/ajax/header.php`, `/ajax/menu.php`, `/lk/ajax.php?action=get_basket` — all **404 at 1.1 s** (WAF/router shortcut)
- `/personal/orders/` — **3.6 s** authenticated
- `/personal/favorite/` — not probed (expected similar to orders)
- `/ajax/delivery_order/basket_recalc.php` POST — **1.6–2.3 s** ✓

Also a secondary benefit: `basket_recalc` is the exact endpoint Phase 63 is trying to avoid fighting with `basket_add.php`. Warming it preemptively has the side effect of populating VkusVill's basket cache for the user, so subsequent cart-adds may also be faster. This is strictly additive — worst case no change, best case ~500 ms–1 s off cold-path cart-add p95.

### D2. Don't widen timeout

If we bumped the timeout from 5 s to 10 s, we'd turn timeouts into 6–8 s successes for `/personal/`. But that would mean 100% of the scheduler cycle's warmup budget is spent on a slow path — not what we want. With `basket_recalc` at 2 s we have 3 s of headroom, which is correct.

### D3. Why not POST with empty body

Our cart code sends `data={}` for `basket_recalc` in `get_cart()`. Testing with empty body: would it still return 200? Possibly, but the real cart path sends user_id+sessid to prove it isn't a CSRF leak. Safer to mirror the exact call shape — copy what `VkusVillCart.get_cart()` does.

### D4. Don't change the schema

`warmup_events.jsonl` already has 8 keys with a clear `endpoint` label. We keep it at 8 keys, just change the `endpoint` string. Existing smoke check 62-B asserts the 8 required keys are present and the timestamp is ≤21 min old — it'll keep passing with the new endpoint string. No retroactive JSONL normalization.

### D5. Backward compat: old events in warmup_events.jsonl

Pre-66.3 events show `endpoint: "GET /personal/"`. Post-66.3 events show `endpoint: "POST basket_recalc"`. Ops can distinguish pre/post by this field in any future forensic analysis. No cleanup needed.

- `WARMUP_URL = "https://vkusvill.ru/ajax/delivery_order/basket_recalc.php"`
- `WARMUP_ENDPOINT_LABEL = "POST basket_recalc"`
- `WARMUP_TIMEOUT` unchanged (5 s read/connect)
