## Evidence from Live Probes 2026-05-12

### Why this change — the bug

Before 66.3, `data/warmup_events.jsonl` on EC2 (production) showed:

```
11:00:23 GET /personal/ outcome=timeout   latency=5198ms
11:20:18 GET /personal/ outcome=timeout   latency=5181ms
11:20:23 GET /personal/ outcome=ok        latency=5002ms   sessid_changed=true
11:40:18 GET /personal/ outcome=timeout   latency=5288ms
11:40:23 GET /personal/ outcome=timeout   latency=5251ms
```

1 success in 5 attempts. The warmup was almost never actually warming the session because `/personal/` takes 5-7 s through our bridge and our timeout is 5 s.

### Direct live probes confirm the fix

From `ubuntu@13.60.174.46` through the running xray bridge, using a real user's stored cookies (`data/auth/9166076650/cookies.json`):

| Endpoint | TTFB samples | Avg |
|---|---|---|
| `GET /personal/` authenticated ×4 | 5.3, 5.4, 6.3, 7.4 s | **6.1 s** (times out) |
| `POST basket_recalc.php` authenticated ×4 | 1.6, 1.7, 1.9, 2.3 s | **1.9 s** (succeeds) |

3x speedup on the exact work we care about (session touch on VkusVill's side). `basket_recalc` returns `{"success":"Y","error":"","basket":{...}}` on success — same 200 OK criterion.

### Secondary benefit

`basket_recalc` is the same endpoint `cart/vkusvill_api.py::get_cart` calls. Warming it preemptively also populates VkusVill's basket cache for the user, so the subsequent cart-add hot path can skip some of its cold-cart compute. Additive, not replacement, to PERF-03/04/05.
