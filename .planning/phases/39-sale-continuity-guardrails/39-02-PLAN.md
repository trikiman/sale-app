# Plan 02: Grace Window Session Semantics

## Objective

Update sale-session recording so one bad or short missing window does not split a continuous sale into fake daily re-appearances, and add regression coverage for those decisions.

## must_haves

- [ ] Unhealthy cycles never close active sessions for missing products
- [ ] Healthy absence shorter than 60 minutes keeps the session active
- [ ] Healthy absence of 60 minutes or more closes the session
- [ ] Reappearance before close extends the existing session rather than opening a new one

<action>
Modify `database/sale_history.py:record_sale_appearances(...)` so session closure uses the new cycle-state contract instead of immediate disappearance:

1. Add a helper that reads `data/scrape_cycle_state.json` and defaults to `continuity_safe = false` when the file is missing or invalid.
2. When an active session's `product_id` is missing from `current_products`:
   - if `continuity_safe` is false, keep the session active, preserve the stored `last_seen`, and emit a diagnostic reason code `KEEP_ACTIVE_UNSAFE_CYCLE`
   - if `continuity_safe` is true and `now - last_seen < 60 minutes`, keep the session active and emit `KEEP_ACTIVE_GRACE_WINDOW`
   - if `continuity_safe` is true and `now - last_seen >= 60 minutes`, close the session and emit `CLOSE_CONFIRMED_ABSENCE`
3. When closing after confirmed absence, keep `last_seen` equal to the product's last actual observation timestamp (do not overwrite it with the wall-clock close time) and compute `duration_minutes` from `first_seen -> last_seen`.
4. When a product reappears while its session is still active, extend the existing session rather than opening a new one.
5. When a product reappears after a confirmed close, open a new session and emit `OPEN_CONFIRMED_REENTRY`.
6. Make every keep/close/reopen log line include:
   - product ID
   - sale type
   - decision code
   - missing minutes (when applicable)
   - the cycle-health reason or `continuity_safe` state
</action>

<acceptance_criteria>
- database/sale_history.py reads `data/scrape_cycle_state.json`
- missing products do not close active sessions when `continuity_safe` is false
- missing products do not close active sessions when `continuity_safe` is true but `last_seen` is less than 60 minutes old
- missing products close active sessions only when `continuity_safe` is true and the absence window is at least 60 minutes
- close/keep/reopen code paths emit diagnostic reason codes `KEEP_ACTIVE_UNSAFE_CYCLE`, `KEEP_ACTIVE_GRACE_WINDOW`, `CLOSE_CONFIRMED_ABSENCE`, or `OPEN_CONFIRMED_REENTRY`
</acceptance_criteria>
</task>
