# Plan 01: Cycle Health Snapshot And Admin Visibility

## Objective

Create a pre-merge cycle-health snapshot that records whether red/yellow/green data is trustworthy for continuity decisions, then expose that state through the backend admin contract.

## must_haves

- [ ] Scheduler writes a machine-readable `data/scrape_cycle_state.json`
- [ ] The snapshot is written before `scrape_merge.py` records sale history
- [ ] The snapshot distinguishes `ok`, `warning`, `error`, and `timeout` per source
- [ ] `/admin/status` exposes the latest cycle-health summary without crashing on missing/invalid state

## Tasks

<task id="01" title="Persist continuity-safe cycle state before merge runs">
<read_first>
- scheduler_service.py
- .planning/phases/39-sale-continuity-guardrails/39-CONTEXT.md
- .planning/phases/39-sale-continuity-guardrails/39-RESEARCH.md
- scrape_merge.py
</read_first>

<action>
Update `scheduler_service.py` so every scheduler cycle writes `data/scrape_cycle_state.json` immediately after the red/yellow/green scrapers finish and before `run_script("scrape_merge.py", "MERGE")` is called.

Use this concrete contract:

1. Add a constant for `data/scrape_cycle_state.json`.
2. Capture `cycle_started_at` at the beginning of `run_full_cycle(...)`.
3. After all three color scrapers finish, build a snapshot with:
   - `cycle_started_at`
   - `cycle_finished_at`
   - `continuity_safe`
   - `overall_status`
   - `reasons`
   - `sources.red`
   - `sources.yellow`
   - `sources.green`
4. For each color source store:
   - `status`: one of `ok`, `warning`, `error`, `timeout`
   - `status_text`: the existing human-readable summary string (for example `OK (data updated)` or `WARNING (exit 0 but data NOT updated)`)
   - `file_updated`
   - `data_file`
   - `counted_for_continuity`
5. Set `counted_for_continuity = true` only when the scraper exit path is a genuine success (`code == 0` and `file_updated == true`).
6. Set `continuity_safe = true` only when red, yellow, and green all have `counted_for_continuity = true`; otherwise set `continuity_safe = false`, `overall_status = "degraded"`, and populate `reasons` with concrete values such as `red:error`, `yellow:not_updated`, or `green:timeout`.
7. After merge/notifier complete, refresh the snapshot with merge/notifier status fields if helpful, but do not change the continuity decision that sale history should rely on for that cycle.
</action>

<acceptance_criteria>
- scheduler_service.py defines a path for `data/scrape_cycle_state.json`
- scheduler_service.py writes `continuity_safe`, `overall_status`, `reasons`, and per-source status blocks for red/yellow/green
- the cycle-state write happens before the `run_script("scrape_merge.py", "MERGE")` call
- `counted_for_continuity` is false when a scraper times out, errors, or exits without updating its data file
- `continuity_safe` becomes false when any color source is not counted for continuity
</acceptance_criteria>
</task>

<task id="02" title="Expose cycle state in admin status and lock the route contract with tests">
<read_first>
- backend/main.py
- backend/test_admin_routes.py
- scheduler_service.py
- .planning/phases/39-sale-continuity-guardrails/39-CONTEXT.md
</read_first>

<action>
Extend the admin backend contract so operators can inspect the latest continuity decision without opening raw log files:

1. Add a helper in `backend/main.py` that reads `data/scrape_cycle_state.json`.
2. Extend `/admin/status` to include a `cycleState` object containing:
   - `continuity_safe`
   - `overall_status`
   - `reasons`
   - per-source statuses for `red`, `yellow`, and `green`
3. If the file is missing or invalid JSON, return a non-crashing `cycleState` payload with an explicit `error` field instead of raising.
4. Update `backend/test_admin_routes.py` with a route-level test that writes a temporary cycle-state file, calls `/admin/status`, and asserts the returned payload includes `cycleState.continuity_safe` plus source status fields.
</action>

<acceptance_criteria>
- backend/main.py returns `cycleState` from `/admin/status`
- `cycleState` includes `continuity_safe`, `overall_status`, `reasons`, and source status blocks for red/yellow/green
- missing or invalid cycle-state files do not crash `/admin/status`
- backend/test_admin_routes.py contains an assertion for `cycleState`
- `pytest backend/test_admin_routes.py -q` exits 0
</acceptance_criteria>
</task>
