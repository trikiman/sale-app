# Plan 03: Confirmed Reentry Alerts And Newness APIs

## Objective

Replace the current ever-seen "new products" behavior with confirmed sale-entry semantics so notifier/admin alerts fire only after a true confirmed exit and later return.

## must_haves

- [ ] `backend/notifier.py` no longer treats one transient miss as a true new sale return
- [ ] confirmed reentry after a healthy 60-minute exit becomes eligible as "new" again
- [ ] `/api/new-products` follows the same confirmed-entry semantics as the notifier
- [ ] diagnostic logs make it obvious why a product was or was not treated as newly returned

## Tasks

<task id="01" title="Replace ever-seen sale entry logic with confirmed session-entry helpers">
<read_first>
- backend/notifier.py
- database/db.py
- backend/main.py
- database/sale_history.py
- .planning/phases/39-sale-continuity-guardrails/39-CONTEXT.md
- .planning/phases/39-sale-continuity-guardrails/39-RESEARCH.md
</read_first>

<action>
Align every "new sale item" surface with confirmed session reentry instead of `seen_products`:

1. Add a helper in `database/db.py` (or a shared sale-history helper it calls) that returns current product IDs eligible as confirmed sale entries by using active `sale_sessions` semantics rather than `seen_products`.
2. Define "eligible confirmed sale entry" concretely as:
   - the product belongs to an active sale session
   - that session was opened after a confirmed close under the Phase 39 grace-window rules, or it is the product's first confirmed sale session
   - the product is currently present in the merged products snapshot being evaluated
3. Update `backend/notifier.py:detect_new_products()` to call the new confirmed-entry helper instead of `db.get_new_products(product_ids)`.
4. Update `backend/main.py:/api/new-products` and `backend/main.py:/api/sync` so they no longer compute `new_products` from `db.get_new_products(...)`; they must use the same confirmed-entry helper used by the notifier.
5. Do not delete `seen_products` in this phase, but stop using it as the source of truth for sale-entry alerts.
</action>

<acceptance_criteria>
- backend/notifier.py no longer calls `db.get_new_products(` inside `detect_new_products()`
- backend/main.py no longer uses `db.get_new_products(` in `/api/new-products`
- backend/main.py no longer uses `db.mark_product_seen(` to compute sale-entry counts in `/api/sync`
- a confirmed reentry after a real close is eligible as "new" again
- a reappearance during the grace window is not eligible as a new sale entry
</acceptance_criteria>
</task>

<task id="02" title="Extend tests to prove notifier and API semantics match confirmed reentry behavior">
<read_first>

- backend/test_notifier.py
- backend/main.py
- backend/notifier.py
- .planning/phases/39-sale-continuity-guardrails/39-VALIDATION.md

<action>
Extend the regression harness so alert semantics stay aligned with the repaired session contract:

1. Add tests proving that a product missed during an unsafe cycle or inside the 60-minute grace window is **not** returned as a new sale entry by the confirmed-entry helper.
2. Add tests proving that a product closed after a healthy 60-minute absence and then reappearing is returned again as a confirmed new sale entry.
3. Add route-level coverage for `/api/new-products` so it returns the same confirmed-entry IDs as `backend/notifier.py:detect_new_products()`.
4. Add notifier/API assertions on the diagnostic reason text or reason codes captured for keep/close/reopen decisions so debugging evidence is preserved.
</action>

<acceptance_criteria>
- backend/test_sale_continuity.py asserts grace-window reappearances are not new entries
- backend/test_sale_continuity.py asserts confirmed reentries after close are new entries
- backend/test_notifier.py or backend/test_sale_continuity.py contains a route/API assertion for `/api/new-products`
- the tests assert diagnostic reason codes or reason strings for keep/close/reopen decisions
- `pytest backend/test_sale_continuity.py backend/test_notifier.py -q` exits 0
</acceptance_criteria>
</task>
