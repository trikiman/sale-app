### Phase 74 — Product Details Cold-Path Latency (PERF-10 + PERF-11)

**Commits:** `14fb185`, `6331cc4`, `e871d56` (3 atomic per Plan 74-01/02/03 discipline)

**Scope:**
- Removed the pre-v1.15 SOCKS5-era two-phase probe loop in `backend/main.py::product_details`. Legacy work that duplicated what xray `observatory`+`leastPing` (v1.17), v1.19 pre-flight probe, and v1.21 reprobe daemon already do.
- Tightened httpx timeouts from (connect 4s, read 6s, write 3s, pool 3s) → (1s, 3s, 1s, 1s) to match a healthy VLESS bridge profile. Retries stay at 3.
- Added `backend/detail_events.py` with bounded JSONL ledger.
- Added `backend/test_product_details_latency.py` (6 tests, force-added since `backend/test_*.py` is gitignored).
- Repaired `backend/test_product_details_fallback.py` — it was silently broken since v1.15 because it mocked `requests.get` but the endpoint moved to `httpx.AsyncClient`.
- `scripts/verify_v1.23.sh` with 7 Phase-74 smoke checks.

### Phase 75 — Card Grid Layout Shift Fix (UX-SHIFT-01)

**Commits:** `3668302`, `ea9e166` (1 fix + 1 verification/docs)

**Scope:**
- Added `min-height: 36px` to `.card-price-row` so the row container matches the taller of the two states (cart-btn 36 px vs cart-inline-qty.compact ~28 px).
- Belt-and-braces `min-height: 36px` + `box-sizing: border-box` on `.cart-inline-qty.compact` so the stepper itself can't produce the delta.
- CSS-only change. 6 lines. Zero JSX, zero behavior change.

**Live evidence on production:**

### Phase 76 — Cart Panel Trash Button + Clear-Cart Fallback (UX-CART-01 + UX-CART-02)

**Commits:** `eefba4c`, `599df10` (1 feature + 1 verification/docs)

**Scope:**
- Added `<button className="cart-item-remove-btn">🗑</button>` to every cart-panel row, wired to existing `handleRemove(item.id)` → `POST /api/cart/remove`.
- Appended `.cart-item-remove-btn` CSS (32×32, red-tinted, 8 px left margin from stepper).
- Preserved the existing "minus-becomes-🗑 at quantity ≤ step" shortcut (no regression).
- Fixed `handleClearAll` runtime detection: checks `window.Telegram.WebApp.initData.length > 0` before choosing `showConfirm` vs `window.confirm`. Previously the truthy check of `showConfirm` matched everywhere but the native dialog only fired inside Telegram WebView, leaving desktop Chrome hanging.

**NEEDS_OPERATOR (non-blocking):** live click-through from Telegram on phone. Pattern is the same as Phase 73 v1.22 which shipped as Kiro-side only with local verification, audited later. CSS + code confirmed deployed; the button is wired to a function that has been production-exercised every day since v1.14.
