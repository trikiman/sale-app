# Requirements: v1.13 Instant Cart & Reliability

**Status:** Shipped, audit-closed 2026-04-22
**Scope:** 5 phases (47-51), 8 formal requirements
**Goal:** Make add-to-cart feel instant and fix the failures that made cart adds silently drop on the path that matters to real users.

## Cart Reliability (Phase 47)

- [x] **CART-15**: Cart-add endpoint returns a structured `error_type` field (`auth_expired`, `product_gone`, `transient`, `timeout`, `api`, `unknown`) instead of a generic 500 so the frontend can route typed states.
- [x] **CART-16**: Every cart-add failure path in the backend logs the specific root cause with diagnostic context (proxy result, upstream response summary, session metadata state) so operators can trace why a given attempt was classified the way it was.

## Performance (Phase 48)

- [x] **PERF-01**: On login, `sessid` and `user_id` are pre-extracted and persisted into `cookies.json` alongside a `sessid_ts` timestamp, so the first cart add never blocks on a warmup GET.
- [x] **PERF-02**: Stale `sessid` (older than `SESSID_STALE_SECONDS = 1800`) is auto-refreshed via a bounded warmup GET before it causes a cart failure; `sessid_ts` is rewritten after refresh.

## Error Recovery (Phase 49)

- [x] **ERR-01**: User sees distinct messages for sold-out, session-expired, VkusVill-down, and network-error states instead of a single generic cart error.
- [x] **ERR-02**: After a transient error, user can retry the add in-place without reloading the MiniApp (button returns to `retry` state).

## Cart Optimistic State (Phase 51 — gap closure)

- [x] **CART-17**: After a successful cart add, the quantity stepper (`CartQuantityControl`) appears on the product card in the same render, not only after the next refresh round-trip.
- [x] **CART-18**: `refreshCartState` does not overwrite optimistic cart items when the backend returns `source_unavailable` — the optimistic row is preserved until real truth arrives.

## Out of Scope

- Reverse-engineering a private VkusVill cart API to bypass `basket_add.php` — this milestone stuck to the existing upstream path.
- Cross-tab cart outcome propagation — tracked as future work (v2 requirement `CART-22` under v1.14).
- Formal Playwright or device-timing harness for the hard budget — covered downstream by v1.14 phase 55 live evidence.
