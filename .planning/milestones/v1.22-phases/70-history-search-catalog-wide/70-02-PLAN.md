# 70-02 — Frontend badge uses `currentSaleType` first

**Phase:** 70 History Search Catalog-Wide
**Requirements:** UX-BUG-01
**Depends on:** 70-01 (backend must ship `currentSaleType`)

## Goal

Update `miniapp/src/HistoryPage.jsx` so the top-left type-badge and card border use `product.currentSaleType` when non-null, falling back to `product.last_sale_type` for history-only rows. The live-dot behavior (driven by `is_currently_on_sale`) stays unchanged.

## Change

### `miniapp/src/HistoryPage.jsx`

Locate the per-card render block near line 73:

```jsx
const tc = TYPE_COLORS[type] || TYPE_COLORS.green
const hasSales = product.total_sale_count > 0
const isGhost = !hasSales && !product.is_currently_on_sale
```

`type` is derived earlier from `product.last_sale_type || 'green'`. Change the derivation to prefer live type first.

Find the line that sets `type`:

```jsx
const type = product.last_sale_type || 'green'
```

Replace with:

```jsx
// v1.22 UX-BUG-01: prefer live sale type (from today's proposals.json)
// over historical last_sale_type. Live search results should render
// with their current badge color, matching the main page.
const type = product.currentSaleType || product.last_sale_type || 'green'
```

Also update the `isGhost` condition to match the new reality. Currently:

```jsx
const isGhost = !hasSales && !product.is_currently_on_sale
```

`is_currently_on_sale` is now OR'd with proposals truth, so this already behaves correctly without change.

The badge-text rendering for discount percentage (`avg_discount_pct`) should only show when the product HAS sale history (`hasSales === true`). Products that are live but have no recorded history (fresh catalog seed that just landed on sale) should show the color badge without a percentage — keep the existing guard `{hasSales && product.avg_discount_pct > 0 && (...)}`.
