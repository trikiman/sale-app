### Conclusion

The fix is surgical: filter removes phantom stale-typed products without changing any other response field. Banner, counts, and source metadata stay intact. The user-facing list on the MiniApp now matches VkusVill's own "Зелёных ценников сейчас нет" UX whenever our green source goes stale.

## Phase 66.1 Ledger Notes

- Threshold stays at 10 minutes (inherits from `_build_source_freshness(stale_minutes=10)` — the same threshold used by the existing "Данные устарели" banner). No new constant introduced.
- When all three sources go stale simultaneously (e.g. scheduler fully down >10 min), the `products` list is empty. The banner still fires via the existing `dataStale` + `staleInfo` pipeline so users see the empty-state warning, not a silent empty page.
- `greenLiveCount` (our count of what VkusVill reports existing on their live page) is NOT decremented by the filter. The field documents VkusVill's side, not our side.
- This phase is an intentional late insert into the v1.20 milestone — scope wasn't formally in the 15-requirement list, but it's a UX regression that predates v1.20 and was caught by live observation during the milestone audit. Landed before `/gsd-complete-milestone` per user direction.
