## Domain

Screenshots from the user on 2026-05-12 show:
- VkusVill's own `/promo/` page displays **"Зелёных ценников сейчас нет"** (there are currently no green items).
- Our MiniApp at `vkusvillsale.vercel.app` shows the header "⚠ Данные устарели: зелёные" AND ALSO renders 2 green-typed product cards ("Мусс творожный Малина", "Торт бисквитный") as fully interactive + tappable.

Root cause: when `data/green_products.json` has not been refreshed in >10 min, `_build_source_freshness` flags `sourceFreshness.green.isStale=true` and sets `dataStale=true`. The frontend shows the banner. But the backend `/api/products` response STILL includes stale green-typed products in the `products[]` list. Nothing filters them out. Users tap → `/api/cart/add` → VkusVill returns `product_gone` 410 (if the item was actually removed) or silently adds a discounted item that no longer has that discount on VkusVill's live site.

The banner was designed as an advisory, not an authoritative filter. Users don't read banners — they tap cards. We need the list to match reality.

---

## SPEC Lock

LOCKED — planner must NOT re-litigate:

- **Source of truth for "is this color stale?":** `sourceFreshness.<color>.isStale` as already computed by `_build_source_freshness(stale_minutes=10)` in `backend/main.py`. Threshold stays at 10 min (matches existing banner semantics).
- **Fix surface:** backend-side filter in `/api/products`. The frontend already renders whatever the backend sends; if we filter on the client we end up with two truths and race conditions.
- **Filter shape:** when `sourceFreshness.<color>.isStale === True`, DROP (not gray) every product with `type === <color>` from the `products[]` list in the response.
- **Counts stay truthful:** `greenLiveCount` (what VkusVill's live page reports) is unchanged. The dropped products do NOT decrement `greenLiveCount` — it's our count of what VkusVill says exists, not what we're showing.
- **Banner stays visible:** existing `dataStale=true` + `staleInfo=["green (Xm)"]` + `sourceFreshness.green.isStale=true` fields continue to populate. Banner text is unchanged. The change is that no phantom cards accompany the banner.
- **When ALL three sources go stale:** we drop ALL products, the page shows just banners + empty list. Acceptable: this is the "VkusVill unreachable" scenario and an empty list is more honest than 400 stale cards.
- **Scope guardrails:** NO change to `/api/products` response schema, NO new fields, NO frontend React changes. One function change in `backend/main.py::get_products`.
- **OPS-09 smoke:** 1 new check (66.1-A) in `scripts/verify_v1.20.sh` that asserts `/api/products` with synthetic stale-green state returns 0 green-typed products.
- **OPS-11 rollback rehearsal:** light — single file revert, no data-structure changes.

## Decisions

### D1. Why drop instead of gray?

Grayed-out cards invite taps ("maybe it's back?"). Tapping a stale card → `/api/cart/add` → VkusVill 410 `product_gone` → the exact failure v1.20 Phase 65 was designed to surface more gracefully. Dropping matches VkusVill's own UX on `/promo/` ("Зелёных ценников сейчас нет").

### D2. Why server-side not client-side?

1. Backend already computes `sourceFreshness` — one source of truth.
2. Client-side filter would have to re-implement the 10-min threshold check on every render.
3. Shared behavior between MiniApp + any future clients (admin panel, bot previews).

### D3. Scope: colors only, not cycle freshness

`cycleState` is scheduler-run freshness (last full merge). If the scheduler itself is down but `green_products.json` still has fresh content, we do NOT drop products — the banner already handles that case via `cycleState.sources.green.status`. Only the per-color `sourceFreshness.green.isStale` drives the drop.

### D4. No change to `/api/products` schema

All existing keys (`products`, `sourceFreshness`, `dataStale`, `staleInfo`, `greenLiveCount`, `greenMissing`, `cycleState`, `updatedAt`) keep their shape and semantics. The only behavior change: `products[]` length can shrink when any color is stale.

### D5. Test surface

- Unit test in `backend/test_products_stale_filter.py` (new file):
  - Monkeypatch `_build_source_freshness` to return `green.isStale=True, red.isStale=False, yellow.isStale=False`.
  - Write a synthetic `proposals.json` with 2 green, 1 red, 1 yellow.
  - Call `GET /api/products` via TestClient.
  - Assert response has 2 products (1 red, 1 yellow), no green. Assert `dataStale=True` and `sourceFreshness.green.isStale=True` remain.
- Second test: all three stale → empty products list.
- Third test: none stale → full 4-product list (no regression).

## Locked Defaults

- Stale threshold: 10 minutes (inherits `_build_source_freshness(stale_minutes=10)`).
- Drop applies to products where `p.type in {"green", "red", "yellow"}`. Any other type value passes through untouched (belt-and-suspenders — there are no other types today but the filter must not wreck other types if schema grows).
- Counts in response body (`total`, `greenLiveCount`) unchanged.
