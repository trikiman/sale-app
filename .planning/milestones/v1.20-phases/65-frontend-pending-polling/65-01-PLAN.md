# 65-01: Backend `client_request_id` Idempotency + `/api/cart/add-status-by-client-id/{cri}` Endpoint

**Delivers:** UX-03 (frontend retry with same client_request_id never produces a second basket_add.php call to VkusVill).

**Risk:** Low. Adds a secondary index + a delegating endpoint. The new idempotency gate short-circuits BEFORE the existing pair-dedupe so old behavior is preserved for callers that omit `client_request_id`.

## Goal

Phase 65 UX-01/UX-02 asks the frontend to retry a stalled cart-add POST via `/api/cart/add-status-by-client-id/{cri}`. That only works if the backend guarantees:

1. Same `client_request_id` from two concurrent POSTs maps to the same `attempt_id`.
2. A GET by `client_request_id` returns the same payload as a GET by `attempt_id`.

This plan implements both.

## Implementation

### `backend/main.py`

1. Add module-level secondary index right next to `_cart_add_attempt_index`:
   ```python
   _cart_add_attempt_by_client_id: dict[str, str] = {}
   ```
   Guarded by the existing `_cart_add_attempts_lock`. No new lock.

2. In `_prune_cart_add_attempts(now)`: after popping an attempt out of `_cart_add_attempts`, also drop its `client_request_id` from the new index if present.

3. In `_get_or_create_pending_cart_attempt(user_id, product_id, client_request_id)`: insert a FIRST gate that checks the `_cart_add_attempt_by_client_id` index. If a hit exists and the attempt is still alive, return `(existing.copy(), False)` with `source="client_request_id_dedupe"`. This gate runs BEFORE the existing `(user_id, product_id)` pair dedupe so same-client_request_id always wins. On create, also register `_cart_add_attempt_by_client_id[cri] = attempt_id`.

4. Add new endpoint:
   ```python
   @app.get("/api/cart/add-status-by-client-id/{client_request_id}")
   def cart_add_status_by_client_id_endpoint(client_request_id: str, request: Request):
       # Lock, prune, look up attempt_id via the secondary index, 404 if missing,
       # then delegate to cart_add_status_endpoint(attempt_id, request) so status
       # semantics (pending -> success/failed/expired transitions) remain the
       # single source of truth in cart_add_status_endpoint.
   ```

### `backend/test_cart_idempotency.py`

New test file with 3 tests using the existing TestClient + monkeypatched `VkusVillCart` pattern from `backend/test_cart_pending_contract.py`:

- `test_client_request_id_dedupe_returns_same_attempt` — two POSTs with same cri → same attempt_id, 1 VkusVill call.
- `test_client_request_id_different_creates_new_attempt` — two POSTs with different cris → 2 different attempt_ids, 2 VkusVill calls.
- `test_cart_add_status_by_client_id_lookup` — create pending attempt, GET by cri returns it, GET by unknown cri returns 404.
