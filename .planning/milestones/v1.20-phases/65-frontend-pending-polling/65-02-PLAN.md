# 65-02: Frontend 8s to 5s + Polling on AbortError (UX-01, UX-02)

**Delivers:** UX-01 (poll eventual resolution instead of false-fail), UX-02 (tighter 5s abort + 15s total SLA).

**Risk:** Medium. The polling block is the user-visible safety net. A bug that locks the UI in 'pending' or fires the same success handling twice would regress cart correctness. Mitigations: budget-gate with `performance.now() - t0 < 15000`, superseded-check via `pendingCartAttemptsRef`, explicit state cleanup in every exit branch.

## Implementation

In `miniapp/src/App.jsx::handleAddToCart`:

1. `setTimeout(() => controller.abort(), 8000)` becomes `setTimeout(() => controller.abort(), 5000)`.

2. Move `clientRequestId` generation BEFORE the fetch (currently line 938 is inside `try`; move to the `const timer = ...` region so the AbortError catch can read it).

3. In `catch (err) { if (err.name === 'AbortError') { ... } }`: replace the existing "show retry toast" with a polling loop:
   - Seed `pendingCartAttemptsRef.current.set(String(pid), clientRequestId)`.
   - Show "Proveryaem korzinu..." info toast (NOT error).
   - Loop while `performance.now() - t0 < 15000`:
     - `await sleep(1000)`, increment `pollIteration`.
     - If `pendingCartAttemptsRef.current.get(String(pid)) !== clientRequestId`, return (superseded).
     - `fetch('/api/cart/add-status-by-client-id/' + encodeURIComponent(clientRequestId), { headers })`.
     - 404 -> continue (attempt not created yet).
     - 401 -> auth-expired handler.
     - non-OK -> continue.
     - `data.status === 'pending'` -> continue.
     - `data.status === 'success'` -> inline the same cart-state updates the fast path does + success toast. Delete pending ref. Return.
     - `data.status === 'failed' | 'expired'` -> error toast (sold-out vs generic). Return.
   - Budget exhausted -> "Dobavlyaem v fone" info toast. Return. No error toast.

4. Preserve all existing non-AbortError catch-branch behavior (network error handler unchanged).
