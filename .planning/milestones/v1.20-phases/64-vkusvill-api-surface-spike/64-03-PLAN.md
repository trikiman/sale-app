# 64-03: NEEDS_OPERATOR - HAR Capture + Live Ablation + Go/No-Go Decision

**Delivers:** the live portion of PERF-08 (HAR surface discovery) and PERF-09 (ablation-measured payload trim) plus OPS-10 (p95 gate) and OPS-11 (rollback rehearsal). Operator-driven; this plan has no autonomous step.

**Estimated effort:** 2-4 hours wall clock (plus 15-30 min for ablation run + EC2 deploy + smoke).

**Risk:** Medium. Live network calls against VkusVill from an operator-controlled account. A GO decision also means touching the cart-add hot path; mitigated by the feature flag staying OFF until the EC2 smoke passes.

---

## Goal

Turn the 64-01/64-02 scaffolding into a shipped PERF-08/PERF-09 outcome: either (a) swap to a faster endpoint behind the feature flag, or (b) document that no faster endpoint exists and trim the 16-field payload to minimum. Either outcome closes Phase 64.

---

## Success Criteria (operator-filled)

1. `.planning/research/v1.20-API-SPIKE.md` Section A has a note referencing the captured HAR (path outside the repo, or `.gitignored` path inside the repo).
2. Section B has the baseline row plus at least one candidate-endpoint row. If the HAR shows only `basket_add.php`, Section B states "no candidate endpoints found in HAR" explicitly.
3. Section C has one ablation-JSON output per reachable endpoint. Each JSON is stored in `.planning/research/ablation-<slug>.json` and committed (ablation JSON is safe; no personal data - only user_id which is a VkusVill numeric ID, not a phone or email).
4. Section D has an explicit GO / NO-GO / REJECT decision with operator sign-off (name + UTC timestamp).
5. Section E has the trimmed payload field set with before/after latency.
6. If GO: `FAST_CART_ADD_URL` is populated, the fast-path body replaces the `NotImplementedError`, a new pytest in `tests/test_cart_feature_flag.py` (or an adjacent file) asserts the fast path is taken when the flag is on. Full `pytest tests/ -q` still at baseline + new tests green.
7. If GO: `scripts/verify_v1.20.sh 64` on EC2 passes (all 3 checks). `scripts/verify_v1.19.sh all` still 24/24 green.
8. `64-VERIFICATION.md` filled with deploy log (if GO), live p95 numbers, rollback rehearsal evidence, full smoke-script output.

## Operator Runbook

### Step 1 - Capture HAR

Follow `.planning/research/v1.20-API-SPIKE.md` Section A:
- Launch Chrome with `--proxy-server=socks5://127.0.0.1:10808`.
- DevTools Network tab, Preserve log on, Disable cache on, No throttling.
- Log into `vkusvill.ru` with the test account.
- Open any product detail. Clear network log. Tap "Add to cart".
- Wait 3 seconds for follow-up AJAX. Right-click, Save all as HAR with content.
- Store at a path not tracked by git (HAR contains personal sessid).

### Step 2 - Fill Section B (Endpoint Table)

For each candidate endpoint observed in the HAR:
1. Reproduce the request via `httpx` through the xray bridge (`socks5h://127.0.0.1:10808`) using the sessid from `data/browser_cookies_live.json`.
2. Run 50 sequential calls (sleep 250 ms between).
3. Record p50 / p95 / p99.
4. Classify reachability (reachable / auth-rejected / blocked).

### Step 3 - Fill Section C (Ablation Evidence)

For each reachable endpoint from Step 2:

```bash
python scripts/ablate_basket_add_payload.py \
  --user-id <test-user-id> \
  --product-id <low-risk-test-product-id> \
  --n-per-field 20 \
  --output .planning/research/ablation-<endpoint-slug>.json
```

Paste output path + baseline success + `verdict.droppable` / `verdict.required` into Section C of the research doc.

### Step 4 - Decide Go/No-Go (Section D)

Apply the 4-case decision tree. Write a 2-3 sentence rationale + name + UTC timestamp.

### Step 5a - If GO: implement + ship the swap

1. Set `FAST_CART_ADD_URL = "https://vkusvill.ru/..."` in `cart/vkusvill_api.py`.
2. Replace the `NotImplementedError` in the feature-flag gate with the fast-path implementation (new method `_add_fast(...)` mirroring the existing legacy flow but targeting `FAST_CART_ADD_URL` with the minimum payload from Section E).
3. Add a pytest that asserts the fast path is taken when the flag is on (mirror `test_legacy_path_when_flag_off` but with flag=True + URL set).
4. Deploy to EC2, DO NOT flip `USE_FAST_CART_ADD_ENDPOINT=1` yet.
5. Run `scripts/verify_v1.20.sh 64` (flag still off; 64-A confirms default-off, 64-B trips because URL is no longer None; update the smoke check to match the post-GO state OR replace 64-B with a "swap landed, URL populated" positive assertion).
6. Run `scripts/verify_v1.19.sh all`; must stay 24/24.
7. Only after (5) and (6): set `USE_FAST_CART_ADD_ENDPOINT=1` in the systemd drop-in for `saleapp-backend` + `saleapp-scheduler`. Restart services. Measure cart-add p95 over 50 synthetic adds.

### Step 5b - If NO-GO: ship payload trim only

1. Edit `cart/vkusvill_api.py::add()`'s 16-field dict literal to contain only the fields listed as `required` in Section E.
2. Add a regression test in `tests/test_vkusvill_cart.py` that assembles the payload in `add()` (mocking `_request` to capture it) and asserts `set(payload.keys()) == EXPECTED_KEYS`.
3. Deploy + smoke as in Step 5a (omit the flag flip).

### Step 5c - If REJECT: document + move on

Only fill Section D and 64-VERIFICATION.md; no code change. Leaves the feature-flag scaffolding as dormant infrastructure for a future milestone.

### Step 6 - Rollback Rehearsal (OPS-11)

On a throwaway worktree:
1. `git revert` the GO / NO-GO commit.
2. `python -c "from cart.vkusvill_api import VkusVillCart; print('ok')"` must succeed.
3. `python -m pytest tests/ -q` still green at the pre-64 baseline.
4. `bash scripts/verify_v1.20.sh 64` returns the Phase-64 block to pre-GO state.

### Step 7 - Fill 64-VERIFICATION.md

Paste the smoke-script outputs, the p95 numbers, the rollback rehearsal log, and the v1.19 regression gate outcome.

## Rollback

For the live swap (if GO landed):

```bash
# 1. Flip the flag off first (fastest mitigation)
ssh "$EC2_HOST" "sudo sed -i 's/USE_FAST_CART_ADD_ENDPOINT=1/USE_FAST_CART_ADD_ENDPOINT=0/g' /etc/systemd/system/saleapp-backend.service.d/*.conf"
ssh "$EC2_HOST" "sudo systemctl daemon-reload && sudo systemctl restart saleapp-backend saleapp-scheduler"
# 2. If the code is genuinely broken (not just the flag), revert the GO commit
ssh "$EC2_HOST" "cd /home/ubuntu/saleapp && git revert <64-03-sha> && git push origin HEAD"
```

For a NO-GO payload-trim deploy:

```bash
ssh "$EC2_HOST" "cd /home/ubuntu/saleapp && git revert <64-03-sha> && sudo systemctl restart saleapp-backend saleapp-scheduler"
```
