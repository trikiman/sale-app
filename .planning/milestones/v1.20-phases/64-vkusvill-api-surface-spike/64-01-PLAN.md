## What Was Built

### `cart/vkusvill_api.py`

Two module-level constants near `SESSID_STALE_SECONDS`:

```python
USE_FAST_CART_ADD_ENDPOINT = os.environ.get("USE_FAST_CART_ADD_ENDPOINT", "0") == "1"
FAST_CART_ADD_URL = None  # populated in Phase 64 if spike identifies a faster endpoint
```

In-code comment explains the operator flow: HAR capture -> ablation -> fill research doc -> populate URL -> flip flag only after 64-smoke passes on EC2. Until then, flipping the flag while URL is None is a no-op; flipping the flag while patching a URL raises `NotImplementedError`, which is the intentional guard against a half-configured deploy.

Gate inside `VkusVillCart.add()`, placed immediately after the existing session check (before the legacy `basket_add.php` loop):

```python
if USE_FAST_CART_ADD_ENDPOINT and FAST_CART_ADD_URL:
    raise NotImplementedError(
        "Phase 64 fast cart-add path is not yet implemented. "
        "Spike (64-03) must populate FAST_CART_ADD_URL and wire "
        "the fast path before USE_FAST_CART_ADD_ENDPOINT=1 is "
        "safe to flip. See .planning/research/v1.20-API-SPIKE.md."
    )
```

When the flag is False (default), this branch is a cheap short-circuit; the legacy path runs byte-for-byte as on the Phase-63 branch tip.

### `tests/test_cart_feature_flag.py`

Three tests, all passing locally (Python 3.12, pytest 9.0.2):

1. `test_fast_endpoint_disabled_by_default`: clean env plus `importlib.reload(cart.vkusvill_api)` asserts `USE_FAST_CART_ADD_ENDPOINT is False` and `FAST_CART_ADD_URL is None`.
2. `test_fast_endpoint_raises_when_flag_on_but_url_none`: sets env var to "1", reloads the module, monkey-patches `FAST_CART_ADD_URL` to a dummy non-None string so the gate condition is truthy, mocks `_ensure_session` to a no-op, asserts `cart.add(product_id=731)` raises `NotImplementedError` with "Phase 64" in the message.
3. `test_legacy_path_when_flag_off`: default flag=False, mocks `_ensure_session` and `_request` to return a synthetic VkusVill success payload, asserts `cart.add()` succeeds AND that the mocked `_request` was called with `url == BASKET_ADD_URL` AND the payload contains `id`, `user_id`, and `sessid` fields. Pins the legacy path so a future refactor that dispatches to a different URL or drops required fields trips this test first.

All three rely on `importlib.reload` so env-var changes take effect in the module's `os.environ.get(..., "0") == "1"` initializer.

### `.planning/research/v1.20-API-SPIKE.md`

Six sections, with A/B/C/D/E marked `NEEDS_OPERATOR` for Phase 64-03 fill-in:

- **A: HAR Capture Methodology**: preconditions (RU test account + clean Chrome + socks5 bridge), capture procedure (8 steps), evidence-to-extract list, candidate endpoint hypotheses (`/ajax/quick_add.php`, `/api/cart/add`, miniapp/telegram variants).
- **B: Endpoint Table**: empty rows operator fills with URL, method, auth requirements, payload schema, response schema, p50/p95/p99 measured through bridge, reachability classification.
- **C: Per-Endpoint Ablation Evidence**: command template + reporting table rows filled by running the Phase 64-02 ablation harness against each reachable endpoint.
- **D: Go/No-Go Recommendation**: 4-case decision tree (>= 30% faster + reachable + no blocking risk -> GO; otherwise NO-GO variants).
- **E: Payload Minimization Evidence**: final field set + latency delta placeholders + regression-test placement guidance.
- **F: Scaffolding Inventory (LOCAL, completed 2026-05-10)**: exhaustive list of exactly what this plan shipped so future operators can orient.
