# 62-03: EC2 Deploy + Live Verification + Rollback Rehearsal

**Delivers:** PERF-03/04/05 live evidence on EC2, OPS-09 (smoke executed on live host), OPS-10 (p50/p95/p99 cart-add measurement against v1.19 baseline, pass only if p95 does not regress > 500 ms), OPS-11 (rehearsed rollback before any merge to main).

**Estimated effort:** 120 minutes — baseline capture pre-deploy (15 min), deploy (10 min), observation of one full warmup cycle (20 min real-time), smoke run (5 min), post-deploy latency capture (15 min), rollback rehearsal on throwaway worktree (30 min), evidence write-up (25 min).

**Risk:** Medium — this is the only Phase 62 plan that touches production. Mitigated by:
- Baseline captured BEFORE deploy (OPS-10 has a true comparison point).
- Rollback rehearsed on a throwaway worktree BEFORE the merge.
- Live rollback is a single `git revert` + `systemctl restart`, identical shape to the rehearsal.
- No database migrations. Only new persistent state is `data/warmup_events.jsonl` (append-only, safe to delete).

***

## Goal

Land Phase 62 on the live EC2 scraper host, prove the keep-alive mechanism actually reduces cart-add latency in production conditions, and document a rehearsed rollback path before the phase is considered shipped.

This plan is `autonomous: false` because two checkpoints require operator confirmation:

- **Checkpoint A:** after baseline latency capture, BEFORE deploy — operator confirms baseline numbers look sane (not polluted by an outage).
- **Checkpoint B:** after post-deploy smoke + latency capture, BEFORE merging 62-01+62-02 to main — operator confirms numbers meet ROADMAP success criteria (5 smokes green, p95 <= 6 s, no v1.19 regression).

The plan produces no code changes on the production branch — only `62-VERIFICATION.md` + latency CSVs are committed.

Honors OPS-09 (smoke script on live host), OPS-10 (p95 regression gate), OPS-11 (rollback rehearsed pre-merge).

***

## Success Criteria (observable / testable)

1. `scripts/verify_v1.20.sh 62` on the operator's local machine (SSH to EC2) returns exit code 0 with all 5 Phase-62 checks green.
2. `scripts/verify_v1.19.sh all` on the operator's local machine returns exit code 0 with 24/24 green AFTER 62-01+62-02 deploy.
3. `data/warmup_events.jsonl` on EC2 shows >= 1 complete warmup cycle (one line per linked user, majority `outcome: "ok"`) within 25 min of service restart.
4. Cart-add p50/p95/p99 sampled over 50 attempts on EC2 (methodology in Step 7) meets:
   - p95 <= 6 s on warm-sessid path (Phase 62 ROADMAP success criterion #4).
   - p95 regression vs v1.19 baseline <= 500 ms (OPS-10).
5. Warmup per-attempt p95 from JSONL <= 3 s (PERF-05 budget).
6. Rollback rehearsal on a throwaway worktree: `git revert <range>` + `python -m pytest tests/ -q` green + `scripts/verify_v1.19.sh all` 24/24 green. Evidence captured in 62-VERIFICATION.md.
7. `62-VERIFICATION.md` exists, is committed, contains: deploy command log, smoke output, baseline+post-deploy latency tables, warmup histogram, rollback evidence, and a go/no-go stamp.

### Step 2 — Capture v1.19 baseline cart-add latency (pre-deploy, OPS-10 anchor)

Drive 50 synthetic cart-adds via the Vercel public edge (matches real-user path):

```bash
# Operator's local machine
mkdir -p .planning/phases/62-sessid-keepalive-warmup/latency
LAT_FILE=".planning/phases/62-sessid-keepalive-warmup/latency/baseline_v1.19.csv"
echo "attempt,status,elapsed_ms" > "$LAT_FILE"

: "${SMOKE_USER_ID:?set SMOKE_USER_ID to a linked test telegram_id}"
: "${SMOKE_PRODUCT_ID:?set SMOKE_PRODUCT_ID to a VkusVill product id that is in stock}"
: "${VERCEL_BASE:=https://vkusvillsale.vercel.app}"

for i in $(seq 1 50); do
    START=$(date +%s%3N)
    HTTP=$(curl -s -o /dev/null -w "%{http_code}" --max-time 15 \
        -X POST "$VERCEL_BASE/api/cart/add" \
        -H "X-User-Id: $SMOKE_USER_ID" \
        -H "Content-Type: application/json" \
        -d "{\"user_id\":\"$SMOKE_USER_ID\",\"product_id\":$SMOKE_PRODUCT_ID,\"client_request_id\":\"baseline-$i-$(date +%s)\"}")
    END=$(date +%s%3N)
    echo "$i,$HTTP,$((END-START))" >> "$LAT_FILE"
    sleep 5
done

python3 - <<'PY'
import csv, statistics
rows = [int(r['elapsed_ms']) for r in csv.DictReader(open('.planning/phases/62-sessid-keepalive-warmup/latency/baseline_v1.19.csv')) if r['status']=='200']
if len(rows) < 40:
    raise SystemExit(f'FAIL: only {len(rows)}/50 successful — retry later')
rows.sort()
print(f'v1.19 BASELINE (n={len(rows)}): p50={rows[len(rows)//2]}ms p95={rows[max(0,int(len(rows)*0.95)-1)]}ms max={max(rows)}ms')
PY
```

Record the printed p50/p95/max in `62-VERIFICATION.md` under `## v1.19 Baseline`.

**Checkpoint A:** operator reviews the baseline numbers. If p95 is wildly off from recent history (> 2x typical), the EC2 environment is probably in a degraded state — DO NOT deploy until it recovers. Proceed only if baseline looks representative.

### Step 7 — Post-deploy cart-add latency measurement (OPS-10 + ROADMAP #4)

Repeat Step 2's procedure, writing to `latency/post_62.csv`. Same 50 attempts, 5 s spacing.

Compute the comparison:

```python
# Run this locally
import csv
def pct(data, p):
    return sorted(data)[max(0, int(len(data)*p) - 1)]
base = [int(r['elapsed_ms']) for r in csv.DictReader(open('.planning/phases/62-sessid-keepalive-warmup/latency/baseline_v1.19.csv')) if r['status']=='200']
post = [int(r['elapsed_ms']) for r in csv.DictReader(open('.planning/phases/62-sessid-keepalive-warmup/latency/post_62.csv')) if r['status']=='200']
b50, b95, b99 = pct(base,0.50), pct(base,0.95), pct(base,0.99)
p50, p95, p99 = pct(post,0.50), pct(post,0.95), pct(post,0.99)
print(f'           {"p50":>8}{"p95":>8}{"p99":>8}')
print(f'baseline   {b50:>8}{b95:>8}{b99:>8}')
print(f'post-62    {p50:>8}{p95:>8}{p99:>8}')
print(f'delta      {p50-b50:>+8}{p95-b95:>+8}{p99-b99:>+8}')
print('VERDICT:',
      'PASS' if (p95 <= 6000 and p95 - b95 <= 500) else
      f'FAIL (p95={p95}ms, baseline={b95}ms, delta={p95-b95}ms; target <=6000ms, delta <=+500ms)')
```

Write the table + VERDICT into `62-VERIFICATION.md` under `## Cart-Add Latency (OPS-10 + ROADMAP #4)`.

### Step 8 — Warmup latency histogram (PERF-05)

```bash
ssh "$EC2_HOST" "python3 - <<'PY'
import json
lats = []
with open('/home/ubuntu/saleapp/data/warmup_events.jsonl') as f:
    for ln in f:
        try:
            d = json.loads(ln)
            if d.get('outcome') == 'ok':
                lats.append(int(d['latency_ms']))
        except Exception:
            pass
if not lats:
    print('NO_OK_ATTEMPTS'); raise SystemExit(0)
lats.sort()
p50 = lats[len(lats)//2]
p95 = lats[max(0, int(len(lats)*0.95)-1)]
p99 = lats[max(0, int(len(lats)*0.99)-1)]
print(f'n={len(lats)} p50={p50}ms p95={p95}ms p99={p99}ms max={max(lats)}ms')
print('PERF-05 p95 budget (<=3000ms):', 'PASS' if p95 <= 3000 else 'FAIL')
PY"
```

Record in `62-VERIFICATION.md` under `## Warmup Latency (PERF-05)`.
