# Requirements: v1.12 Add-to-Cart 5s Hard Cap

**Status:** Shipped
**Scope:** Single-phase milestone (Phase 46)
**Goal:** Guarantee a hard user-visible time budget on every add-to-cart tap so the UI never hangs past the budget, with a clear background-recovery message when upstream is too slow to finish inline.

## Hard Cap on Visible Wait

- [x] **CART-10**: Tap add-to-cart and see a final success/error or explicit background state within the hard time budget (originally 5s; relaxed to 8s in v1.13 after live tuning).
- [x] **CART-11**: The initial `POST /api/cart/add` fetch uses an AbortController with a timer bound to the budget, so no request can outlive the budget without aborting.
- [x] **CART-12**: The pending-attempt poll loop uses a remaining-time budget (not a fixed iteration count), and each individual poll fetch is capped by a per-poll AbortController bounded to the remaining budget.
- [x] **CART-13**: If the poll status endpoint returns 404, the poll loop stops immediately instead of retrying the missing attempt.
- [x] **CART-14**: Backend pending-attempt TTL (from v1.11 phase 43, 30s) stays longer than the frontend budget, so the frontend cannot outrun the backend's state.

## Out of Scope

- Changing the VkusVill upstream response time.
- Redesigning the product card or cart panel layout.
- Cross-tab cart outcome propagation (listed as future follow-up CART-10/OPS-05 under v1.11 archive).
