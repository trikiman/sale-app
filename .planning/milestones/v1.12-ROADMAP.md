### Phase 46: Enforce 5s Cart Budget

**Goal:** Cap the visible add-to-cart interaction at 5 seconds with explicit background handoff when upstream is too slow.

**Depends on:** v1.11 pending-attempt contract (phase 43 backend `202 pending` + `attempt_id` polling).

**Success Criteria:**

1. User taps add-to-cart and sees success or error within 5 seconds. *(Budget later tuned to 8 s in v1.13 — architectural guarantee preserved.)*
2. If the initial `POST /api/cart/add` fetch takes longer than 4 s and returns `202 pending`, the user sees "Добавляем в фоне" instead of entering the poll loop. *(Gate later tuned to 7 s in v1.13.)*
3. The poll loop exits as soon as the remaining budget drops below 800 ms.
4. The poll loop stops immediately if the status endpoint returns `404`.
5. Backend pending-attempt TTL (30 s) stays comfortably longer than the frontend budget.

## Key Decisions

| Decision | Rationale | Outcome |
|----------|-----------|---------|
| AbortController as the only source of truth for the cap | Single abort signal makes timeout behavior inspectable and idempotent; no race between multiple timers. | ✓ Good |
| Separate per-poll AbortController, capped to `min(1500, remaining-100)` ms | Any single slow poll cannot starve the remaining budget. | ✓ Good |
| D3 gate at 4 s before entering poll loop | If initial fetch already burned most of the budget, polling is guaranteed to time out — show "Добавляем в фоне" instead. | ✓ Good (tuned to 7 s in v1.13) |
| 5-second hard cap rather than aligning with backend 30 s TTL | User-facing wait budget must be tighter than upstream recovery window. | ✓ Good (tuned to 8 s in v1.13) |
| Keep all existing `[CART-ADD]`/`[CART-POLL]` console logs | Diagnostic visibility during budget tuning was deliberately preserved. | ✓ Good |
