# Plan 03: Synced In-Cart Controls

## Objective

Turn confirmed in-cart products into the persistent VkusVill-like quantity control on every visible surface, with typed input for `шт` and `кг`, and without treating pending as sold-out or hard failure.

## must_haves

- [ ] Confirmed in-cart products switch from the plain add button to a persistent quantity control
- [ ] The same product stays visually synced between the product card and the detail drawer
- [ ] `шт` supports integer entry and weighted items support decimal entry
- [ ] Confirmed-only failure treatment is preserved: pending is neutral, sold-out/failure only after confirmation

<action>
Update the product-card and detail-drawer UI so confirmed in-cart products use the persistent quantity control:

1. Replace the current plain `cartState`-driven card button in `miniapp/src/App.jsx` with a branch that can render:
   - normal add button
   - pending/checking state
   - confirmed in-cart quantity control
2. Update `miniapp/src/ProductDetail.jsx` to use the same control model and avoid drifting from the main card.
3. Keep the card version compact and the detail version slightly larger, but use the same overall VkusVill-like `- amount +` pattern.
4. Drive both surfaces from shared per-product cart state so one product cannot show a plain add button on one surface and a quantity control on another after confirmation.
</action>

<acceptance_criteria>
- App.jsx no longer always renders the plain cart button once a product is confirmed in cart
- ProductDetail.jsx uses the same control-state model as the main card
- the same product switches in sync across card and detail drawer
- index.css contains dedicated styles for the in-card quantity control and the pending/checking state
</acceptance_criteria>
</task>

<task id="02" title="Support typed шт/кг entry and confirmed-only failure treatment">
<read_first>
- miniapp/src/App.jsx
- miniapp/src/ProductDetail.jsx
- miniapp/src/productMeta.js
- backend/main.py
- .planning/phases/44-frontend-bounded-add-ux/44-CONTEXT.md
</read_first>

<action>
Wire the new quantity controls to the requested quantity-entry rules:

1. For `шт` items, allow whole-number entry only and make `+` / `-` step by `1`.
2. For weighted items, allow decimal input (examples like `0.73 кг`) and accept both comma and dot while normalizing the displayed value.
3. Use the product/cart unit metadata already available in the app to decide whether the control is in `шт` mode or weighted mode.
4. Keep `quantity = 0` as remove-from-cart.
5. Pending state must never mark the item sold out or hard-failed.
6. Only confirmed sold-out/failure should collapse back to the normal add button with a retryable error state.
</action>

<acceptance_criteria>
- шт-mode input rejects non-integer final values
- weighted-mode input accepts decimal values with comma or dot input
- App.jsx or ProductDetail.jsx normalizes displayed quantity text for weighted items
- quantity `0` removes the item
- pending state does not apply sold-out or hard-failure styling
- confirmed failure returns the product to the normal add-button state
</acceptance_criteria>
</task>
