# Plan 01: Pending-Aware Add Flow

## Objective

Replace the legacy add timeout path with the new Phase 43 pending contract so the UI stops blocking on inline cart refreshes and keeps the rest of the MiniApp usable.

## must_haves

- [ ] The visible add interaction no longer waits on `refreshCartState(3, 1200)` after timeout
- [ ] `/api/cart/add` is called with the new pending-aware contract fields
- [ ] Pending uses a neutral "checking cart" state instead of a red error state
- [ ] Only the active product control is locked while the rest of the MiniApp remains usable

<action>
Update `miniapp/src/App.jsx` so the main add handler uses the new Phase 43 contract instead of the old inline timeout path:

1. Replace the current `AbortController` + `5500ms` timeout add path with a request that sends:
   - `allow_pending: true`
   - `client_request_id`
   - the existing `user_id`, `product_id`, `is_green`, and `price_type`
2. Handle three explicit backend outcomes:
   - immediate success → keep the short success flash, then hand off to the in-cart control state
   - pending (`202`, `pending = true`, `status = "pending"`) → switch into a neutral checking state and start background polling of `/api/cart/add-status/{attempt_id}`
   - clear failure → keep the existing unauth/sold-out/error branches, but do not use them for pending
3. Remove the current `refreshCartState(3, 1200)` call from the timeout/error path so the click path no longer waits on three extra cart checks.
4. Keep the rest of the MiniApp usable while a product is pending. Only that product’s control should be locked.
5. Update `miniapp/src/ProductDetail.jsx` to consume the same pending-aware state instead of showing only `loading/success/error`.
</action>

<acceptance_criteria>
- App.jsx sends `allow_pending` and `client_request_id` in `/api/cart/add`
- App.jsx no longer calls `refreshCartState(3, 1200)` from the add timeout path
- App.jsx polls `/api/cart/add-status/` after a pending add response
- ProductDetail.jsx has a distinct pending/checking state that is not the error state
- filters, scrolling, and other product cards remain usable while one product is pending
</acceptance_criteria>
</task>
