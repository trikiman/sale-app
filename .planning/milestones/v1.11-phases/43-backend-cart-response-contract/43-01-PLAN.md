# Plan 01: Session Metadata And Bounded Cart Client

## Objective

Make the cart client fast enough to support a pending-based contract by avoiding unnecessary warmup GETs and by removing inline timeout recovery from the same add request.

## must_haves

- [ ] Login cookie files can persist `sessid` and `user_id` metadata alongside the raw cookies list
- [ ] `VkusVillCart` uses saved metadata first and only falls back to `_extract_session_params()` when the metadata is missing
- [ ] `VkusVillCart.add()` enforces an approximately 1.5-second hot-path deadline for the add attempt
- [ ] `VkusVillCart.add()` no longer calls `get_cart()` inside its timeout exception path

## Tasks

<task id="01" title="Persist cart session metadata when login saves user cookies">
<read_first>
- backend/main.py
- cart/vkusvill_api.py
- .planning/phases/43-backend-cart-response-contract/43-CONTEXT.md
- backend/test_cart_items_fallback.py

<action>
Update the login cookie-save path in `backend/main.py` so user cookie files can carry session metadata that the cart client can reuse without a warmup GET:

1. In the successful verify/login save branch, stop writing the cookie file as a raw JSON list only.
2. Write a JSON object with these concrete top-level keys:
   - `cookies`
   - `sessid`
   - `user_id`
   - `saved_at`
3. Keep `cookies` equal to the existing `cookies_list`.
4. Populate `sessid` from the login/verification flow when available; populate `user_id` from the verified session context when available; if either value is unavailable, write `null` instead of omitting the key.
5. Set `saved_at` to an ISO-style timestamp string at write time.
6. Do this for both the phone-keyed and legacy user-keyed cookie-save branches so all future cart callers can benefit from the same file contract.
</action>

<acceptance_criteria>
- backend/main.py writes cookie JSON objects with `cookies`, `sessid`, `user_id`, and `saved_at`
- backend/main.py no longer writes only a bare cookie list in the success save path
- the cookie-save logic updates both phone-keyed and legacy user-keyed branches
- the keys `sessid` and `user_id` are present even when their values are null
</acceptance_criteria>
</task>

<task id="02" title="Refactor VkusVillCart to use metadata-first session bootstrap and a bounded add deadline">
<read_first>
- cart/vkusvill_api.py
- backend/main.py
- .planning/phases/43-backend-cart-response-contract/43-CONTEXT.md
- backend/test_cart_items_fallback.py
</read_first>

<action>
Refactor `cart/vkusvill_api.py` so cart add no longer chains extra network recovery work inside the same request:

1. Keep backward compatibility with existing cookie files by continuing to accept both:
   - the old raw cookie list format
   - the new object format with `cookies`, `sessid`, and `user_id`
2. In `_ensure_session()`, read `sessid` and `user_id` from the new object format first. Call `_extract_session_params()` only when one or both metadata values are still missing.
3. Add a concrete add hot-path deadline constant of approximately `1.5` seconds and use it inside `add(...)` to bound how long the method waits before returning an ambiguous result.
4. Remove the current timeout recovery that catches `httpx.TimeoutException` and immediately calls `get_cart()` inside `add(...)`.
5. Replace that timeout branch with a structured ambiguous result that clearly says the add outcome is still unknown, for example by setting:
   - `success = False`
   - `pending = True`
   - `error_type = "pending_timeout"`
   - `error = "pending_timeout"`
6. Keep clear HTTP and malformed-response failures distinguishable from this new pending result.
</action>

<acceptance_criteria>
- cart/vkusvill_api.py accepts both raw cookie-list files and object-format cookie files
- `_ensure_session()` reads `sessid` and `user_id` from file metadata before attempting `_extract_session_params()`
- cart/vkusvill_api.py defines a concrete add deadline of about 1.5 seconds
- `add(` no longer calls `get_cart()` from its timeout exception path
- timeout ambiguity returns a structured pending result with `pending = True` and `error_type = "pending_timeout"`
</acceptance_criteria>
</task>
