# Plan 03: Pending Contract Backend Coverage

## Objective

Protect the new Phase 43 backend contract so later frontend work can depend on it without re-debugging timeout ambiguity and duplicate-add races.

## must_haves

- [ ] Legacy callers still get the existing timeout behavior until they opt in
- [ ] Pending-aware callers receive HTTP 202 plus a stable `attempt_id`
- [ ] Duplicate unresolved adds reuse the same attempt instead of issuing a second upstream add
- [ ] Status lookup can resolve pending attempts into final success or failure

<action>
Create `backend/test_cart_pending_contract.py` to lock the new backend behavior:

1. Add a test where `allow_pending = true` and the cart client returns the new pending result; assert `/api/cart/add` responds with:
   - HTTP `202`
   - `pending = true`
   - `status = "pending"`
   - a non-empty `attempt_id`
2. Add a test that sends two pending-aware add requests for the same `user_id` and `product_id` within the dedupe window and asserts both responses return the same `attempt_id`.
3. Add a test for `GET /api/cart/add-status/{attempt_id}` that simulates a later cart success and asserts the attempt transitions from `pending` to `success`.
4. Add a second status-route test that simulates a failed reconciliation and asserts the attempt transitions to `failed` or `expired` with `last_error` populated.
</action>
