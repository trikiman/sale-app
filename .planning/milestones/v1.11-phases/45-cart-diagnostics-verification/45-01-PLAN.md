# Plan 01: Cart Attempt Diagnostics Surface

## Objective

Make the new pending-aware cart flow inspectable without opening raw code by surfacing recent cart attempt lifecycle data through `/admin/status`, the admin dashboard, and backend logs.

## must_haves

- [ ] Recent cart attempt data is exposed through `/admin/status`
- [ ] Attempt diagnostics include start, pending/final state, timeout class or source, and total duration until final truth
- [ ] Admin dashboard renders recent cart attempt diagnostics without introducing a new operator surface
- [ ] Backend route tests cover the new diagnostics payload

## Tasks

<task id="01" title="Add recent cart attempt diagnostics to the backend status contract">
<read_first>
- backend/main.py
- .planning/phases/45-cart-diagnostics-verification/45-CONTEXT.md
- .planning/phases/43-backend-cart-response-contract/43-CONTEXT.md
- backend/test_admin_routes.py
</read_first>

<action>
Extend the cart attempt registry and `/admin/status` payload in `backend/main.py` so operators can inspect the new cart flow:

1. Add timing fields to cart attempts, including:
   - `started_at`
   - `resolved_at`
   - `duration_ms`
2. Record enough lifecycle information to distinguish:
   - immediate success
   - pending then success
   - pending then failure
   - clear failure before pending
3. Store the reconciliation source / timeout class in a stable field, for example:
   - `source`
   - `final_status`
   - `last_error`
4. Add a helper that returns a recent list of cart attempts from the in-memory registry in newest-first order.
5. Extend `/admin/status` to include a `cartDiagnostics` object with at least:
   - `recentAttempts`
   - `pendingCount`
   - `lastResolvedAt`
6. Update backend logging so cart attempt creation, pending transition, reconciliation success, and reconciliation failure emit explicit log lines with attempt ID and product/user IDs.
</action>

<acceptance_criteria>
- backend/main.py stores `started_at`, `resolved_at`, and `duration_ms` for cart attempts
- `/admin/status` returns `cartDiagnostics`
- `cartDiagnostics` includes `recentAttempts`, `pendingCount`, and `lastResolvedAt`
- recent attempt entries include enough fields to distinguish immediate success, pending→success, pending→failure, and clear failure
- backend logs include explicit cart attempt lifecycle messages with attempt IDs
</acceptance_criteria>
</task>

<task id="02" title="Render recent cart attempt diagnostics in the existing admin dashboard">
<read_first>

<action>
Update `backend/admin.html` so the current admin dashboard can display recent cart attempt diagnostics from `/admin/status`:

1. Add a compact cart diagnostics section to the existing dashboard instead of creating a new page.
2. Show at least:
   - number of currently pending cart attempts
   - last resolved cart attempt time
   - a short recent-attempt list with product ID, final status, source, and duration
3. Reuse the existing auto-refresh path in `applyStatus(...)` so cart diagnostics update with the same dashboard polling cycle.
4. Keep the UI mechanical and operational; no new end-user styling patterns are needed.
</action>

<acceptance_criteria>
- backend/admin.html contains a cart diagnostics section
- `applyStatus(...)` reads `cartDiagnostics`
- the dashboard displays pending count, last resolved time, and a recent attempt list
- cart diagnostics refresh with the existing `/admin/status` polling cycle
</acceptance_criteria>
</task>
