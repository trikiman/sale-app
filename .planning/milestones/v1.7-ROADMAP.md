# Milestone v1.7: Categories & Subgroups

**Status:** ✅ SHIPPED 2026-04-03
**Phases:** 29-33
**Total Plans:** 1 formal phase plan, plus ad-hoc implementation/verification for the remaining phases

## Overview

This milestone added VkusVill group/subgroup hierarchy across the full user journey: scraper data, product catalog, main-page filters, favorites, history filters, and Telegram notifications. It also closed a live regression on the History page where subgroup chips could appear for catalog data that had no matching history results.

## Phases

### Phase 29: Subgroup Data Layer

**Goal:** Scrape VkusVill subgroups and add group/subgroup columns to the data pipeline.
**Depends on:** —
**Plans:** 1 plan

**Delivered:**
- Subgroup discovery in `scrape_categories.py`
- `category_db.json` hierarchy with 16,426 products, 524 subgroups, 46 groups
- `product_catalog.group_name` / `product_catalog.subgroup` columns and indices
- Merge-pipeline support for `group` / `subgroup`
- Backend group/subgroup API/filter support

### Phase 30: Main Page Group/Subgroup UI

**Goal:** Add subgroup drill-down to the main page category filter.
**Depends on:** Phase 29
**Plans:** ad-hoc implementation

**Delivered:**
- Main-page group chips driven by hierarchy data
- Second-row subgroup drill-down
- Correct subgroup-row hide behavior
- Group/subgroup-aware product filtering on the main page

### Phase 31: Group/Subgroup Favorites

**Goal:** Enable favoriting groups/subgroups as single entries.
**Depends on:** Phases 29-30
**Plans:** ad-hoc implementation

**Delivered:**
- `/api/favorites/{user_id}/categories` GET/POST/DELETE endpoints
- Exact favorite key contract: `group:X` and `subgroup:X/Y`
- Heart toggles on main-page chips with optimistic updates

### Phase 32: History Page Groups & Favorites

**Goal:** Add the same group/subgroup filtering and favorites to the history page.
**Depends on:** Phases 29-31
**Plans:** ad-hoc implementation

**Delivered:**
- History-page group/subgroup chips
- Server-side `group` / `subgroup` filtering
- Post-ship fix: history chips now use history-backed scope instead of full-catalog scope, removing empty subgroup rows like `Для собак (24)` with zero results

### Phase 33: Group/Subgroup Notifications

**Goal:** Telegram notifier alerts users when products in their favorited groups/subgroups go on sale.
**Depends on:** Phases 29-32
**Plans:** ad-hoc implementation

**Delivered:**
- Notifier now checks product favorites together with group/subgroup favorites
- `product_catalog` fallback enriches products when `proposals.json` lacks hierarchy fields
- Per-product dedupe across product/group/subgroup matches
- Telegram message includes the most specific match reason
- Regression test for category-alert behavior

## Milestone Summary

**Key Decisions:**
- Use exact serialized favorite keys (`group:X`, `subgroup:X/Y`) across backend, frontend, and notifier
- Use history-aware group scope on the History page so chip rows match the actual history dataset
- Let the notifier fall back to `product_catalog` because the current sale snapshot in `proposals.json` still does not reliably include `group` / `subgroup`

**Issues Resolved:**
- `Product` response model was stripping `group` / `subgroup` from API responses
- History page could show empty subgroup rows due to chip scope mismatch
- EC2 backend deploy got stuck on a stale manual `uvicorn` process holding port `8000`; service was restored under systemd

**Issues Deferred:**
- Search completeness for non-sale products on the History page
- Deeper subgroup pagination beyond page 1

**Technical Debt Incurred:**
- `product_catalog.subgroup` still stores only one subgroup per product
- Production category notifications currently depend on `product_catalog` fallback until merged sale JSON always carries hierarchy metadata
- Most v1.7 phases were executed ad-hoc outside the full GSD plan/summary pipeline
