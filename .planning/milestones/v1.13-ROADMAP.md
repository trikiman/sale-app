## Milestone Goal

Make add-to-cart feel instant and fix the failures that made cart adds silently drop on the user-facing path. Scope was explicitly reworked on 2026-04-08 (commit `8f35acb`, "docs: rework v1.13 — drop optimistic UI, focus on real API under 5s") from an earlier optimistic-UI framing toward concrete classified failure handling and real-API timing targets.

### Phase 47: Diagnose & Fix Cart Failures

**Goal:** Cart adds succeed reliably; every failure produces a structured diagnostic payload and a typed `error_type`.
**Requirements:** CART-15, CART-16
**Depends on:** —
**Success Criteria:**

1. User can tap add-to-cart and the product actually appears in their VkusVill cart (or the endpoint returns a classified error).
2. Every failure mode is logged with enough context to attribute the root cause.
3. The endpoint returns a typed `error_type` field instead of a generic 500.

**Verification:** `47-VERIFICATION.md` (3/5 code-level PASS, 2 pending marked `resolved_by_downstream` via v1.14 phase 55)
**Summary commits:** `36d1c3c` (Plan 01), `996b2a6` (Plan 02), `6dcffb2` (tests)

### Phase 48: Session Warmup Optimization

**Goal:** First cart add is fast because session metadata is already cached; stale sessid auto-refreshes before it causes a failure.
**Requirements:** PERF-01, PERF-02
**Depends on:** Phase 47
**Success Criteria:**

1. `sessid` and `user_id` are pre-extracted at login with a `sessid_ts` timestamp.
2. Cart add completes with real VkusVill API confirmation within the budget.
3. Stale sessid (>30 min) is auto-refreshed before it causes a failure.

**Verification:** `48-VERIFICATION.md` (5/7 code-level PASS, 2 pending marked `resolved_by_downstream` via v1.14 phase 55)

### Phase 49: Error Recovery & Polish

**Goal:** Users see distinct, actionable error messages and can recover without refreshing.
**Requirements:** ERR-01, ERR-02
**Depends on:** Phase 48
**Success Criteria:**

1. Distinct messages per failure mode (sold-out, session-expired, VkusVill-down, network).
2. Session-expired errors surface a re-login prompt, not a generic cart error.
3. Transient errors leave the button in a retry state.

**Verification:** `49-VERIFICATION.md` (`passed`), `49-HUMAN-UAT.md` (`resolved`)

## Key Decisions

| Decision | Rationale | Outcome |
|----------|-----------|---------|
| Scope rework from optimistic UI to classified failures | The earlier framing hid the real problem (cart drops); users wanted real successful cart adds with honest errors when they failed | ✓ Good |
| `error_type` field on cart responses instead of HTTP status reuse | HTTP status cannot express both "502 transient" and "410 product gone" without overloading codes | ✓ Good |
| `sessid_ts` persistence + 30-min staleness threshold | Matches the empirical VkusVill session drift window observed during 48-RESEARCH | ✓ Good |
| Treat v1.14 phase 55 live evidence as the UAT for v1.13 47/48 | v1.13 code paths are still live and unchanged; v1.14 added repair + verification on top, so its live proof transitively covers v1.13 behaviors | ✓ Good (documented in this audit) |
| Skip Nyquist VALIDATION.md for every phase | Same pattern as v1.10/v1.11; verification + downstream live proof is sufficient for this family of milestones | ⚠️ Accept as advisory debt |
