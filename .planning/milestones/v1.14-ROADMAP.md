## Milestone Goal

Make MiniApp add-to-cart actually work for real users (not just in code-path theory), remove fake restock/reentry semantics from sale history, repair already-persisted fake sessions, and verify both against fresh production-like evidence before treating the milestone as done.

## Granularity

Fine \u2014 four small, strictly dependent phases, driven by real user-reported outcomes from the v1.13 post-ship period.

### Phase 52: Real Cart Failure Reproduction & Diagnostics

**Goal:** Reproduce the real user-facing add-to-cart failure and capture enough evidence (backend logs, direct upstream calls, admin/status output) to stop guessing about root cause.
**Requirements:** OPS-05
**Depends on:** \u2014
**Success Criteria:**

1. The broken add-to-cart flow is reproduced end-to-end from MiniApp interaction through backend/cart logs.
2. Admin/status or logs explain exactly why the failing cart attempt was classified the way it was.
3. The next fix scope (Phase 53) is anchored in observed failure evidence, not inferred theory.

**Verification:** `52-VERIFICATION.md` (3/3 PASS \u2014 reproduced `source_unavailable` on `/api/cart/items`, direct `basket_recalc.php` + `basket_add.php` calls succeeded on EC2, production `sale_sessions` queries showed thousands of sub-60-minute splits that 54 then repaired)

### Phase 53: Cart Truth Path Fixes

**Goal:** Make MiniApp add-to-cart succeed reliably and keep frontend cart state aligned with confirmed cart truth.
**Requirements:** CART-19, CART-20, CART-21
**Depends on:** Phase 52
**Success Criteria:**

1. User can add a product from the MiniApp and the product actually appears in the real VkusVill cart.
2. Success and quantity-stepper UI only appear when backed by confirmed add/cart truth.
3. Ambiguous or failed add flows end in a truthful stable state with a clear retry/recovery path.

**Verification:** `53-VERIFICATION.md` (3/3 PASS \u2014 live `/api/cart/add` 200 on an authenticated guest flow, live `/api/cart/items` returned actual basket lines, stale-cookie simulation completed in ~2.7 s)

### Phase 54: History Reentry Semantics Correction

**Goal:** Make history, notifier, and "new again" behavior reflect only true sale exits and true sale returns, and repair the already-corrupted current data.
**Requirements:** HIST-09, HIST-10, HIST-11
**Depends on:** Phase 53
**Success Criteria:**

1. Stale or missing scrape cycles do not create fake restocks or fake reentries.
2. Session reopening only happens when the product truly left sale and then returned.
3. User-visible history/notifier semantics distinguish continued sale from genuine return-to-sale events.
4. Existing persisted history/session records are repaired so current data no longer shows fake restocks/reentries.

**Verification:** `54-VERIFICATION.md` (3/3 PASS \u2014 `tests/test_sale_history_repair.py` PASS, production repair removed thousands of fake splits, post-repair `short_gaps_remaining = 0`)

### Phase 55: Live Verification & Release Decision

**Goal:** Prove cart truth and corrected history semantics with fresh evidence before deciding whether the milestone can close.
**Requirements:** QA-05
**Depends on:** Phase 54
**Success Criteria:**

1. Verification includes a live or production-like proof that add-to-cart works for a real user path.
2. Verification includes evidence that fake restock/reentry behavior is no longer present in history semantics.
3. Milestone closure decision is based on fresh evidence, not only code review or inferred behavior.

**Verification:** `55-VERIFICATION.md` (3/3 PASS) + `55-01-SUMMARY.md` (live `POST /api/cart/add 200` for product 33215, `/api/cart/items` real basket lines, `/api/cart/remove 200` back to single-banana state, stale `sessid_ts=1` add ~2715 ms, product 100069 sessions 56 \u2192 5, short_gaps_remaining 0)

## Key Decisions

| Decision | Rationale | Outcome |
|----------|-----------|---------|
| Reproduce live before fixing | v1.13 shipped "instant cart" paper-wins that didn't hold up for users; repro-first was the corrective for that anti-pattern | \u2713 Good |
| Repair existing persisted history, not only prevent forward corruption | Users were still seeing old fake restocks even after the continuity logic was fixed; forward-only fix would have left visible lies in the UI for weeks | \u2713 Good |
| Treat phase 55 live evidence as the single "milestone passed" gate | QA-05 was written to force a fresh evidence check rather than repeating the v1.13 "verified in code" mistake | \u2713 Good |
| No new features in this milestone | Scope was deliberately narrow \u2014 truth + repair, not UX polish | \u2713 Good |
