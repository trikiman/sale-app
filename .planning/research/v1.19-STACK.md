## Proposed additions

### Circuit breaker library: **roll our own, do NOT add `pybreaker`**

- Industry library: [`pybreaker`](https://pypi.org/project/pybreaker/) 1.4+ gives thread-safe circuit with Closed/Open/Half-Open and a sliding window.
- **Why not adopt it here:** the scheduler's failure model is cycle-shaped (every 5 min one full cycle happens or doesn't), not request-shaped (many RPC calls per second). `pybreaker` is designed for the request model — wrapping each scraper invocation in a decorator wouldn't model "3 consecutive failed cycles" any better than the current handwritten counter.
- **What to build instead:** extend `scheduler_service.py`'s existing `proxy_state["consecutive_fails"]` into a 3-state machine:
  - `closed` — normal cadence
  - `open` — 120 s cooldown + proxy refresh (current behavior)
  - `half_open` — one green-only "probe cycle" after cooldown before returning to full cycles. If probe succeeds → closed. If probe fails → back to open with exponential backoff (`wait = min(120 * 2^n, 30min)` ).
- Live evidence this matters: production journal (2026-05-03 20:56) shows `Circuit breaker: 162 consecutive failures. Waiting 2 min before next full cycle.` — with only two states (open/closed) and fixed 120 s cooldown, we lost ~5.4 h before the system self-healed. A half-open probe state would have caught recovery as soon as upstream was workable again.

### Retry library: **keep ad-hoc `try/except + sleep`, do NOT add `tenacity`**

- `tenacity` 9.x would give us decorators like `@retry(stop=stop_after_attempt(3), wait=wait_exponential())`.
- **Why not:** we already have retry semantics in `scheduler_service.py::_run_scraper_set` (scraper run → kill Chrome → rotate proxy → retry once). Adding `tenacity` would touch every retry site and the win is aesthetic, not functional.
- Re-evaluate only if v1.19 requires more than 2 retry sites.

### Metrics / Prometheus: **defer to future milestone**

- `prometheus_client` 0.21+ would give us `/metrics` for an external scraper.
- **Why defer:** the operator is a family (up to 5 users) + external uptime ping. Full Prometheus + Grafana is over-engineered. A simple `/api/health/deep` endpoint with a JSON body is enough for:
  - An external uptime ping (Uptime Robot, Cronitor, or a simple cron + curl)
  - Telegram bot alerts (already in-stack)
- If v1.19 ends and we still can't explain production behavior, promote Prometheus to v1.20.

### External uptime ping: **integrate one, don't build one**

- Candidates: Uptime Robot (free tier: 50 monitors, 5 min interval), Cronitor (heartbeat-style), BetterStack. A plain cron + Telegram bot webhook also works.
- **Recommended:** pick one external service that pings `/api/health/deep` every 5 min and posts to the existing Telegram bot on state change. Keeps the alert loop inside the stack the user already owns.
- Zero new dependencies in the repo — the ping is external; the bot is already built.

### Xray config changes (no new library, just config)

- **Change `observatory.probeURL`** from `https://www.google.com/generate_204` to `https://vkusvill.ru/` (or a lightweight stable VkusVill endpoint). This is the most important stack-level change in v1.19. See `PITFALLS.md` §3 and `ARCHITECTURE.md` §2 for the why.
- **Add `observatory.probeInterval` tuning:** 5 min is fine for steady state, but consider 2 min during the first hour after a refresh so newly-admitted nodes get evaluated faster.

## What is intentionally NOT in scope for v1.19

- **No Kubernetes / container migration.** EC2 + systemd is fine for this workload.
- **No replacing xray-core with WireGuard or a different bridge.** v1.15 chose xray for real reasons; reliability, not substitution.
- **No adding a paid APM (Datadog, New Relic).** Family-scale project.
- **No frontend framework change.** React + Vite stays.
- **No new database.** SQLite is enough; any pool/health/circuit state fits in existing files (`vless_pool.json`, `proxy_events.jsonl`, a new `scheduler_state.json`).

## Integration constraints to honor

1. **xray restart cost.** Any rotation that calls `_remove_host_and_restart` rebuilds the xray config and re-execs the process. PR #25 loop × 5 restarts × ~500 ms each = ~2.5 s of pure xray thrash per bad probe window. Any new rotation logic must cap restart frequency. See `PITFALLS.md` §2.
2. **`next_proxy()` and `mark_current_node_blocked()` BOTH restart xray.** PR #26's revert description said otherwise — that was wrong; both paths lead to `_remove_host_and_restart`. Don't rely on that (false) distinction when designing a fix. Any non-destructive rotation has to go *through xray's own balancer*, not through Python-side config rewrites.
3. **SOCKS5 bridge is shared** by scraper, cart API, login flow, and detail-proxy. A reliability change that slows down admission probes slows down all four. Budget accordingly.
4. **Vercel-side is cached by design.** `vkusvillsale.vercel.app/api/products` can return HTTP 200 with stale data even when EC2 is fully broken. The deep health endpoint must not rely on frontend 200s as a signal.
