## 2. The big architectural decision: probe/target alignment

**Current (broken) state.** `bin/xray/configs/active.json::observatory.probeURL` is `https://www.google.com/generate_204`. The `leastPing` balancer uses observatory latency to choose outbounds. If a VLESS exit node can reach Google but is blocked by VkusVill (e.g. because VkusVill 403'd the exit IP), observatory still reports "fast" and `leastPing` keeps routing real VkusVill traffic through it.

**Proposed state.** Change `observatory.probeURL` to a VkusVill endpoint. Two candidates:

| Candidate | Pros | Cons |
|---|---|---|
| `https://vkusvill.ru/` | Same domain as our traffic, same RU-IP check, no extra endpoints to maintain | ~50-100 KB response, runs every 5 min × 13 outbounds = ~130 MB/day of wasted bandwidth per EC2 box |
| `https://vkusvill.ru/favicon.ico` | Tiny (~5 KB), stable, same edge as main site | Still same domain/policy — equally indicative |
| A lightweight VkusVill page we know is fast | Minimal bytes | Fragile if VkusVill restructures |

**Recommendation:** `https://vkusvill.ru/favicon.ico`. Small, stable, same origin policy as the real traffic.

**Rollout risk.** This is a single config-file change. The impact on pool behavior is *large* — nodes that were artificially "fast" in leastPing will drop in ranking, and nodes that were previously "slow for google but fast for VkusVill" may rise. Needs live validation (Cat F, §F1-F2) before merging.

## 4. Data flow for new endpoints

### `/api/health` (light)

```
  client → GET /api/health
         → backend/main.py → backend/health.py::compute_health(light=True)
         → returns 200 {"status": "ok", "uptime_s": ...}
         → rate-limit: 10/s per IP
```

Effectively just proves uvicorn is alive. Meant for systemd watchdog / VC monitors.

### `/api/health/deep` (full)

```
  client → GET /api/health/deep
         → backend/main.py → backend/health.py::compute_health(light=False)
                         ↓
                      reads:
                        - scheduler_state.json  (breaker phase, next_attempt_at)
                        - vless/manager::pool_snapshot()  (pool_size, quarantined)
                        - data/proposals.json mtime  (last merge)
                        - data/cycle_state.json  (last cycle status)
                        - xray_status()  (process running + pid)
                         ↓
                      computes:
                        status: healthy if all criteria pass
                              : degraded if 1-2 criteria fail (e.g. pool=6/7)
                              : unhealthy if 3+ criteria fail or critical one
                              (xray process dead, scheduler heartbeat stale)
                         ↓
         → returns 200 + JSON if healthy/degraded
         → returns 503 + JSON if unhealthy
         → rate-limit: 1/s per IP
```

Response shape:

```json
{
  "status": "degraded",
  "reasons": [
    "pool_size 6 below MIN_HEALTHY 7",
    "circuit_breaker half_open (next_attempt in 42s)"
  ],
  "data": {
    "pool_size": 6,
    "min_healthy": 7,
    "quarantined_count": 3,
    "circuit_state": "half_open",
    "circuit_backoff_level": 2,
    "last_successful_cycle_s_ago": 127,
    "xray_running": true,
    "products_count": 196,
    "products_age_s": 420
  },
  "uptime_s": 374400
}
```

**Privacy check:** no node IPs, no user identifiers, no cookie state, no cart data, no proxy-event log lines. Safe for public GET.

## 5. State machine: the new circuit breaker

```
                ┌──────────┐
           ┌───▶│  CLOSED  │────── any scrape succeeds ───────┐
           │    └─────┬────┘                                  │
           │          │                                       │
           │          │ 3 consecutive fully-failed cycles     │
           │          ▼                                       │
           │    ┌──────────┐                                  │
           │    │   OPEN   │                                  │
           │    └─────┬────┘                                  │
           │          │                                       │
           │          │ cooldown expired                      │
           │          │ cooldown = min(120 * 2^level, 30min)  │
           │          ▼                                       │
           │    ┌────────────┐                                │
           │    │ HALF_OPEN  │                                │
           │    └─────┬──────┘                                │
           │          │                                       │
           │          │ run 1 green-only probe cycle (~30s)   │
           │          │                                       │
           │   probe  │  probe                                │
           │   OK     │  FAILS                                │
           │          │                                       │
           └──────────┘  ↓                                    │
                  backoff_level++, back to OPEN               │
                                                              │
                  after M successful full cycles ─────────────┘
                  → reset backoff_level to 0
```

**Persisted state (`data/scheduler_state.json`):**

```json
{
  "phase": "closed" | "open" | "half_open",
  "backoff_level": 0,
  "opened_at": null | "2026-05-03T20:56:00",
  "next_attempt_at": null | "2026-05-03T20:58:00",
  "consecutive_failed_cycles": 0,
  "consecutive_succeeded_cycles": 5,
  "last_full_cycle_outcome": "success" | "all_failed" | "partial_failed",
  "last_full_cycle_at": "2026-05-03T20:55:00"
}
```

## 7. Explicit risks

1. **Observatory probeURL change can worsen things first.** Nodes that were artificially fast-in-ranking for Google will drop; real VkusVill latency can be higher for some nodes. Live evidence (phase 60) must show pool stays ≥ `MIN_HEALTHY` for 30 min after the change.
2. **Half-open probe cycles add ~30 s per breaker trip.** Acceptable — beats the current "162 useless full cycles" pattern.
3. **State machine persistence adds a read on scheduler start.** If `scheduler_state.json` is corrupt, fall back to `closed` + `backoff_level=0`. No silent failures.
4. **Unauth health endpoint is a tiny DoS surface.** Rate-limited to 1 req/s per IP via existing FastAPI middleware (if any; if not, add a simple token bucket in `backend/health.py`). No DB queries in the health path — all reads are O(1) file reads or in-memory accessors.
5. **`/api/health/deep` during scheduler startup races.** `scheduler_state.json` may not exist yet. Treat missing file as `{"phase": "closed", "backoff_level": 0}` and proceed.

## 8. What success looks like end-to-end

After v1.19 ships:

- An external uptime ping hits `/api/health/deep` every 5 min and gets either 200 or 503 truthfully.
- A single VkusVill exit IP getting blocked triggers: observatory latency spike on that node → leastPing de-prefers it → if repeated failures, Python-side quarantine → pool stays ≥ 7 because refresh replenishes on `MIN_HEALTHY - 1`.
- If the whole pool degrades, circuit breaker enters `open` → exponential backoff (2 min → 4 → 8 → 16 → 30 cap) → periodic green-only probe cycles in `half_open` → first success flips to `closed` automatically, no human intervention.
- Telegram alert fires **once** on closed→open, **once** on open→half_open->closed recovery, not on every failed cycle. 24 h of chaos = O(3-5) messages, not 162.
- Operator inspecting `/admin/status` or `/admin` sees the reliability block: circuit state, pool trend, last probe result, observatory probe target.
