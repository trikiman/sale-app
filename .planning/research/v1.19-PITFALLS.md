# PITFALLS — v1.19 Production Reliability & 24/7 Uptime

Common mistakes when adding reliability features to an existing scraping + proxy bridge stack. Most of these are grounded in *this project's* PR #25/#26 evidence, not abstract best-practice listicles.

## 1. Probe timeout under empirical latency

**The mistake.** Set a probe timeout based on what "feels fast" rather than measured behavior. PR #25 picked 5 s. Healthy nodes through this bridge respond in 7-9 s (`curl -x socks5h://127.0.0.1:10808 --max-time 10 https://vkusvill.ru/` = 7.4, 9.2, 7.8 s in the post-revert verification). Every probe false-negatived.

**Why it happens.** Developers default to "HTTP probes should be snappy, so 3-5 s is plenty". In an overlay network (xray VLESS+Reality through 2 ISPs and a TLS handshake) the ceiling is structurally higher. Google's Reality handshake alone can eat 1-3 s on a cold connection.

**How to prevent.**
- Before setting any probe timeout, measure healthy-node latency on the actual bridge N ≥ 5 times and take the 95th percentile + 50% safety margin.
- Codify the measurement in a test that asserts `timeout > measured_p95 * 1.5`. If someone lowers the timeout without re-measuring, the test fails.
- In `vless/preflight.py`, comment the 12 s value with a link to the measurement evidence.

## 2. Restart cascade from aggressive rotation

**The mistake.** Calling `next_proxy()` in a loop on probe failure. Each call removes a node and triggers `_remove_host_and_restart`, which rewrites `active.json` and re-execs the xray binary. PR #25 looped × 5 before giving up = 5 xray restarts in ~25 s.

**Why PR #26's revert reasoning was partially wrong.** PR #26 said "use `mark_current_node_blocked` because it doesn't restart xray". Actually BOTH paths restart xray — `mark_current_node_blocked → mark_vkusvill_blocked → _remove_host_and_restart` (`vless/manager.py:230`). The real fix is not "use the non-restarting path" (there isn't one), it's "cap the number of rotations".

**How to prevent.**
- Cap rotations at 2 per scraper launch. If after 2 the bridge still doesn't respond, trip the circuit breaker instead of trying a 3rd/4th/5th.
- Any PR that adds rotation logic must include a test that asserts `xray restart count <= N per probe window`.
- Prefer letting xray's own `leastPing` balancer route away from slow nodes (no restart needed) before resorting to Python-side removal.

## 3. Probe/target mismatch (the silent killer)

**The mistake.** xray's observatory probes `https://www.google.com/generate_204` but our traffic goes to `https://vkusvill.ru/`. A node that can reach Google but is blocked by VkusVill (403 / 451 / connection reset from VkusVill's edge) stays "fast" in the `leastPing` balancer. Real requests fail silently; observatory stays green.

**Why it happens.** Observatory's default in every xray config snippet on the internet is either `generate_204` or a Cloudflare endpoint. Nobody thinks about target alignment until it bites — and it bites quietly because the symptom is "scrapers fail but xray looks fine".

**Live evidence this is biting us right now.** 2026-05-03 production journal: pool size 13, xray service active 4 days, 162 consecutive scrape failures. xray itself is fine; individual exit IPs are just blocked by VkusVill while Google still loves them.

**How to prevent.**
- Change `observatory.probeURL` to `https://vkusvill.ru/favicon.ico` (see ARCHITECTURE §2).
- Add a test that reads the xray config file and asserts probeURL hostname matches a known-good target domain.
- Document in `vless/config_gen.py` why this matters so future contributors don't "optimize" it back to Google.

## 4. Consecutive-count breaker misses partial failure

**The mistake.** Scheduler's current breaker: `consecutive_fails += 1 if all scrapers failed else reset to 0`. Consequence: if 2 of 3 scrapers fail every cycle for an hour, the counter stays at 0 forever and the breaker never trips. Data is badly stale; breaker is happy.

**Why it happens.** "All-failed" is an easy to implement unambiguous signal. Partial failure is noisy. Developers default to the cleaner signal and miss the degraded-but-not-broken state.

**How to prevent.**
- Use a sliding-window failure ratio instead of a consecutive-all-failed count. Industry default: 50% failure rate over the last 10 cycles (see `STACK.md` circuit breaker section).
- Or: track per-scraper consecutive failures separately and trip if *any one scraper* exceeds a threshold.
- Test: simulate "2 of 3 fail every cycle for 20 cycles". Current code: breaker doesn't trip. After fix: breaker trips or alerts.

## 5. "Active but broken" — systemd status lies

**The mistake.** Relying on `systemctl is-active saleapp-backend` as a health signal. In production right now all three services (`backend`, `scheduler`, `xray`) report `active` but every detail-proxy request returns "all proxies timed out" and the scraper's exit 1 stream hasn't paused in hours.

**Why it happens.** systemd only knows "is the PID alive". A Python process with a dead socket pool is alive. A scheduler in a 2-min `time.sleep(120)` loop is alive. An uvicorn with a deadlocked thread pool is alive.

**How to prevent.**
- `/api/health/deep` (§E1 in FEATURES) is the *real* health signal. systemd's `active` is necessary but not sufficient.
- Add a `systemd watchdog` directive pointing to `/api/health` (the light version) so systemd can restart the backend if the endpoint stops responding, but *not* trust active-status alone.
- Never use `systemctl is-active` as a pre-deploy check for "is this service healthy". Use `curl /api/health/deep`.

## 6. Deploy without live verification (the PR #25 story)

**The mistake.** Write reasonable code, add unit tests, merge. Skip the EC2 smoke test. Discover in 8 minutes that the code actually breaks production.

**Why it happens.** Unit tests don't replicate: real xray process lifecycle, real VLESS+Reality handshake latency, real VkusVill edge behavior. PR #25's unit tests passed. The 5 s timeout choice was never tested against a healthy-node-through-the-real-bridge measurement because no such test existed.

**How to prevent.**
- Mandatory per-phase smoke test script (§F2). Runs on EC2 over SSH, returns pass/fail per criterion. Scripted = reproducible = auditable.
- Verification evidence required before merging: smoke script output + Vercel miniapp `/api/cart/add` HTTP 200 capture + journal excerpt showing the target behavior actually happens.
- If the verification script doesn't exist yet for a given criterion, writing the script is part of the phase scope.

## 7. Alerting fatigue vs alerting silence (both failure modes)

**The mistake A (fatigue).** Alert on every failed cycle. 162 failures = 162 Telegram messages = operator mutes the bot = next real alert gets ignored.

**The mistake B (silence).** Alert only on "everything is on fire". Miss slow drifts (pool 25 → 13 over 8 days; no single alertable event).

**How to prevent.**
- Alert on *state transitions* of the circuit breaker, not per-failed-cycle. closed→open and half_open→closed each fire once.
- Add a daily digest: "Today's pool size min 13, max 27, median 22, quarantined 8." Cheap, signal without noise.
- Test the alert path itself: phase verification includes "trigger fake breaker trip, confirm exactly one Telegram message arrived".

## 8. Pool replenish cadence < quarantine drain cadence

**The mistake.** Set refresh cadence at 24 h (`CACHE_TTL = 86400`), quarantine duration at 4 h (`VKUSVILL_COOLDOWN_S = 4*3600`). During a bad day when 10 nodes per day get VkusVill-blocked, drain rate is ~10/day while replenish rate is bounded by whatever 24 h refresh found last. Pool monotonically shrinks.

**Live evidence.** Admitted pool: 25 at v1.18 ship (2026-04-25), 13 by 2026-05-03 — lost ~1.5 nodes/day.

**How to prevent.**
- `ensure_pool()` already triggers refresh when `pool_count() < MIN_HEALTHY`. Verify it's firing — if not, fix.
- Refresh to `MAX_CACHED` rather than stopping at `MIN_HEALTHY + 1`. Adds headroom.
- Shorter cooldowns for non-block failures (`TLS handshake fail` = 15 min, not 4 h). 4 h is for confirmed VkusVill blocks only.
- Log pool trend in `proxy_events.jsonl` on every refresh so drift is visible in operator review.

## 9. Auto-restart masks root cause

**The mistake.** When things break, auto-restart xray / scheduler / backend. Brings service "back" but loses the state needed to diagnose the original failure.

**How to prevent.**
- Auto-recovery belongs *inside* the application's state machine (circuit breaker half-open probe), not in systemd `Restart=always` + external watchdog that kills processes.
- If xray genuinely dies, systemd can restart it. But the scheduler's breaker trip should *not* restart xray — it should work with what's there.
- Keep the pre-fix-attempt state recoverable: write a dump to `data/debug_snapshots/` on any state transition, so post-mortem investigation has the inputs.

## 10. Unauthed health endpoint leaks too much

**The mistake.** Write `/api/health/deep`, dump every internal field into it. Now an attacker knows: how many VLESS nodes you have, which IPs, when your last refresh was, how healthy your scheduler is. Useful recon.

**How to prevent.**
- Explicit allow-list of fields in the unauth response. No IPs, no identifiers, no counts that could reveal user base size.
- Admin version (`/admin/status`) gets the full data behind the token.
- Review the unauth response shape in code review with a "what does this tell an attacker?" lens.

## 11. Assuming xray's `leastPing` auto-rotates

**The mistake.** "xray has leastPing balancer; it'll automatically route away from bad nodes." Truth: `leastPing` routes based on the observatory probe results. If the probe target is Google and all nodes can reach Google, `leastPing` rotates among "equally good to Google" nodes regardless of whether any of them can reach VkusVill.

**Related xray issues referenced:**
- [XTLS/Xray-core#872](https://github.com/XTLS/Xray-core/issues/872) — leastPing routing to direct on DNS lookup mismatch.
- [XTLS/Xray-core#3042](https://github.com/XTLS/Xray-core/issues/3042) — leastPing first-probe failure behavior.

**How to prevent.** §3 above — fix probeURL. Also: when adding a new balancer strategy, read its semantics from source, not from blog posts.

## 12. Changing config without understanding the blast radius

**The mistake.** Edit `active.json` → rewrite + restart xray → break all scraper + cart + login traffic simultaneously because the shared bridge is shared.

**How to prevent.**
- Any config change (probeURL, policy, balancer) goes through `vless/config_gen.py::build_xray_config` (centralized builder, never hand-edit `active.json`).
- Add a test that renders the config with the new inputs and compares to a golden file; forces explicit review when the structure changes.
- Atomic deploy: write config to temp file, validate with `xray run -test -config <temp>`, then atomic rename + HUP or restart. `vless/xray.py::_atomic_write_text` already does half of this; extend with the `-test` validation step.

## 13. Trusting Vercel HTTP 200 as a health signal

**The mistake.** Vercel returns HTTP 200 for `/api/products` because the rewrite proxies to EC2 successfully — but the data in that response can be hours stale. External uptime pings that only check status code say "all green" while product-detail requests are 100% failing.

**Live evidence.** 2026-05-03 21:00 production: Vercel `/api/products` returns 200 with 196 products; EC2 journal shows every `/api/product/{id}/details` call timing out for the last N minutes.

**How to prevent.**
- External uptime ping must hit `/api/health/deep`, not `/api/products`. The deep endpoint returns 503 when degraded.
- Separate "pipe is open" (Vercel → EC2 responds) from "data is fresh" (last successful cycle ≤ 15 min ago). Both must pass for healthy.

## 14. Forgetting test coverage catches only the known cases

**The mistake.** Write tests for the bug you just fixed. Don't test for the *class* of bug.

**How PR #25 failed this.** New tests covered "probe returns failure → rotation happens". They did not cover "healthy node responds in 8 s → 5 s probe must not false-negative". The class of bug ("timeout under empirical latency") wasn't in the test suite, so the specific case slipped through.

**How to prevent.**
- For every bug, add a test for the *class* of bug ("all probe timeouts must exceed measured p95 healthy latency"), not just the specific instance.
- Explicitly write a "what bug would this have caught?" list in the test docstring.
- Test with real-ish parameters — in this project, that means mocking httpx responses at the 8 s / 12 s boundary, not just fast/slow extremes.

## 15. Assuming prod and dev have the same network timing

**The mistake.** Tuning probe timeouts locally on a dev box with 10 ms RTT to VkusVill, deploying to EC2 with 150+ ms RTT through VLESS, and wondering why probes fail.

**How to prevent.**
- All timing-sensitive constants (probe timeouts, breaker cooldowns, handshake budgets) must have been measured on the actual EC2 bridge before landing in code.
- Record the measurement provenance in a comment: `# Measured 2026-05-03: healthy p95 = 9.2s through VLESS bridge on EC2`.
- Avoid hard-coding timing constants in tests — parameterize them so dev-box and EC2 can use different values.
