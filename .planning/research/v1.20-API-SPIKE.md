## Section A — HAR Capture Methodology (**NEEDS_OPERATOR**)

Goal: one complete HAR of an authenticated add-to-cart flow on `vkusvill.ru`
captured through a real Chromium profile, so every endpoint the real site
touches during the hot path is visible and comparable to our bridge-driven
`basket_add.php` request.

### Preconditions

- Throwaway VkusVill account with a verified RU phone + delivery address
  (otherwise `basket_add.php` returns 403 / `POPUP_ANALOGS` before the
  comparison is meaningful).
- Chrome 124+ (Edge works but Firefox HAR output differs in a way that
  makes endpoint-timing comparison painful).
- Clean Chrome profile (no extensions, empty cache) so the HAR does not
  attribute vendor timing to VkusVill.
- Network conditions: from a Russian exit IP. Easiest path: run
  Chrome inside the same xray SOCKS5 bridge our Python stack uses
  (`socks5://127.0.0.1:10808`) via `--proxy-server=socks5://127.0.0.1:10808`.

### Capture procedure

1. Launch Chrome with DevTools open, Network tab.
2. Check **Preserve log** and **Disable cache**.
3. Set Network throttling to **No throttling** (we want VkusVill's real
   server time, not client-simulated).
4. Navigate to `https://vkusvill.ru/`, log in, open any product detail.
5. Clear network log right before tapping "Add to cart".
6. Tap "Add to cart" once. Wait 3 seconds for all follow-up AJAX to
   settle (basket_recalc, delivery recompute, etc.).
7. Right-click network log, choose **Save all as HAR with content**.
8. Save as `.planning/research/v1.20-har-add-to-cart.har` (git-ignored
   by default; add to `.gitignore` if it isn't already; HAR may contain
   the operator's sessid plus personal data).

### Evidence to extract

For each request in the HAR:
- Method plus URL (full path plus query params).
- `Content-Type` of request and response.
- Approximate response body size (VkusVill's `basket_add.php`
  response is ~3-8 KB; anything dramatically smaller suggests a lighter
  endpoint worth testing).
- `timing.wait` (server think time — the number we want to beat).
- Request body when POST (parse to identify lighter payload schemas).

### Candidate endpoints to look for

Hypothesized but unverified variants:
- `/ajax/delivery_order/basket_add_short.php`
- `/ajax/quick_add.php`
- `/api/cart/add` (GraphQL or REST-style)
- Anything under `/miniapp/` or `/telegram_integration/`

**Important:** a request appearing in the HAR is not evidence it's
reachable from our bridge with our cookie surface. Section B fills
that gap through direct probing.

## Section D — Go / No-Go Recommendation (**NEEDS_OPERATOR**)

Decision tree (apply in order; stop at first match):

1. **Found a faster endpoint AND it's reachable with our cookie surface AND
   p95 through bridge is >= 30% faster than `basket_add.php` p95** ->
   **GO** (phase 64 ships the swap behind `USE_FAST_CART_ADD_ENDPOINT`
   feature flag, defaulting OFF until the 50-sample smoke gate on EC2
   passes). Populate `FAST_CART_ADD_URL` in `cart/vkusvill_api.py` and
   implement the fast-path branch inside `add()` replacing the current
   `NotImplementedError`.

2. **Found a faster endpoint BUT the speed difference is < 30%** ->
   **NO-GO on swap**. The feature-flag scaffolding stays in place
   for a future milestone but the spike result records that the
   marginal benefit wasn't worth new failure modes. Continue with
   payload minimization only (Section E).

3. **No faster endpoint found, or all candidates are auth-blocked /
   unreachable via bridge** -> **NO-GO on swap**. Accept the current
   `basket_add.php` as canonical. Proceed with payload minimization
   only (Section E).

4. **The spike uncovered any blocking risk** (e.g. endpoint works but
   produces double-adds on retry, or requires a sessid source we don't
   have) -> **NO-GO**, document the risk, mark PERF-08 as "evaluated
   plus rejected".

Operator's decision: `NEEDS_OPERATOR` (one of: GO / NO-GO / REJECT).

## Section F — Phase-64 Scaffolding Inventory (LOCAL, completed 2026-05-10)

This section is filled on the scaffolding commit so future operators
can see exactly what is already in place.

- `cart/vkusvill_api.py`
  - Added module-level `USE_FAST_CART_ADD_ENDPOINT` (reads
    `os.environ["USE_FAST_CART_ADD_ENDPOINT"] == "1"` at import, default False).
  - Added module-level `FAST_CART_ADD_URL = None` (populated in 64-03 if go).
  - `VkusVillCart.add()` gains a gate after the session check: if both
    `USE_FAST_CART_ADD_ENDPOINT` and `FAST_CART_ADD_URL` are truthy,
    raises `NotImplementedError` (intentional guard until the swap lands).
    When the flag is False (default), the legacy `basket_add.php` path
    runs byte-for-byte as it did pre-Phase-64.

- `scripts/ablate_basket_add_payload.py` (~270 LOC)
  - CLI: `--user-id`, `--product-id`, `--n-per-field`, `--output`,
    `--cookies-path`, `--dry-run`.
  - Live mode: iterates `PAYLOAD_FIELDS` (the exact 16-field tuple from
    `VkusVillCart.add()`), runs baseline plus per-field ablation, classifies
    each field as `droppable` (success_rate >= 0.95) or `required`.
  - Dry-run mode: validates the payload shape matches `PAYLOAD_FIELDS`
    and emits synthetic results; zero network, CI-safe.
  - Output JSON schema: `{mode, product_id, user_id, fields_ablated,
    baseline: {n, success_rate, p50_ms, p95_ms}, per_field: {<field>: {...}},
    verdict: {droppable: [...], required: [...]}}`.

- `tests/test_cart_feature_flag.py` (3 unit tests)
  - `test_fast_endpoint_disabled_by_default`
  - `test_fast_endpoint_raises_when_flag_on_but_url_none`
  - `test_legacy_path_when_flag_off`

- `scripts/verify_v1.20.sh` Phase 64 block (3 smoke checks, EC2-runnable)
  - **64-A**: flag defaults False with clean env.
  - **64-B**: `FAST_CART_ADD_URL is None` (pre-spike state).
  - **64-C**: ablation harness `--dry-run` exits 0 without network.

Nothing in this scaffolding changes the live cart-add behavior.
`USE_FAST_CART_ADD_ENDPOINT=0` (default) means the legacy `basket_add.php`
path runs exactly as it did on the Phase-63 branch tip.
