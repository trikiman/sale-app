# FEATURES — v1.19 Production Reliability & 24/7 Uptime

Feature catalog for a "keep the site healthy from the user's perspective, 24/7" milestone, grouped by category with table-stakes vs differentiators vs anti-features.

## Category A: Pre-flight proxy health (REL-probe)

What it is: detect that the VLESS bridge cannot actually reach VkusVill *before* spending 30-45 s per scraper launch + Chrome startup only to fail at page load.

### Table stakes
1. **A1. Probe bridge liveness before each scraper launch** — GET `https://vkusvill.ru/` through `socks5h://127.0.0.1:10808` with a timeout that *exceeds* empirical healthy-node latency.
2. **A2. Empirical probe timeout** — 12 s minimum (healthy node observed at 7-9 s; 5 s false-negatived in PR #25). Codify the measurement in a test so future tuning is evidence-based.
3. **A3. Cap mid-cycle rotations** — at most 2 rotations per scraper launch. The goal is "one quick swap if the current exit is dead", not "exhaustive pool scan".
4. **A4. Non-destructive rotation path** — when PR #25 rotates, it currently triggers `_remove_host_and_restart` via `next_proxy()`. Prefer rotation through xray's own balancer decision; fall back to Python-side removal only when the balancer has no other healthy member.

### Differentiators
5. **A5. Probe result cached for 30 s** — if a scraper launch N-1 probed the bridge and it was OK, the next launch N within 30 s does not need to re-probe. Saves 1-2 s per cycle.
6. **A6. Probe returns *why* it failed** — "DNS", "TLS", "HTTP 4xx", "timeout" — logged with the node identity so repeat offenders get quarantined explicitly.

### Anti-features
- ❌ **Probing with 5 s timeout** — already proven wrong (PR #25 / PR #26).
- ❌ **Restart xray on every false-negative probe** — already proven wrong (PR #25 cascade).
- ❌ **Block the scraper until a healthy node is found** — scheduler must keep forward progress; use circuit breaker to sit out cycles instead.

## Category B: Probe/target alignment (REL-probe-target)

What it is: align xray's observatory probeURL with the traffic pattern we actually care about (VkusVill), not an unrelated target (Google).

### Table stakes
1. **B1. Change observatory probeURL to VkusVill** — currently `https://www.google.com/generate_204`. Change to a stable, lightweight VkusVill endpoint (or `https://vkusvill.ru/` root). The `leastPing` balancer is only useful if it measures the *real* target latency, not third-party latency.
2. **B2. Document that probeURL changes matter** — comment in `vless/config_gen.py` + test that admits regression if someone accidentally reverts to google.

### Differentiators
3. **B3. Multi-target probe** — probe VkusVill + a secondary target (e.g. ipinfo.io for geo) and only admit a node if *both* succeed. Prevents "node reaches Google but not VkusVill" from ever hitting the balancer.
4. **B4. Per-node VkusVill failure counter** — observatory only reports latency, not request outcomes. Track VkusVill-specific HTTP 4xx/5xx/timeout per node in `proxy_events.jsonl` and auto-quarantine after K failures in a sliding window.

### Anti-features
- ❌ **Probing twelve different URLs** — diminishing returns + probe duration inflation.
- ❌ **Using xray `leastLoad`** — GitHub issue [#3138](https://github.com/XTLS/Xray-core/discussions/3138) shows `leastLoad` on real inbounds + portals breaks routing; `leastPing` is the right strategy, just needs the right target.

## Category C: Graduated circuit breaker recovery (REL-breaker)

What it is: when the scraper fails repeatedly, don't sit indefinitely at "retry every 2 min". Move to a half-open probe state, back off exponentially, and resume gradually.

### Table stakes
1. **C1. Add half-open state** — after cooldown elapses, run *one green-only probe cycle* (fast, ~30 s) before committing to a full cycle. If the probe succeeds → resume full cycles. If it fails → back to open with longer cooldown.
2. **C2. Exponential backoff** — current 120 s fixed. Replace with `min(120 * 2^n, 30 min)` where `n` is the number of open→half-open→open transitions without a full recovery. Prevents 162 useless cycles in 5 h.
3. **C3. Reset counter on partial success** — current counter only decrements on a fully-clean cycle (all scrapers green). If 2 of 3 succeed, the breaker stays at its current count. Change: reset on any successful scrape.
4. **C4. Expose breaker state** — `state: closed | open | half_open`, `opened_at`, `last_probe_at`, `consecutive_failed_cycles`, `next_attempt_at` visible in `/api/health/deep`.

### Differentiators
5. **C5. Auto-trigger pool refresh on breaker trip** — already exists (line 638-640 of scheduler_service.py). Keep, but also trigger on entering half-open if probe fails (not just on initial trip).
6. **C6. Telegram bot alert on breaker state change** — fires once per transition (closed→open, open→half_open, half_open→open). Dedup: no repeated alerts while state is stable.

### Anti-features
- ❌ **Auto-restart services** — if xray or scheduler is genuinely stuck, human should investigate. Auto-restart masks root causes.
- ❌ **Alerting on every failed cycle** — fatigue. Alert on state transitions only.

## Category D: Pool drain/replenish balance (REL-pool)

What it is: during high-churn periods (e.g. VkusVill tightening blocks), the 4 h cooldown quarantine can drain nodes faster than the 24 h refresh replenishes them. Production on 2026-05-03 shows this: admitted pool 25 (v1.18 shipping day) → 13 (8 days later).

### Table stakes
1. **D1. Min-healthy triggered refresh** — already exists (`ensure_pool` at line 289 of `vless/manager.py`). Confirm it fires before the pool hits `MIN_HEALTHY = 7`, not after.
2. **D2. Pool size trend logged** — each cycle logs `pool_size=N, quarantined=M, active_outbounds=K` so drift is visible. Emit to `proxy_events.jsonl` at refresh start and end.

### Differentiators
3. **D3. Replenish-to-target, not replenish-to-min** — when a refresh runs, admit up to `MAX_CACHED` instead of stopping at `MIN_HEALTHY + 1`. Keeps headroom for future quarantines.
4. **D4. Shorter cooldown for non-block failures** — 4 h is correct for confirmed VkusVill blocks (403 / 451 / ReadTimeout on vkusvill.ru specifically). For connection-layer failures (TLS handshake, outbound unreachable), 15 min is enough.
5. **D5. Predictive refresh** — when quarantine rate in the last hour > replenish rate in the last 24 h, schedule an unplanned refresh before `MIN_HEALTHY` is hit.

### Anti-features
- ❌ **Lowering `MIN_HEALTHY` below 7** — too few nodes = leastPing has no real choice, noise dominates.
- ❌ **Doubling `MAX_CACHED`** — the xray config rebuild is O(N outbounds); 30 is already near the sweet spot.

## Category E: Deep health observability (OBS-health)

What it is: a single endpoint that truthfully reports whether the stack is healthy, accessible without auth, machine-readable, and checkable from Vercel-side or an external uptime ping.

### Table stakes
1. **E1. `GET /api/health/deep`** — unauthenticated, returns HTTP 200 + JSON when healthy, HTTP 503 + JSON when unhealthy. No sensitive data (no node IPs, no user counts).
2. **E2. Healthy criteria** — all of: pool size ≥ MIN_HEALTHY, circuit breaker closed or half-open, last successful full cycle ≤ 15 min ago, xray process running, merged products.json mtime ≤ 15 min ago.
3. **E3. Degradation tiers** — response body always contains `status: healthy | degraded | unhealthy` and a `reasons: []` array listing whichever criteria failed.
4. **E4. Rate-limited** — max 1 req/s per IP. Uptime pings don't need more; this is not a user endpoint.

### Differentiators
5. **E5. Light variant: `GET /api/health`** — returns HTTP 200 as long as the process is alive. Cheap, for load balancers / systemd watchdogs.
6. **E6. Same data in `/admin/status`** — operator view behind admin token gets *everything* E1 has plus node-level detail.
7. **E7. Telegram heartbeat** — optional: bot sends a "still healthy" ping every 6 h. Low noise floor; proves the alerting loop itself is alive.

### Anti-features
- ❌ **Exposing node IPs to unauth endpoint** — privacy + operational security.
- ❌ **Requiring admin token for `/api/health/deep`** — defeats the purpose; external ping services won't have the token.
- ❌ **SSL cert issuance for a new subdomain** — we already have `vkusvillsale.vercel.app` with Vercel-managed TLS; route `/api/health/deep` through the same rewrite.

## Category F: Live verification gate (OPS-verify)

What it is: a non-optional verification step that every phase in v1.19 must pass before closure. PR #25 skipped this and reverted in 8 minutes — this milestone's cultural commitment is that we don't let that happen again.

### Table stakes
1. **F1. Per-phase VERIFICATION.md** — each phase includes an EC2 deploy + smoke test + Vercel-side check + *rollback evidence* (what to do if it breaks). Pattern already exists (phase 57 + 58); make it mandatory.
2. **F2. Smoke script for each phase** — shell-executable, idempotent, runs on EC2 via SSH, reports pass/fail per criterion. Not a manual checklist.
3. **F3. Rollback rehearsal** — for each phase, prove the revert works by dry-running it against a staging branch before merging.

### Differentiators
4. **F4. Pre-merge live verification on a branch** — deploy the PR branch to EC2, run the smoke script, only then merge. Avoids the PR #25 scenario where main got a broken commit.
5. **F5. Post-deploy observation window** — after a phase lands on main + EC2, the smoke script runs hourly for 24 h and alerts on any regression.

### Anti-features
- ❌ **Only running unit tests** — PR #25 passed unit tests (new tests were added). Unit tests don't catch empirical timing bugs.
- ❌ **Manual verification in a chat message** — not reproducible, not auditable. Must be scripted.

## Category G: User-visible degraded mode (UI-degraded)

What it is: when the backend *is* degraded, tell the user truthfully instead of showing a fake-healthy UI.

### Table stakes
1. **G1. Stale banner reflects deep health** — existing stale banner (UI-07, from v1.0) fires on `data age > 15 min`. Augment to also fire when `/api/health/deep` reports `status != healthy`.
2. **G2. Detail drawer handles null** — currently `/api/product/{id}/details` returns HTTP 200 with fallback content when proxy fails. The UI should visually indicate "live detail unavailable, showing cached info" instead of silently showing empty sections.

### Differentiators
3. **G3. Cart add uses deep health** — if deep health says unhealthy, miniapp disables the add button with a tooltip "VkusVill temporarily unreachable, try again in a few minutes" instead of spinning for 8 s and failing.
4. **G4. Admin panel surfaces breaker state** — `/admin` gets a "Reliability" tab showing circuit state, pool size, recent probe results.

### Anti-features
- ❌ **Silent failure to 500** — current detail-proxy path does this (returns 200 + empty fallback). Anti-pattern; mask is worse than no mask.
- ❌ **Blocking the miniapp on a `/api/health/deep` check at load time** — extra RTT before anything renders. Health is a parallel signal, not a load-time gate.
