# SUMMARY — v1.19 Production Reliability & 24/7 Uptime

Synthesis of `v1.19-STACK.md`, `v1.19-FEATURES.md`, `v1.19-ARCHITECTURE.md`, `v1.19-PITFALLS.md`. Grounded in live 2026-05-03 production evidence (162 consecutive scraper failures, pool 25 → 13 over 8 days, detail-proxy 30/30 timeouts in the last 10 min) and the PR #25 → PR #26 revert post-mortem.

## Three findings that drive everything

### 1. The observatory probes Google; our traffic goes to VkusVill.

This is the single biggest architectural mistake in the current stack. `active.json::observatory.probeURL = https://www.google.com/generate_204`. xray's `leastPing` balancer ranks outbounds by probe latency, but a node can be blocked by VkusVill (403/451/reset) while Google still answers in 200 ms. That node stays "fast" in the balancer — real requests fail silently; observatory stays green.

This silently explains a large portion of the "162 consecutive failures with an 'active' xray service" pattern we see right now. Fixing this one config line is probably the highest-leverage change in the entire milestone.

### 2. Any rotation operation restarts xray. PR #25 + PR #26 both missed this.

Both `next_proxy()` and `mark_current_node_blocked()` call `_remove_host_and_restart` (`vless/manager.py:210` and `:230`). PR #26's revert description claimed the fix was "use `mark_current_node_blocked` because it doesn't restart xray" — that was wrong. Both paths rebuild the config and re-exec the binary. The actual lesson from PR #25 is: **cap the number of rotations, don't rely on "the non-restarting path" (there isn't one).**

Any future rotation logic must prefer xray's internal balancer decision (no restart) and fall back to Python-side removal only when the balancer has no healthy member left.

### 3. The circuit breaker has no half-open state; recovery is purely time-based.

Current: at 3 consecutive fully-failed cycles, sleep 120 s, then try a full cycle again. If it fails, +1 to counter, sleep 120 s, try again. 162 consecutive failures × 120 s ≈ 5.4 hours of production "broken but running" before anything recovers.

Industry best practice (Resilience4j, pybreaker, gobreaker) is a 3-state machine: closed → open → half-open → closed. The half-open state runs a cheap probe (one green-only cycle) before committing to full cycles. We need this, with exponential backoff capped at 30 min.

## Key decisions (recorded here, not negotiable in later phases)

| # | Decision | Rationale |
|---|---|---|
| D1 | **Roll our own circuit breaker, do not add `pybreaker`.** | Scheduler failure model is cycle-shaped, not request-shaped. `pybreaker` decorators fit RPC, not cycles. |
| D2 | **Probe timeout = 12 s (not 5 s).** | Measured healthy-node p95 through bridge = 9.2 s; 5 s was wrong (PR #25 evidence), 12 s gives ~30% margin. |
| D3 | **Cap rotations per scraper launch at 2.** | PR #25's 5 rotations × xray restarts cascaded into 2.5 s of xray thrash per bad probe window. |
| D4 | **Change `observatory.probeURL` to `https://vkusvill.ru/favicon.ico`.** | Aligns balancer probe with actual traffic target. Biggest leverage per line of config. |
| D5 | **Unauthed `/api/health/deep` returns 503 on degradation.** | External uptime pings need status code, not JSON inspection. Admin endpoints stay token-gated. |
| D6 | **No privacy-sensitive fields in unauth health response.** | No IPs, no node identifiers, no user counts. Admin version gets the full dump. |
| D7 | **Every phase in v1.19 includes a scripted EC2 smoke test as a mergeable artifact.** | PR #25 reverted in 8 min because this was skipped. Making it mandatory is the cultural commitment. |
| D8 | **Retain `MIN_HEALTHY = 7`, `MAX_CACHED = 30`, `VKUSVILL_COOLDOWN_S = 4h` as defaults.** | Values are well-tuned. Any change must come with evidence (pool drift data from phase 61 observation). |
| D9 | **Defer user-facing degraded mode (Category G) to v1.20 unless bandwidth.** | v1.19 is backend reliability. UI changes are cleaner if stack is solid first. |
| D10 | **Use external uptime service (e.g. Uptime Robot) rather than building one.** | Family-scale project; Prometheus + self-hosted monitoring is over-engineering. |

## Recommended phase breakdown

**Phase 59: Corrected pre-flight VLESS probe** (redo PR #25 with the lessons applied)
- Scope: `vless/preflight.py::probe_bridge_alive(timeout=12)`, cap 2 rotations in `_run_scraper_set`, regression test for the 5 s false-negative case.
- Ships standalone. Easy to revert. Closes the original PR #25 intent cleanly.

**Phase 60: Observatory probeURL + graduated circuit breaker**
- Scope: change `observatory.probeURL` to VkusVill favicon, replace `consecutive_fails` counter with `scheduler/breaker.py::CircuitBreaker` (closed/open/half_open + exponential backoff), persist state in `data/scheduler_state.json`.
- Two related changes in one phase because both are "how does the system recover from node failure" and benefit from one coherent test pass.

**Phase 61: Deep health endpoint + pool snapshot**
- Scope: new `backend/health.py`, mount `/api/health` and `/api/health/deep`, extend `/admin/status` with reliability block, add `pool_snapshot()` to `vless/manager.py`.
- Enables external uptime monitoring and operator visibility.

**Phase 62 (optional): External uptime integration**
- Scope: set up one external uptime service pinging `/api/health/deep`, wire Telegram alert on state change.
- Config-only work; no code changes. Include if bandwidth; defer to v1.20 otherwise.

## What v1.19 is NOT

- **Not a "replace xray with something else" milestone.** xray + VLESS+Reality is correct for this use case.
- **Not a "migrate to Kubernetes" milestone.** EC2 + systemd is fine for this scale.
- **Not a user-facing feature milestone.** UI work (degraded banners, detail drawer improvements, admin reliability tab) is deferred.
- **Not about scraper internals.** v1.18 already hardened the CDP-WS recovery; scrape_green/red/yellow logic stays put.

## Watch-out-for list (directly actionable guardrails)

Every phase in v1.19 must:

1. **Measure before tuning.** Any timing constant (timeout, cooldown, backoff level) must reference a measurement with date + provenance. No copy-pasted values.
2. **Test the class, not the case.** For each bug fixed, write a test asserting the *category* of the bug (e.g. "all probe timeouts must exceed measured p95 healthy latency") in addition to the specific instance.
3. **Smoke-test on EC2 before merge.** Scripted, reproducible, captured in `scripts/verify_v1.19.sh` (grows per phase). No manual chat-message verification.
4. **Capture rollback evidence.** Prove the revert works by dry-running it against a branch before merging the phase commit.
5. **Alert on state transitions, not failure counts.** Closed→open fires once. Open→half-open→closed fires once on recovery. No per-cycle alerts.
6. **Guard the class of mistake.** Every PR adds a test that would have caught PR #25's specific regression had it existed earlier. Growing this test set is how the cultural commitment becomes structural.
